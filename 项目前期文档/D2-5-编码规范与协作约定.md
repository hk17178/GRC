# GRC 平台 · 编码规范与协作约定

**文档编号：** D2-5 ｜ **版本：** v1.0 ｜ **日期：** 2026-06-24
**性质：** 项目级强制标准（所有提交须遵守；CI 自动 gate）
**依据：** 用户开工前明确的五项标准 + D1-1 架构 + D0-3 前端设计规范

> 本文是开发期的"宪法"。任何代码、任何提交都须满足以下五条；不满足不合并。

---

## 一、部署与跨发行版兼容（目标：Rocky Linux 9.4，兼容主流 Linux）

**目标环境**：Rocky Linux 9.4（RHEL 9 系，glibc 2.34 / systemd / SELinux enforcing / firewalld）。
**兼容目标**：Ubuntu 22.04+/Debian 12+/openEuler/Anolis 等主流发行版。

**强制约定**
1. **全栈容器化优先**：PostgreSQL16+pgvector / Redis / MinIO / 后端 / 前端 均以容器交付，运行时不依赖宿主发行版差异——这是跨发行版兼容的根本手段。
2. **基础镜像发行版无关**：JDK 用 `eclipse-temurin:17`（非 OS 自带 JDK）；前端构建用 `node:20`；避免绑定特定发行版包管理器。
3. **换行符强制 LF**（见根 `.gitattributes`）：杜绝 CRLF 导致 Rocky 上 shell/entrypoint `bad interpreter`。
4. **脚本用 POSIX sh**：部署/运维脚本 `#!/usr/bin/env sh`，不使用 bash-only 语法；如必须区分包管理器，同时提供 `dnf` 与 `apt` 分支。
5. **SELinux 适配**：容器挂载卷加 `:Z`（私有）/`:z`（共享）标签；文档说明 `semanage`/`restorecon` 注意项；不建议 `setenforce 0`。
6. **防火墙**：交付 `firewall-cmd --add-port` 与 `ufw allow` 两套端口开放说明（应用 8080 / PG 5432 / MinIO 9000-9001 等）。
7. **时区/编码**：容器内统一 `UTC`+应用层按 `Asia/Shanghai` 展示；DB `LC_COLLATE`/`LC_CTYPE` 用 `C.UTF-8` 或 `en_US.UTF-8`，避免发行版默认差异。
8. **生产交付物**：提供 `docker compose`（中小规模）与 systemd unit / Helm（可选）两种；不依赖图形界面。

**验收**：在 Rocky 9.4（SELinux enforcing）全新机一键 `docker compose up` 起栈通过冒烟。

## 二、代码注释（强制：完整中文注释）

1. **每个源文件**有文件级注释，说明职责、所属模块、关键设计依据（指向 D1-x 文档/章节）。
2. **每个类/接口**有中文用途说明；**每个公有方法**说明意图、参数、返回、副作用、异常。
3. **复杂逻辑/算法/红线**（隔离、权限、加密、状态机、并发、调度）必须有"为什么这么写"的中文说明，而非复述代码。
4. 注释**与代码同步更新**；过期注释视为缺陷。
5. 后端 Javadoc 风格中文；前端 Vue 用 `<!-- -->`/JSDoc 中文；SQL 迁移每段有中文注释。
6. **不为废话注释**（`i++ // i 加一` 禁止）；注释解释意图与约束，不解释语法。

**验收**：评审时无中文注释或注释名不副实的方法，打回。

## 三、前端开发（强制：严格遵循设计规范与原型，100% 复原）

1. **唯一真源**：`前端设计/前端原型/高保真完整演示·驾驶舱版.html` 与 `前端设计/设计规范/tokens.css` 是前端的权威真源；设计令牌、配色、间距、圆角、组件形态以其为准（D0-3 已声明"高保真原型 HTML 为可复原最终真源"）。
2. **技术栈**：Vue 3 + Vite；样式**复用 `tokens.css`**（CSS 变量 + `body.t-*` 五主题），不得另起一套色板。
3. **复原标准**：组件视觉/交互须与原型一致——五主题、i18n 中英切换、登录页(AD 优先)、可编辑看板栅格、热力矩阵等行为逐一对齐。
4. **核对手段**：开发中用 preview/DOM 度量核对；偏差须修正或经设计确认。
5. **不臆造 UI**：原型未定义的新界面，先补设计（设计规范/原型）再实现，保持单一真源。
6. **i18n**：中英文案走外置语言包；新增文案同时补 zh/en，杜绝硬编码与中英混杂残留。

### 三·补 前端"浏览器验收闭环"（强制，前端唯一的"跑绿"判据）
> `npm run build` 绿 **不等于** 页面能用——运行时错误/样式错位/与原型偏差，构建都抓不到。前端每个阶段开发完成，必须用**真实 Chrome** 完成下述闭环，浏览器里验过才算"跑绿"：
> 1. **能渲染**：起 dev server，Chrome 打开页面，查控制台**无报错**、关键 DOM/文案在位（不是空白）。
> 2. **功能对**：逐一实跑交互——主题切换（含品牌侧/各分区配色）、中英 i18n（无中文残留）、标签/表单/按钮等，行为符合预期。
> 3. **比原型**：与 `前端设计` 文件夹中对应的**规范与原型并排比对**（同一浏览器分别打开原型 HTML 与本工程页面），配色/布局/间距/组件形态/各主题表现**逐项一致**，发现偏差即修。
> 4. 三步全过才提交；任一不过即视为未完成。CI 的 `npm build` 只是底线门禁，不替代浏览器验收。
> 5. 子代理"已浏览器实测"之类的自述**不可作为通过依据**，最终渲染与原型一致性由主代理在 Chrome 亲自确认。
> 建议补 **Vitest mount 冒烟测试**（mount 关键视图，渲染抛错即红）纳入 CI，把"能渲染"也变成自动门禁。

**验收**：关键页面与原型并排核对一致；切主题/切语言无样式错乱、无中文残留。

## 四、测试与提交门槛（强制：测试跑绿才提交）

1. **每个功能**附自动化测试：后端单元 + 集成（Testcontainers 起真实 PG，覆盖隔离/权限/状态机等红线）；前端组件测试（Vitest）+ 关键交互测试。
2. **绿灯才提交**：本地/CI 全部测试通过方可 commit & push；**红线用例（隔离/留痕/权限/数据不出域）必须通过**，不得跳过或 `@Disabled` 绕过。
3. **本项目工具链现实**：开发机本机 Java 为 8 且无 Maven → **后端构建/测试统一用 Docker 跑**：
   `docker run --rm -v "$PWD":/app -w /app maven:3.9-eclipse-temurin-17 mvn verify`
   前端用本机 Node 直接 `npm test && npm run build`。
4. **CI 自动 gate**：`.github/workflows/ci.yml` 在每次 push/PR 跑 `mvn verify`（Runner 内置 Docker 支持 Testcontainers）+ 前端测试构建；**CI 红则不合并**。
5. **覆盖率**：核心模块行覆盖率目标 ≥70%，红线路径 100% 有用例（与 DM-1 RTM、D3-2 对齐）。
6. 提交信息：中文、说明动机与影响，遵循既有风格；红线相关注明对应用例编号。

**验收**：CI 全绿；红线用例存在且通过。

---

## 五、测试节奏（约定：统一人工测试放在整体开发完成后）

1. **开发期**：以**自动化测试**保障质量（单元/集成/Testcontainers/前端组件），每功能跑绿即进；不在每个模块停下来做人工验收。
2. **集成期末**：整体功能开发完成后，按 D3-2 测试用例 + D5-x 用户手册组织**统一人工测试 / UAT**（含红线场景、跨子公司隔离、端到端业务闭环、性能、安全测试 D3-3）。
3. **回归**：人工测试发现的问题修复后补自动化用例，纳入 CI 防回归。
4. 该节奏不豁免红线：红线用例在开发期即须自动跑绿，人工测试是叠加确认而非首次验证。

**验收**：进入统一人工测试前，自动化测试全绿且红线全覆盖。
