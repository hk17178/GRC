## 一、引言

### 1.1 目的
定义平台的逻辑/物理数据模型，**首要确立组织维度数据隔离的落库方式**（需求 7.3：隔离边界须在数据模型阶段定死，后期补救成本极高），并给出表单引擎、留痕库等关键子系统的数据结构。

### 1.2 范围
覆盖 M1–M11 业务模块及系统管理域（模型接入、建议与反馈 M-FB）的核心数据实体、组织隔离机制、版本化、留痕防篡改、分类分级。Flowable 工作流自有表由引擎托管，本文只描述平台侧与之关联的数据。

### 1.3 建模五原则
1. **组织隔离第一**：每个业务实体携带组织归属，隔离在数据访问层统一拦截 + PG 行级安全（RLS）兜底，**绝不在各模块各写一遍**。
2. **配置即数据**：模板、控件、体系、通知规则、权限、法规源均为数据（支持运营自助、零发版）。
3. **弹性与可统计兼顾**：表单答案等异构结构用 JSONB；但打分/等级等需聚合的字段**抽取为规范化列**，保证 M9 看板可聚合。
4. **版本化与快照**：制度、模板、评估、法规均版本化；评估启动即固化模板版本快照。
5. **留痕不可篡改**：证据/操作日志写入即只读、哈希校验、独立存储，保留期取各监管口径最大值。

## 三、组织维度与数据隔离设计 ⭐地基

### 3.1 组织树（org）
统一承载集团/子公司/部门多级结构，作为隔离与通知路由的判断基础（需求 8.2）。

```sql
CREATE TABLE org (
  id          BIGINT PRIMARY KEY,            -- 雪花ID
  parent_id   BIGINT REFERENCES org(id),
  org_type    VARCHAR(16) NOT NULL,          -- GROUP集团/SUBSIDIARY子公司/DEPT部门
  code        VARCHAR(64) UNIQUE NOT NULL,
  name        VARCHAR(128) NOT NULL,
  path        VARCHAR(512) NOT NULL,         -- 物化路径 /1/12/120 便于子树查询
  ad_dn       VARCHAR(256),                  -- 对应AD的可分辨名
  status      VARCHAR(16) DEFAULT 'ACTIVE',
  ...审计字段
);
CREATE INDEX idx_org_path ON org (path text_pattern_ops);
```

### 3.2 隔离落库约定（关键）
- **每个业务实体表都带 `org_id`（所属子公司）** 与必要时 `dept_id`，作为隔离锚点。
- 隔离判断由**统一数据访问层**根据当前用户的 `角色×组织` 自动追加过滤条件（`org_id ∈ 用户可见组织集合`），**业务代码不手写隔离条件**。
- **纵深防御**：对敏感表启用 PostgreSQL **行级安全（RLS）**，即使应用层漏判，数据库层也拦截越权。
  ```sql
  ALTER TABLE assessment ENABLE ROW LEVEL SECURITY;
  CREATE POLICY p_org ON assessment USING (org_id = ANY (current_setting('app.visible_orgs')::bigint[]));
  ```
- **集团管理层 / 只读审阅角色**例外：可见组织集合=全集团（需求 10.2/10.3）。
- 可见组织集合按"用户-角色-组织"绑定 + 角色是否跨子公司汇总（集团层）计算，缓存于会话。

### 3.3 四元组权限锚点
任一数据访问 = `用户(app_user) × 角色(role) × 组织(org) × 数据(行)`；前三者决定"可见组织集合"，第四由 `org_id` 落地（详见 D1-3）。

## 四、身份与组织（AD 同步）

```sql
CREATE TABLE identity_source (                 -- 身份源/域控（CR-003 多域控）
  id          BIGINT PRIMARY KEY,
  code        VARCHAR(32) UNIQUE NOT NULL,    -- 域代码，如 CORP / PAYSVC / OVERSEAS
  name        VARCHAR(128),
  domain      VARCHAR(128),                   -- AD 域/林，如 corp.example.com
  ldap_url    VARCHAR(256),                   -- ldaps://dc1;dc2（可多控制器高可用）
  base_dn     VARCHAR(256),
  upn_suffix  VARCHAR(128),                   -- 用于认证路由（user@suffix）
  org_id      BIGINT REFERENCES org(id),      -- 绑定子公司组织子树
  sync_cron   VARCHAR(32),
  status      VARCHAR(16) DEFAULT 'ACTIVE',
  ...凭证经密钥管理引用，不落明文
);

CREATE TABLE app_user (
  id          BIGINT PRIMARY KEY,
  domain_id   BIGINT REFERENCES identity_source(id),  -- 所属身份源/域（CR-003）
  username    VARCHAR(64) NOT NULL,           -- 来自AD（域内唯一）
  upn         VARCHAR(160),                   -- userPrincipalName，登录标识 user@domain
  display_name VARCHAR(128),
  email       VARCHAR(128),
  org_id      BIGINT REFERENCES org(id),     -- 主组织（子公司/部门）
  ad_object_id VARCHAR(64),                  -- AD唯一标识，用于增量同步
  status      VARCHAR(16) DEFAULT 'ACTIVE',  -- ACTIVE/DISABLED（离职即停用→权限回收）
  last_sync_at TIMESTAMPTZ,
  ...审计字段,
  UNIQUE(domain_id, username)                 -- 唯一性改为"域内唯一"，避免跨域同名冲突
);
```
- 账号与组织由 AD **单向同步**（需求 8.2、10.4）；平台不独立维护账号体系。**多域控（CR-003）下**为每个 `identity_source`（各子公司/海外独立 AD 域/林）各自单向同步，集团仅聚合、不依赖域间信任（见上文 §四 多域控说明）。
- 离职/调岗：同步时置 `DISABLED` 或更新 `org_id`，触发权限即时回收（D1-3 详述时效）。
- **多域控（CR-003）**：支持配置多个 `identity_source`（各子公司/海外独立 AD 域），每个绑定 `org_id`；用户唯一键为 `(domain_id, username)`，登录用 `DOMAIN\user` 或 `user@domain` 做认证路由；跨域用户落在不同 `org_id`，沿用组织隔离（见 D1-3 §3.1.1）。

```sql
CREATE TABLE asset (
```

## 六、制度体系（M1）

```sql
-- 制度主体
CREATE TABLE policy (
  id          BIGINT PRIMARY KEY,
  org_id      BIGINT NOT NULL REFERENCES org(id),
  code        VARCHAR(64),
  name        VARCHAR(256) NOT NULL,
  framework   VARCHAR(32)[],                 -- 所属体系（ISO27001/等保/PIPL/PBOC…）多值
  level       VARCHAR(16),                   -- 集团级/子公司级/部门级
  owner_dept  BIGINT REFERENCES org(id),
  is_core     BOOLEAN DEFAULT false,         -- 是否核心制度（决定强制签署范围）
  review_cycle_months INT,                   -- 复审周期
  current_version_id BIGINT,                 -- 指向当前生效版本
  status      VARCHAR(16),                   -- DRAFT/REVIEW/EFFECTIVE/DEPRECATED
  ...审计字段
);
-- 版本（保留完整历史）
CREATE TABLE policy_version (
  id          BIGINT PRIMARY KEY,
  policy_id   BIGINT NOT NULL REFERENCES policy(id),
  version_no  VARCHAR(16) NOT NULL,
  effective_date DATE,
  content_ref VARCHAR(256),                  -- 正文存储引用（MinIO）
  change_type VARCHAR(16),                   -- MAJOR(触发重新确认)/MINOR(仅元数据)
  status      VARCHAR(16),
  ...审计字段
);
-- 引用关系（下位文件引用上位条款）
CREATE TABLE policy_reference (
  from_policy_id BIGINT, from_clause VARCHAR(64),
  to_policy_id   BIGINT, to_clause   VARCHAR(64)
);
-- 签署确认记录（合规证据，写入留痕库）
CREATE TABLE policy_attestation (
  id          BIGINT PRIMARY KEY,
  policy_version_id BIGINT NOT NULL REFERENCES policy_version(id),
  user_id     BIGINT NOT NULL REFERENCES app_user(id),
  org_id      BIGINT NOT NULL,
  confirmed_at TIMESTAMPTZ,                  -- 确认时间
  version_no_snapshot VARCHAR(16),           -- 确认时的版本号
  status      VARCHAR(16)                    -- PENDING/CONFIRMED
);
```
- 强制确认范围按 `is_core` + 制度类型配置（需求 3.2.2，限定核心制度，避免确认疲劳）。
- `change_type=MAJOR` 触发重新确认流程。
- 知识库切片（向量）见 §十四。

## 七、风险评估与表单引擎（M2）⭐最高难点

> 表单引擎是全项目核心难点。数据模型采用"**结构规范化 + 答案 JSONB + 打分抽取列**"混合策略：结构与控件规范化以支持引用/统计/复用，答案弹性存 JSONB，打分结果抽取为列以支撑看板聚合。

### 7.1 体系与控件库

```sql
-- 体系（内置脚手架 + 自定义新增）
CREATE TABLE framework (
  id BIGINT PRIMARY KEY, code VARCHAR(32) UNIQUE, name VARCHAR(128),
  is_builtin BOOLEAN, ...审计字段                -- 等保/ISO27001/PBOC/PCI/27701/27018/9001/供应商 + 自定义
);
-- 统一控件库（"具体控制点"颗粒度，需求4.2.2/V1.7）
CREATE TABLE control_library (
  id BIGINT PRIMARY KEY,
  code VARCHAR(64) UNIQUE,                       -- CTL-007
  name VARCHAR(256),                             -- “特权账号定期审查”
  category VARCHAR(64),                          -- 控制域分类（仅归类用，不替代控制点颗粒度）
  description TEXT, ...审计字段
);
-- 控件↔体系跨映射（一个控件对应多体系的不同条款/编号）
CREATE TABLE control_framework_map (
  control_id BIGINT REFERENCES control_library(id),
  framework_id BIGINT REFERENCES framework(id),
  clause_code VARCHAR(64),                       -- 该体系下的条款/控制点编号
  PRIMARY KEY (control_id, framework_id)
);
```

### 7.2 模板（结构配置层 + 打分/报告映射层，两层解耦）

```sql
CREATE TABLE template (
  id BIGINT PRIMARY KEY,
  framework_id BIGINT REFERENCES framework(id),
  usage VARCHAR(16),                             -- ASSESSMENT风险评估 / AUDIT审计（同一引擎按用途区分，需求5.2.2）
  name VARCHAR(128),
  current_version_id BIGINT, ...审计字段
);
CREATE TABLE template_version (
  id BIGINT PRIMARY KEY,
  template_id BIGINT REFERENCES template(id),
  version_no VARCHAR(16),
  effective_date DATE,
  status VARCHAR(16),                            -- DRAFT/EFFECTIVE/RETIRED
  report_format JSONB,                           -- 报告章节顺序/措辞规范（随版本固化）
  scoring_model JSONB,                           -- 打分模型（定量:可能性×影响 / 定性等级 / 是否适用）
  ...审计字段
);
-- 结构配置层：层级（域→控制项），自引用
CREATE TABLE template_section (
  id BIGINT PRIMARY KEY,
  version_id BIGINT REFERENCES template_version(id),
  parent_id BIGINT REFERENCES template_section(id),
  seq INT, name VARCHAR(256),
  level_label VARCHAR(32)                         -- 命名可配置（如“控制点编号”）
);
-- 结构配置层：题目；可引用控件库条目（复用），也可自定义
CREATE TABLE template_item (
  id BIGINT PRIMARY KEY,
  section_id BIGINT REFERENCES template_section(id),
  control_id BIGINT REFERENCES control_library(id),  -- 引用控件库（NULL=自定义题）
  seq INT,
  question TEXT,
  item_type VARCHAR(24),                          -- SINGLE/MULTI/SCORE/TEXT/MATRIX/ATTACH
  options JSONB,                                  -- 选项/矩阵定义
  required BOOLEAN,
  skip_logic JSONB,                              -- 跳题逻辑（答“不适用”跳过关联题）
  -- 打分/报告映射层（与结构解耦，但随题维护）：
  weight NUMERIC(6,3),                           -- 权重
  report_section_key VARCHAR(64),                -- 得分归入报告的哪个章节（需求4.2.2）
  severity_suggest VARCHAR(16)                   -- 判不合规时建议严重度（审计场景，仅参考）
);
```
- **两层解耦**：结构（section/item 的层级、题型、跳题）与 打分/报告映射（weight、report_section_key、scoring_model、report_format）分列存储，调整报告格式不牵连表单字段，反之亦然（需求 4.2.2/4.4）。
- **发布前测试**：`template_version.status=DRAFT` 时可预览/测试填写，验证通过才置 `EFFECTIVE`（需求 4.2.2 配置治理）。

### 7.3 评估实例与答案（启动即快照版本）

```sql
CREATE TABLE assessment (
  id BIGINT PRIMARY KEY,
  org_id BIGINT NOT NULL REFERENCES org(id),
  asset_id BIGINT REFERENCES asset(id),          -- 评估对象（来自台账）
  template_version_id BIGINT REFERENCES template_version(id), -- 固化的版本快照
  plan_type VARCHAR(16),                          -- PERIODIC周期/AD_HOC专项
  owner_user_id BIGINT,
  due_date DATE,
  status VARCHAR(16),                             -- FILLING/REVIEW/APPROVED/EFFECTIVE
  -- 抽取的可聚合结果列（供M9看板）：
  risk_score NUMERIC(8,2),
  risk_level VARCHAR(10),                         -- 五级 VERY_LOW/LOW/MID/HIGH/VERY_HIGH（极低/低/中/高/极高，见 CR-001-7）
  approved_by BIGINT, approved_at TIMESTAMPTZ,
  ...审计字段
);
CREATE TABLE assessment_answer (
  id BIGINT PRIMARY KEY,
  assessment_id BIGINT REFERENCES assessment(id),
  item_id BIGINT,                                -- 对应快照版本的题目
  value JSONB,                                   -- 弹性答案
  score NUMERIC(8,2),                            -- 抽取得分
  attachment_keys VARCHAR(256)[],                -- 佐证附件（MinIO）
  control_id BIGINT                              -- 冗余控件引用，支撑“一次测试多体系复用”
);
-- 控件测试结果（跨体系复用：同一控件被多体系评估，一次结果多处引用，需求4.2.2）
CREATE TABLE control_test_result (
  id BIGINT PRIMARY KEY,
  org_id BIGINT NOT NULL,
  asset_id BIGINT, control_id BIGINT,
  result VARCHAR(16),                            -- 符合/部分不符/不符合
  tested_at TIMESTAMPTZ, evidence_id BIGINT,
  reuse_window_days INT                          -- 复用有效期
);
```

### 7.4 报告生成
报告由 `scoring_model + report_format + report_section_key + answers` 驱动动态生成（需求 4.2.3），**不为每个体系单独开发报告逻辑**；人工复核签批后 `status=EFFECTIVE`；导出 Word/PDF 异步生成。

## 八、整改任务（M2/M3 共用工单）

> 风险点与审计发现**共用同一整改机制**（需求 4.2.4/5.2.1），避免两套流程。

```sql
CREATE TABLE remediation_task (
  id BIGINT PRIMARY KEY,
  org_id BIGINT NOT NULL,
  source_type VARCHAR(16),                        -- ASSESSMENT/AUDIT/KRI
  source_id BIGINT,                              -- 风险点/发现项/预警
  title VARCHAR(256),
  assignee_id BIGINT, due_date DATE, priority VARCHAR(8),
  state VARCHAR(16),                             -- 状态机：TODO/DOING/DONE/OVERDUE/VERIFIED
  verify_by BIGINT, verified_at TIMESTAMPTZ,
  proof_evidence_id BIGINT,                      -- 整改证明（验证后才能关闭，入留痕）
  ...审计字段
);
```
状态机：`TODO→DOING→DONE→(发起人验证)→VERIFIED`；超期自动 `OVERDUE` 并经 M10 通知。

## 九、审计管理（M3，含内审/外审/监管检查/认证审计，audit_type 区分）

```sql
CREATE TABLE audit_plan (
  id BIGINT PRIMARY KEY, org_id BIGINT, theme VARCHAR(128),
  audit_type VARCHAR(16),                         -- INTERNAL/EXTERNAL/REGULATORY/CERTIFICATION（CR-001-1）
  external_body VARCHAR(128),                     -- 外部审计机构/监管机关/认证机构（外审类）
  plan_start_date DATE,                           -- 计划开始时间（外审常有；为"计划临近提醒"依据）
  reminder_days INT[],                            -- 计划临近提醒提前天数（如 {15,10}）→ 触发 M10「外审计划临近」企微推送
  asset_id BIGINT, template_version_id BIGINT,   -- 检查表=复用模板引擎(usage=AUDIT)
  status VARCHAR(16), ...审计字段
);
CREATE TABLE audit_finding (
  id BIGINT PRIMARY KEY, audit_plan_id BIGINT,
  description TEXT,
  clause_ref VARCHAR(64),                        -- 依据条款（可引用M1制度条款，需求5.2.2）
  policy_clause_id BIGINT,                       -- 关联制度条款，报告自动带出编号
  severity VARCHAR(16),
  evidence_id BIGINT,                            -- 证据材料
  -- 整改复用 remediation_task（source_type=AUDIT）
  ...审计字段
```

## 十三、证据留痕库 & 看板（M9）⭐防篡改

```sql
-- 证据（写入即只读 + 哈希 + 保留期取最大值）
CREATE TABLE evidence (
  id BIGINT PRIMARY KEY,
  ref_type VARCHAR(24),                          -- ASSESSMENT/AUDIT/REMEDIATION/ATTESTATION/NOTIFICATION/KRI
  ref_id BIGINT,
  org_id BIGINT,
  file_key VARCHAR(256),                         -- MinIO 对象键
  sha256 CHAR(64) NOT NULL,                      -- 入库即生成校验值（需求11.3）
  size_bytes BIGINT,
  created_by BIGINT, created_at TIMESTAMPTZ DEFAULT now(),
  retention_until DATE NOT NULL,                 -- 保留期=各监管口径最大值（需求11.2.2）
  immutable BOOLEAN DEFAULT true,                -- 标记只读
  curr_hash   CHAR(64),                          -- 本条哈希链节点（按 org_id 分链，见 §23.13）
  prev_hash   CHAR(64)                           -- 同链前一条 curr_hash，构成防回插哈希链
);
-- 操作日志：独立、仅追加、不可删改（应用账号无 DELETE/UPDATE 权限）
CREATE TABLE operation_log (
  id BIGSERIAL PRIMARY KEY,
  actor_id BIGINT, action VARCHAR(64), target_type VARCHAR(32), target_id BIGINT,
  org_id BIGINT, detail JSONB, occurred_at TIMESTAMPTZ DEFAULT now(),
  curr_hash   CHAR(64),                          -- 本条哈希链节点（按 org_id 分链，见 §23.13）
  prev_hash   CHAR(64)                           -- 同链前一条 curr_hash，构成防篡改/防回插哈希链
);
-- 通过数据库权限/触发器禁止对 evidence、operation_log 的 UPDATE/DELETE
REVOKE UPDATE, DELETE ON evidence, operation_log FROM app_rw;
```
- **不可篡改不仅靠应用层**：数据库层回收应用账号的 UPDATE/DELETE 权限 + 触发器拦截（需求 11.4：不能仅依赖应用逻辑）。
- **哈希链（防篡改增强，A3/C13）**：`evidence`、`operation_log` 增 `curr_hash/prev_hash`，**按 `org_id` 分链、串行化写入**；可信时间戳集中存 §23.13 `trusted_timestamp`。链头计算、断链告警与离线重算校验作业见 §23.13。
- 看板（M9 11.2.1）多为**物化视图/聚合查询**，按组织/时间下钻，标注数据截止时间；不一定独立建表。

## 十六、通用设计约定

| 约定 | 规则 |
|---|---|
| 主键 | 雪花 ID（BIGINT）；高频追加日志可用 BIGSERIAL |
| 审计字段 | 每表含 `created_by/created_at/updated_by/updated_at` |
| 软删除 | 业务表 `status` 标记，不物理删除（留痕表禁删） |
| 并发 | 关键单据乐观锁 `version` 列 |
| 组织字段 | 业务表统一 `org_id`（隔离锚点），子树查询用 org.path |
| JSONB 用法 | 仅用于真异构/弹性结构（答案、规则、属性）；可聚合字段必须抽列 |
| 命名 | 表/列 snake_case；枚举值大写常量 |
| 时间 | 一律 `TIMESTAMPTZ`（UTC 存储） |
| 索引 | org_id、状态、时间、外键、看板聚合维度建索引；向量用 HNSW |

```sql
  at           TIMESTAMPTZ DEFAULT now(),
  repo_target  VARCHAR(32),                           -- GITHUB/INTERNAL_GIT 或外部端点标识
  content_hash CHAR(64),                              -- 出站内容哈希

);
```
- `feedback_repo_sync.egress_audit_id` 指向本表（有效目标）；亦可由 `operation_log` 复用承载，二选一须在 D1-3 出站网关一节固定口径。

### 23.10·补（功能链路补全） 调度/到期扫描台账（B 节内核 + C12 幂等）

```sql
CREATE TABLE audit_plan_reminder_log (
```

### 23.12·补（功能链路补全） levelMatrix 平台统一阈值基线（A1/G1）

```sql
-- 风险五级 = 平台级统一基线（模板不可改阈值；M9 跨子公司聚合按 risk_level 语义一致）
CREATE TABLE risk_level_matrix (
  id        BIGINT PRIMARY KEY,
  level     VARCHAR(10) NOT NULL,                     -- VERY_LOW/LOW/MID/HIGH/VERY_HIGH（极低/低/中/高/极高）
```

### 23.13·补（功能链路补全） 可信时间戳集中表 trusted_timestamp（C13/A3）

```sql
-- 集中存放被各模块引用的可信时间戳（operation_log/evidence/access_review 签认/审批动作等）
CREATE TABLE trusted_timestamp (
  id          BIGINT PRIMARY KEY,
  object_type VARCHAR(32) NOT NULL,                   -- EVIDENCE/OPERATION_LOG/ACCESS_REVIEW/APPROVAL…
  object_id   BIGINT NOT NULL,
  tsa_token   BYTEA NOT NULL,                         -- RFC3161 时间戳令牌
  ts_authority VARCHAR(128),                          -- 时间戳服务机构（TSA）
  at          TIMESTAMPTZ DEFAULT now()
);
```
- **哈希链作业**：`evidence/operation_log` 的 `curr_hash/prev_hash` **按 `org_id` 分链、串行化写入**；断链产告警，配套**离线重算校验作业**逐链回放校验。
- **保留期只增不减**：`evidence.retention_until` 重算 = `max(各关联条款保留期, 入库日 + 下限)`，单调只增。
- 注：§22.3 已在 `evidence` 上加 `tsa_token/ts_authority/prev_hash` 内联列；本表为**集中引用**版本，供 `operation_log`、`access_review` 签认、审批动作等共用，二者并存（证据可内联亦可集中登记）。
