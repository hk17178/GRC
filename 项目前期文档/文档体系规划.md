# GRC 平台 · 项目文档体系规划

**版本：** v1.9 ｜ **日期：** 2026-06-23 ｜ **依据：** 《GRC 平台建设需求文档 V1.9》+ CR-001/002/003/004

> 本文规划 GRC 平台从设计到运维全周期所需的文档清单、每份文档的目的与目录骨架、编制顺序与依赖关系。文档内容**针对本项目特征定制**（M1–M11 业务模块 + 系统管理（含 模型接入、建议与反馈 M-FB）、多域控 AD 集成、SIEM/KRI 接口采集、AI 数据隔离与「数据不出域」、表单引擎、证据留痕防篡改、自建爬虫等），非通用模板。

## 一、文档分层原则

1. **够用即可，不为文档而文档**：中型平台，避免重型瀑布文档堆砌；以"能指导开发、能通过监管/审计举证"为标准。
2. **风险驱动优先级**：可行性评估已识别三大高风险区——**M2 表单引擎、M5+M8 数据隔离、自建爬虫维护**；这些区域的设计文档**必须前置且评审**。
3. **合规可追溯**：本平台自身是合规系统，文档体系须支持**需求→设计→测试→验收**的全链路追溯（RTM），以备等保测评/PBOC 检查。
4. **与已有成果衔接**：需求规格（V1.9，已整合 CR-001~004）、前端设计规范（多主题令牌）、高保真原型**已就位**，后续文档在此基础上展开，不重复。

## 三、关键文档目录骨架

以下给出**优先级 P0/重点文档**的目录骨架，可直接据此展开成文。

### D1-1 总体架构设计文档（HLD）
1. 引言（目的、范围、参考、术语）
2. 架构目标与约束（性能/可用性 99.5%/安全/可扩展；非功能需求映射）
3. 总体架构
   - 3.1 分层架构（接入层 / 应用层 / 横向服务层 M6/M8/M10 / 数据层 / 集成层）
   - 3.2 模块划分与依赖（M1–M11 + 系统管理（模型接入、建议与反馈）关系图，复用关系：M3 审计管理复用 M2 表单引擎，M7 复用 M2，M5/M10 依托 M8）
   - 3.3 技术选型（前端框架、后端、工作流引擎建议选型 Flowable/Camunda、RAG/向量库、消息/任务队列、报表导出）
4. 关键架构决策（ADR）
   - 工作流引擎不自研；表单引擎"通用配置 + 内置脚手架"双轨；统一控件库颗粒度
5. 部署架构（环境、容器化、负载、子公司数据隔离的部署考量）
6. 横向服务设计概述（M6 组织台账 / M8 权限审批 / M10 通知中心 作为共用服务）
7. 非功能架构（性能、可用性、可维护性、可扩展性）
8. 风险与权衡

### D1-2 数据模型设计文档 ⭐地基
1. 数据建模原则（组织维度隔离为第一原则——需求 7.3"后期补救成本极高"）
2. 核心实体与 ER
   - 组织/子公司、身份源（identity_source）/用户（多域控 AD 同步，唯一键 (domain_id, username)）、角色、资产/对象台账、制度、评估、模板、控件、审计（audit_type 区分 内审/外审/监管检查/认证审计）、整改任务、法规、监管事项(M11)、供应商、通知记录、证据留痕
   - **反馈与模型元数据**：feedback / feedback_category / feedback_sync_log（建议与反馈 M-FB）、ai_provider / ai_model_config（模型接入与管理）
3. **数据隔离设计**（行级/组织级，user-角色-组织-数据 四元组落库方式）
4. 数据分类分级（公开/内部/敏感）与脱敏
5. 留痕库设计（防篡改：哈希、写入即只读、操作日志独立存储；保留期限取各监管口径最大值）
6. 版本化设计（制度版本、模板版本与继承、评估快照）
7. 自定义字段扩展机制（JSONB 扩展列 + 字段元数据登记；自定义视图/看板存储，CR-001-2）
8. 索引与性能、数据量增长与扩容
9. 数据字典

### D1-3 安全设计文档 ⭐高风险
1. 安全目标与合规基线（等保三级、PBOC、PIPL、ISO 27001）
2. **多域控身份与认证**（N 个 identity_source、各子公司/海外独立 AD 域/林；(domain_id, username) 唯一；DOMAIN\user 或 user@domain 认证路由；不依赖域间信任；单域故障仅影响该域并维持权限快照；SSO/账号生命周期/离职回收时效）
3. **授权模型（RBAC + 组织归属四元组、一人多角色取并集规则；各子公司管本域、集团聚合）**
4. **数据隔离与 AI 越权防护**（M5 检索阶段过滤、隔离边界、专项越权测试要求）
5. 传输/存储加密
6. 审计日志（不可篡改、留存期限、谁查了什么；AI 配置变更、反馈对外推送出站审计）
7. 证据留痕防篡改机制
8. Web 安全（越权/注入/敏感泄露）与上线安全测试
9. 外部集成凭证安全管理（密钥库统一管理，多域控 AD 服务账号、短信密钥、AI API Key、GitHub Token 均不落明文）
10. **密钥管理与「数据不出域」**：密钥库统一托管不落明文；模型接入「数据不出域」开关（默认开）仅允许内网模型/内网 API；外部模型端点出站白名单防 SSRF；反馈对外推送强制脱敏 + 管理员审批 + 出站审计（CR-004）

### D1-6 表单引擎专项设计 ⭐最高难点
1. 设计目标与"通用性 vs 易用性"权衡（需求 4.4）
2. 结构配置层（域→控制项→问题层级、题型、校验/跳题、多级填写流程）
3. 打分与报告映射层（权重、定量模型默认、定性可选、章节映射）
4. 两层解耦机制
5. 模板版本化与继承（启动时固化版本）
6. 内置体系脚手架（等保/ISO27001/PBOC/PCI/27701/27018/9001/供应商）
7. **统一控件库**（"具体控制点"颗粒度、跨体系映射、一次测试多体系复用、维护工具）
8. 配置治理（集中管控、分级开放；发布前预览测试）
9. 与 M3 审计检查表复用方案

```
需求V1.9(✅) ──┬─► D1-1 总体架构 ──┬─► D1-4 详细设计 ──► D2-1 开发计划 ──► 开发

              │                  └─► D1-6 表单引擎专项(最高难点,先原型验证)
设计规范(✅)/原型(✅) ─────────────────────────────────► UI 落地
D1-* 设计 ──► D3-1 测试计划 ──► D3-2/3/4/5 测试方案/用例
全程：DM-1 需求追溯矩阵(RTM) 随设计/用例同步维护
```

**关键先后**：
1. **数据模型(D1-2) + 安全设计(D1-3) 必须最先并评审**——组织隔离边界一旦定错，后期返工成本极高；
2. **表单引擎(D1-6) 建议先做技术原型验证**再定稿详细设计；
3. **接口设计(D1-5) 前置条件**是锁定 AD/SIEM/日志/漏洞/短信 五个外部系统的接口承诺；
4. 开发计划(D2-1) 依赖架构与详细设计基本成型后排期。

### 1. 业务模块与术语口径（统一）
- **业务模块扩为 M1–M11**：新增 **监管事项（M11）**——一级业务模块，含 报送日历（法定时限预警）/ 监管问询 / 处罚约谈台账 / 重大事件报送 / 年度合规计划（IR/SOC 主体不在本平台，仅"重大事件监管报送"在本平台）。
- **M3 由"内部审计"升级为"审计管理（内/外审）"**：以 `audit_type` 区分 内审 / 外审 / 监管检查 / 认证审计；外审对外回函三段漏斗（已提交外部机构 → 外方受理 → 外方确认关闭）；认证有效期跟踪；等保测评全过程视图。
- **系统管理新增两模块**：模型接入与管理（独立菜单）、建议与反馈（M-FB，系统管理菜单 + AI 助手双入口）。
- **风险五级**：极低 / 低 / 中 / 高 / 极高（VERY_LOW/LOW/MID/HIGH/VERY_HIGH），全文不再保留三级口径。
- **菜单顺序（V1.9 权威口径，其余文档引用此处、不各自维护）**：业务区＝**外部审计 → 内部审计 → 风险评估 → 法规跟踪 → 监管事项 → 合规清单 → 制度体系 → AI 智能问答 → 第三方供应商**；系统管理区＝**通知中心 → 模型接入 → 权限与审批 → 看板与留痕 → 建议与反馈 → 系统设置**。
  - **外部审计与内部审计为两个独立一级菜单（不合并）**：数据层统一为「审计管理」、以 `audit_type` 区分内审/外审/监管检查/认证。
  - **合规清单（原合规义务清单 / Obligation Register）为独立一级业务菜单**：数据归属 M4，承载"法规条款→义务→责任部门→制度/控制点→证据→满足状态"端到端举证链、横跨 M1/M3/M4/M9，置于监管事项与制度体系之间。
  - **模型接入**（原称 AI 接入，即模型接入与管理）为系统管理独立菜单。
  - "合规态势总览"统一为"合规态势"；产品 UI 不出现实现术语（M2/四元组/Flowable 等）。

### 3. 多域控身份集成落点（CR-003）
- **D1-3**：身份与认证扩为**多域控**——N 个 identity_source（各子公司/海外独立 AD 域/林）；唯一键 (domain_id, username)；DOMAIN\user 或 user@domain 认证路由；不依赖域间信任；单域故障仅影响该域并维持权限快照。
- **D1-5**：AD 接口标注为 **N 身份源、按域路由、域内唯一**（仍为 AD → 平台单向同步，由"单域"扩展为"多域"，不改隔离基本原则）。
- **D1-2**：新增 identity_source 表，app_user 增 domain_id/upn，唯一键改 (domain_id, username)。
