# GRC 平台 · 项目文档体系规划

**版本：** v1.0 ｜ **日期：** 2026-06-22 ｜ **依据：** 《GRC 平台建设需求文档 V1.8》

> 本文规划 GRC 平台从设计到运维全周期所需的文档清单、每份文档的目的与目录骨架、编制顺序与依赖关系。文档内容**针对本项目特征定制**（M1–M10 模块、AD/SIEM 集成、AI 数据隔离、表单引擎、证据留痕防篡改、自建爬虫等），非通用模板。

---

## 一、文档分层原则

1. **够用即可，不为文档而文档**：中型平台，避免重型瀑布文档堆砌；以"能指导开发、能通过监管/审计举证"为标准。
2. **风险驱动优先级**：可行性评估已识别三大高风险区——**M2 表单引擎、M5+M8 数据隔离、自建爬虫维护**；这些区域的设计文档**必须前置且评审**。
3. **合规可追溯**：本平台自身是合规系统，文档体系须支持**需求→设计→测试→验收**的全链路追溯（RTM），以备等保测评/PBOC 检查。
4. **与已有成果衔接**：需求规格（V1.8）、前端设计规范（多主题令牌）、高保真原型**已就位**，后续文档在此基础上展开，不重复。

## 三、关键文档目录骨架

以下给出**优先级 P0/重点文档**的目录骨架，可直接据此展开成文。

### D1-1 总体架构设计文档（HLD）
1. 引言（目的、范围、参考、术语）
2. 架构目标与约束（性能/可用性 99.5%/安全/可扩展；非功能需求映射）
3. 总体架构
   - 3.1 分层架构（接入层 / 应用层 / 横向服务层 M6/M8/M10 / 数据层 / 集成层）
   - 3.2 模块划分与依赖（M1–M10 关系图，复用关系：M3 复用 M2 表单引擎，M7 复用 M2，M5/M10 依托 M8）
   - 3.3 技术选型（前端框架、后端、工作流引擎建议选型 Flowable/Camunda、RAG/向量库、消息/任务队列、报表导出）
4. 关键架构决策（ADR）
   - 工作流引擎不自研；表单引擎"通用配置 + 内置脚手架"双轨；统一控件库颗粒度
5. 部署架构（环境、容器化、负载、子公司数据隔离的部署考量）
6. 横向服务设计概述（M6 组织台账 / M8 权限审批 / M10 通知中心 作为共用服务）
7. 非功能架构（性能、可用性、可维护性、可扩展性）
8. 风险与权衡

### D1-2 数据模型设计文档 ⭐地基
1. 数据建模原则（组织维度隔离为第一原则——需求 7.3"后期补救成本极高"）
2. 核心实体与 ER
   - 组织/子公司、用户（AD 同步）、角色、资产/对象台账、制度、评估、模板、控件、审计、整改任务、法规、供应商、通知记录、证据留痕
3. **数据隔离设计**（行级/组织级，user-角色-组织-数据 四元组落库方式）
4. 数据分类分级（公开/内部/敏感）与脱敏
5. 留痕库设计（防篡改：哈希、写入即只读、操作日志独立存储；保留期限取各监管口径最大值）
6. 版本化设计（制度版本、模板版本与继承、评估快照）
7. 索引与性能、数据量增长与扩容
8. 数据字典

### D1-3 安全设计文档 ⭐高风险
1. 安全目标与合规基线（等保三级、PBOC、PIPL、ISO 27001）
2. 身份与认证（AD 对接、SSO、账号生命周期/离职回收时效）
3. **授权模型（RBAC + 组织归属四元组、一人多角色取并集规则）**
4. **数据隔离与 AI 越权防护**（M5 检索阶段过滤、隔离边界、专项越权测试要求）
5. 传输/存储加密
6. 审计日志（不可篡改、留存期限、谁查了什么）
7. 证据留痕防篡改机制
8. Web 安全（越权/注入/敏感泄露）与上线安全测试
9. 外部集成凭证安全管理（AD 服务账号、短信密钥不明文）

### D1-6 表单引擎专项设计 ⭐最高难点
1. 设计目标与"通用性 vs 易用性"权衡（需求 4.4）
2. 结构配置层（域→控制项→问题层级、题型、校验/跳题、多级填写流程）
3. 打分与报告映射层（权重、定量模型默认、定性可选、章节映射）
4. 两层解耦机制
5. 模板版本化与继承（启动时固化版本）
6. 内置体系脚手架（等保/ISO27001/PBOC/PCI/27701/27018/9001/供应商）
7. **统一控件库**（"具体控制点"颗粒度、跨体系映射、一次测试多体系复用、维护工具）
8. 配置治理（集中管控、分级开放；发布前预览测试）
9. 与 M3 审计检查表复用方案

### D2-1 开发计划（项目计划）
1. 建设目标与范围、分期策略（**强烈建议分期**，见可行性评估）
   - 一期：M6+M8+M1+M2 基础表单+M9+M10（纵向闭环）
   - 二期：M5 AI 问答（含隔离测试）+ M2 控件库 + M3 审计 + M1 知识库 RAG
   - 三期：M4 爬虫 + M7 供应商 + M2 KRI 持续监控
2. WBS（工作分解，到可估算粒度）
3. 里程碑与排期（甘特）
4. 团队与角色分工、资源
5. 外部依赖锁定（AD/SIEM/日志/漏洞/短信 接口就绪时点）
6. 依赖与关键路径
7. 风险与应对（联动风险登记册）

### D3-1 测试计划（Test Plan）
1. 测试目标与范围（M1–M10 + 非功能 + 集成 + 数据迁移）
2. 测试策略与级别（单元/集成/系统/UAT；自动化范围）
3. 测试类型（功能、接口、安全、性能、兼容、回归）
4. 环境与数据（含脱敏的测试数据、多子公司样本）
5. 进度与资源、角色
6. **出入口准则**（进入/退出标准，缺陷分级与收敛标准）
7. 专项测试纳入：**AI 数据隔离越权测试**、留痕防篡改校验、爬虫失效告警、KRI 阈值触发
8. 风险与度量指标

### D3-2 测试方案与用例（Test Spec）
1. 用例设计方法与规范
2. 按模块用例集（M1–M10），每条关联需求编号（支撑 RTM）
3. 关键场景用例
   - 跨子公司越权访问（必须失败）
   - 同一控件多体系复用、模板版本固化
   - 整改闭环（验证后才关闭）、逾期通知分级
   - 制度签署确认闭环、重新确认触发
   - 法规变更摘要推送
4. 接口/集成用例、性能用例、兼容性用例
5. 缺陷管理与回归策略

### D3-3 安全测试方案（专项）
1. 范围：越权（水平/垂直）、注入、敏感信息泄露、认证会话
2. **AI 问答数据隔离越权专项**（A 子公司账号尝试获取 B 子公司数据，全部应拒绝并留日志）
3. 权限四元组边界测试（角色×组织组合矩阵）
4. 留痕不可篡改验证（哈希校验、删除/改写防护）
5. 凭证安全、传输/存储加密验证
6. 上线前渗透测试要求与报告模板（对接等保测评格式）

## 四、编制顺序与依赖（路线图）

```
需求V1.8(✅) ──┬─► D1-1 总体架构 ──┬─► D1-4 详细设计 ──► D2-1 开发计划 ──► 开发
              │                  ├─► D1-2 数据模型(地基,先行) 
              │                  ├─► D1-3 安全设计(与数据模型并行,前置评审)
              │                  ├─► D1-5 接口设计(需先锁定外部依赖)
              │                  └─► D1-6 表单引擎专项(最高难点,先原型验证)
设计规范(✅)/原型(✅) ─────────────────────────────────► UI 落地
D1-* 设计 ──► D3-1 测试计划 ──► D3-2/3/4/5 测试方案/用例
全程：DM-1 需求追溯矩阵(RTM) 随设计/用例同步维护
```

**关键先后**：
1. **数据模型(D1-2) + 安全设计(D1-3) 必须最先并评审**——组织隔离边界一旦定错，后期返工成本极高；
2. **表单引擎(D1-6) 建议先做技术原型验证**再定稿详细设计；
3. **接口设计(D1-5) 前置条件**是锁定 AD/SIEM/日志/漏洞/短信 五个外部系统的接口承诺；
4. 开发计划(D2-1) 依赖架构与详细设计基本成型后排期。
