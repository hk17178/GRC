## 一、引言

### 1.1 目的
定义平台的安全架构与控制措施。**最高优先级安全约束是组织维度数据隔离与 AI 问答越权防护**（需求 7.3 明确：后期补救成本极高，需优先评审）。

### 1.2 合规基线
平台须满足并支撑以下监管/标准（本身也是被这些体系评估的对象）：

| 基线 | 关注重点 |
|---|---|
| 等保三级 | 身份鉴别、访问控制、安全审计、数据保密性/完整性、入侵防范 |
| PBOC（非银行支付机构监管） | 敏感数据保护、操作留痕、最小权限 |
| 个人信息保护法（PIPL） | 个人信息最小化、访问控制、留痕、脱敏 |
| ISO 27001 | 访问控制、密码学、日志监控、供应商安全 |
| PCI DSS | 持卡人数据（CHD）字段级加密、CDE（持卡人数据环境）网络/数据隔离、密钥管理、访问最小化与审计 |
| ISO 27701 / ISO 27018 | 隐私信息管理（PIMS）：个人信息处理记录（ROPA）、数据主体权利、跨境与公有云个人数据保护 |

### 1.3 安全设计原则
最小权限 · 纵深防御 · 默认拒绝 · 职责分离 · 不可篡改留痕 · 凭证不落明文 · 安全左移（设计阶段评审）。

## 二、安全架构总览（纵深防御）

```
① 网络/接入：TLS 强制 · 反向代理 · WAF(可选) · 限流
② 认证：AD/LDAP 鉴别 · SSO · 会话安全(Redis,过期/单点失效)
③ 授权：RBAC + 组织四元组 · 默认拒绝 · 多角色并集
④ 数据隔离：统一数据访问拦截 + PostgreSQL 行级安全(RLS)兜底
⑤ AI 越权防护：检索阶段强制组织过滤 + 引用溯源 + 访问日志
⑥ 数据保护：传输/存储加密 · 分类分级 · 脱敏
⑦ 留痕审计：操作日志不可篡改 · 证据哈希 · 数据库层禁删改
⑧ 应用安全：OWASP Top 10 防护 · 文件上传安全 · 上线安全测试
⑨ 凭证管理：密钥库 · 外部对接凭证不明文
⑩ 安全运营：监控告警 · 越权检测 · 应急响应
```

## 三、身份与认证

### 3.1 AD 对接（需求 8.2、13.3）
- 用户账号、组织、部门由 **AD 单向同步**（**多域控集成场景见 §3.1.1：平台同时对接 N 个独立 AD 域/林，各身份源单向同步至对应组织子树**）；平台不独立维护账号体系。
- 认证：用户登录经 AD/LDAP 校验（或 SSO 重定向）；支持与公司现有 SSO 对接（若有）。
- 同步：定时增量同步（基于 `ad_object_id`/变更游标），**同步频率在设计阶段定值**（需求 8.3/10.4）。

#### 3.1.1 多域控 / 多域集成（CR-003 增补，关键）
> 集团下各子公司可能各自运行**独立的 AD 域 / 林**（含海外子公司的独立域）。平台**必须支持同时对接多个域控**，并将每个身份源映射到对应组织，实现跨子公司隔离的统一用户与角色权限管理。
- **多身份源配置**：以"身份源（identity source）"为单位配置 N 个域控连接——每个含 域名/林、LDAP(S) 地址（可多控制器做高可用）、独立服务账号（独立密钥管理）、Base DN、同步字段映射与同步频率、绑定的 `org_id`（一个身份源 → 一个或多个子公司组织子树）。
- **账号唯一性**：用户唯一键由"全局 username"改为 **(domain_id, username)** 复合，或采用 **userPrincipalName（user@domain）/ 域限定名（DOMAIN\user）** 作为登录标识；避免不同域同名账号冲突。登录页域账号输入即支持 `DOMAIN\用户名` 或 `user@corp.com`（原型已体现）。
- **认证路由**：登录时按"域选择 / UPN 后缀 / 域前缀"解析目标域，路由到对应域控校验；各域**密码策略由其自身 AD 承载**。SSO 场景支持多 IdP（按子公司）。
- **组织映射与隔离**：各域同步的组织树挂到其对应子公司 `org_id` 子树下；跨域用户天然落在不同 `org_id`，沿用 §5 组织隔离与 §6 RBAC 四元组——**各子公司管理本域用户与角色，集团管理跨域聚合视图**。
- **跨域信任 / 重名 / 去重**：不依赖域间信任关系（各域独立对接即可）；同一自然人在多域有账号时以 `ad_object_id + domain_id` 区分，不自动合并（如需主数据合并由 M6 人工关联）。
- **可用性与降级**：单个域控不可用仅影响该域用户的登录与同步，其余域不受影响；同步中断维持该域现有权限快照，恢复后增量补同步（见 D4-3）。

#### 3.1.1·补（功能链路补全 · A2/D2/D3）单域故障降级 = 安全优先

> 某子公司 AD 域控故障期间，认证与同步降级处理遵循「安全优先」原则——**宁可拒绝新登录，不可放过越权写**。本节为 §3.1.1「可用性与降级」的可落地化展开，写入 §3.2 / §4.4，并与 D4-3 §3.6 应急预案衔接。

- **禁新登录、仅维持已有会话**：故障域期间**禁止该域新登录**（无可信鉴别源）；仅维持已建立的有效会话，且**缩短会话 TTL**（强制刷新窗口收紧），到期不再续期。其余正常域不受影响。
- **禁敏感写**：故障窗口内对该域 `org_id` 范围**禁止敏感写操作**——评估提交、审批动作、制度发布、数据导出一律拒绝（返回降级提示），只读与非敏感操作可继续。敏感写清单与 §3.3 关键操作、§8.1 审计动作保持一致。
- **应急手动停用通道（不依赖 AD 同步）**：提供独立于 AD 同步的「应急手动停用」入口——置 `platform_disabled=true`（见 §4.7·补 / D2）并**立即杀会话**（销毁该账号 Redis 权限快照与会话），用于域控不可用期间需紧急回收某账号权限的场景；此通道**不写 `app_user.status`**（写源仍是 AD），仅置平台侧 `platform_disabled` 位。
- **恢复后回放与回溯校验**：域控恢复后增量补同步，**优先回放 `DISABLED` 变更**（离职/停用先生效，避免「死人复活」窗口），并对**故障窗口内的高敏操作回溯校验**（结合 §8 哈希链与 `operation_log` 比对，发现故障期违规写入即告警）。
- **鉴权取值**：故障期与恢复期鉴权一律按 `effective_disabled = (AD status=DISABLED) OR platform_disabled` 判定（见 §4.7·补 D2），AD 同步不覆盖 `platform_disabled` 位。

### 3.2 账号生命周期与权限回收（关键）
| 事件 | 处理 | 时效要求 |
|---|---|---|
| 入职 | 同步新增 `app_user`（无业务角色，需另行授权） | 同步周期内 |
| 调岗 | 更新 `org_id` → 可见组织集合随之变化 | 同步周期内 |
| 离职 | `status=DISABLED` → **会话即时失效 + 角色权限回收** | 须明确（建议准实时/≤同步周期，需求 10.5） |
- AD 同步延迟期间的风险（用过期组织信息判隔离）需通过"同步时效 + 关键操作二次校验"缓解（需求 8.3）。
- **平台侧临时禁用（不改 AD 写源）**：UAR「回收」、应急手动停用（§3.1.1·补）等平台侧动作**仅置 `platform_disabled` 标志位并解绑/降权 `user_role_org`**，不改由 AD 同步的 `app_user.status`；鉴权统一取 `effective_disabled = (AD status=DISABLED) OR platform_disabled`，AD 同步回放**不覆盖** `platform_disabled` 位（详见 §4.7·补 D2）。
- **单域故障降级**：故障域期间禁新登录、仅维持有效会话且缩短 TTL、禁敏感写，恢复后优先回放 `DISABLED`（详见 §3.1.1·补 · A2）。

### 3.3 会话安全
- 会话状态外置 Redis（支撑应用无状态/多实例，见 D1-1）。
- 会话过期、空闲超时、单点登出（账号停用即销毁会话）、并发会话策略可配。
- 关键操作（评估提交、审计发现、整改验证、签署确认）防 CSRF + 防重复提交。

#### 3.3.1·补（功能链路补全 · D4）权限快照（permission snapshot）落地

> 「权限快照」= **登录时固化的鉴权上下文对象**，避免每次请求回查 AD/DB，同时保证可控失效。与 §4.4 `visibleOrgs` 缓存为同一份缓存的不同视图（快照内含 `visibleOrgs`）。

- **定义与结构**：登录成功时生成 `permission_snapshot = { user_id, domain_id, 角色集 roles[], visibleOrgs[](§4.4 计算结果), is_group_level, platform_disabled, generated_at }`，序列化存 **Redis**，键带 `session_id`、设**独立 TTL**（与会话 TTL 解耦，便于单独失效与降级期收紧）。
- **鉴权读取**：请求鉴权与 RLS 会话变量注入（`SET app.visible_orgs`，§4.4）均读快照，不回查 AD；`effective_disabled` 实时叠加快照内 `platform_disabled` 与 AD `status`。
- **生成/失效规则**：①登录生成；②角色授权变更 / `user_role_org` 变更 / UAR 回收降权 → 失效相关用户快照并下次请求重算；③组织树变更（新增 `identity_source` / 子公司组织子树同步完成）→ 失效受影响快照；④`platform_disabled` 置位或应急手动停用 → 立即删除快照 + 杀会话；⑤单域故障降级期 → 缩短该域快照 TTL。
- **集团级用户的 visibleOrgs**：`is_group_level=true` 用户的 `visibleOrgs` 为「全集团」，在「新增 `identity_source` / 子公司组织树同步完成」时失效重算，或在查询期**动态展开**为当前全部 `org_id`（二者择一并固化），避免新子公司加入后集团视图遗漏。

## 四、授权模型（RBAC + 组织四元组）⭐

### 4.1 四元组
权限判断精确到 `用户 × 角色 × 组织 × 数据`（需求 10.4）。其中前三确定**可见组织集合**，第四由数据行 `org_id` 落地（D1-2 §3）。

### 4.2 角色体系
- 以需求 10.2 的 **20 个角色为基础集**（平台/制度/评估整改/审计/合规法规/供应商/管理层/AI 等域），随监管事项/模型接入/建议与反馈（出站审批）等新模块**按需扩展、数量不固定**（与 D1-2 §22.6 一致）。
- `role.is_group_level=true` 的角色（集团管理层、只读审阅）可见全集团；其余受本子公司约束。

### 4.3 一人多角色合并规则（需求 10.5）
- 同一用户多角色：**权限取并集**；数据范围取各角色对应组织范围的并集。
- 规则在设计阶段固化，避免歧义；冲突场景（如某角色只读 vs 另一角色可写同一资源）以"更高权限生效（并集）"为默认，特殊敏感操作可配置为"按场景最小化"。
- **与 SoD 的关系（交叉引用 §4.7）**：并集为默认且**优先生效**；属于 SoD 冲突的敏感组合**不在此处阻断**——`enforce_mode=DETECT` 的仅登记冲突待例外审批，`enforce_mode=BLOCK` 的高敏组合才硬阻断（详见 §4.7·补 D6）。

### 4.4 可见组织集合的计算
```
visibleOrgs(user) =
  if 任一角色 is_group_level → 全集团
  else ∪ { 角色绑定的 org 及（按角色定义）其子树 }   // 来自 user_role_org
```
- 计算结果缓存到会话；组织变更/授权变更时失效重算。**`visibleOrgs` 缓存是 §3.3.1 权限快照的一部分**（快照内 `visibleOrgs[]` 字段），二者共用同一失效规则，不再各存一份。
- 注入到数据库会话变量供 RLS 使用：`SET app.visible_orgs = '{1,12,120}'`。
- **服务端强制注入、缺失即拒**：`visibleOrgs` 一律由服务端从权限快照注入，**禁止由前端传入或裸连接绕过**；统一数据访问层在执行任何业务/自定义查询前必须已 `SET app.visible_orgs`，未注入即拒绝执行（写成测试断言，见 §5.4·补 / §十一）。

### 4.5 角色管理（需求 10.5）
- 提供角色新增/停用/权限范围调整界面；角色细分但配套矩阵维护工具，避免角色膨胀失控。
- 权限变更本身入 `operation_log`。

### 4.6 审批职责分离
- 多级审批（Flowable）：如 部门级审核人 → 信息安全负责人 → 集团管理层；支持加签/转办/超时提醒（需求 10.4）。
- 制度发布、评估报告确认、整改签批等关键动作与**发起人分离**（不可自批）。

### 4.7·补（V1.9 对齐 · CR-001/CR-003）用户访问复核（UAR）与职责分离（SoD）

**用户访问复核（User Access Review, UAR）** —— 周期性（建议季度/半年）对账号与权限做合规复核，满足等保/ISO 27001 A.9/PCI DSS 访问审查要求：
- **复核范围**：按子公司/身份源（多域 `domain_id`）拉取 `app_user × 角色 × 可见组织（四元组）`清单；重点覆盖特权角色（平台/集团管理层、只读审阅）、跨子公司可见账号、长期未登录账号。
- **复核人**：各子公司管理员复核本域账号，集团信息安全/合规复核跨域聚合（沿用 §3.1.1「各子公司管本域、集团聚合」）；复核人与被复核账号的授权发起人分离。
- **复核结论**：每条账号-权限给出「**保留 / 回收 / 降权**」决定（落 `access_review_item.decision ∈ {KEEP/REVOKE/DOWNGRADE}`）；判定「回收」即驱动角色解绑/降权，联动 §3.2 权限回收时效。**回收口径见下方 D2**。
- **签认证据**：复核记录（复核轮次、复核人、对象、结论、时间）入库不可篡改并生成签认证据（`signed_hash` + `tsa_token`，链入 §8.5 哈希链/TSA），作为监管举证；一轮 UAR 形成可导出的复核报告。

**§4.7·补（功能链路补全 · D2）UAR「回收」的写源边界 —— 不双写 AD：**
- UAR 判定「回收 / 降权」**仅对平台自管的 `user_role_org` 解绑或降权**（平台为该表唯一写源），并在需要平台侧整账号停用时**置独立标志位 `platform_disabled=true`**；**绝不改由 AD 同步的 `app_user.status`**（AD 才是 status 的写源），避免平台改动被下一次 AD 同步覆盖、或与 AD 实际在职状态冲突。
- 鉴权统一取 `effective_disabled = (AD status=DISABLED) OR platform_disabled` 的「或」；AD 增量同步回放**不覆盖** `platform_disabled` 位（仅维护 `status`）。回收动作触发 §3.3.1 权限快照失效与杀会话。

**职责分离（SoD）冲突检测 = 检测型（detect-by-default）** —— 维护 SoD 冲突规则矩阵 `sod_rule`（如「评估发起 vs 评估报告确认」「制度起草 vs 制度发布」「授权申请 vs 授权审批」不可同一人）：

**§4.7·补（功能链路补全 · D6）SoD 检测型语义与并集交叉引用：**
- **检测型为默认**：角色授予/合并（§4.3 并集）及 UAR 阶段命中 SoD 冲突时**不阻断角色授予、并集照常生效**，仅**登记冲突待例外审批留痕**（记冲突清单 + 例外审批 `sod_exception`）；这是与 §4.3「权限取并集」一致的设计——并集生效在前，冲突登记在后。
- **硬阻断为单配项**：仅当某条规则显式配置 `sod_rule.enforce_mode=BLOCK`（高敏组合）时才硬阻断角色授予/并集生效；其余规则 `enforce_mode=DETECT`。不可自批规则（§4.6）为 SoD 的关键子集，通常配 `BLOCK`。
- **交叉引用**：§4.3 一人多角色并集段在「冲突场景」处指向本节（§4.7 SoD）——并集默认生效，敏感组合的阻断/例外由 `sod_rule.enforce_mode` 与例外审批统一裁决，不在 §4.3 内另写一套。

**权限变更审计** —— 角色新增/停用、权限范围调整、UAR 回收/降权、SoD 例外批准等**全部入 `operation_log`**（沿用 §4.5、§8.1），记录变更前后值（before/after）、操作人与审批人，支持事后追溯与按对象下钻。

## 五、数据隔离设计 ⭐（最高优先级）

### 5.1 双层强制（纵深防御）
| 层 | 机制 | 作用 |
|---|---|---|
| 应用层 | **统一数据访问拦截**：MyBatis/JPA 拦截器自动为业务查询追加 `org_id ∈ visibleOrgs`；业务代码不手写隔离 | 主防线，覆盖全部业务查询 |
| 数据库层 | **PostgreSQL 行级安全（RLS）** 对敏感表启用，基于会话变量 `app.visible_orgs` | 兜底：应用漏判时数据库仍拦截 |

```sql
ALTER TABLE assessment ENABLE ROW LEVEL SECURITY;
CREATE POLICY p_org_iso ON assessment
  USING (org_id = ANY (current_setting('app.visible_orgs', true)::bigint[]));
-- audit_finding / remediation_task / evidence / kb_chunk 等敏感表同样启用
```

#### 5.1.1·补（功能链路补全 · D7）ext 宿主表全量 RLS + 自定义查询隔离约束

> 自定义字段（`ext JSONB`，D1-2 §22.1）会被自定义视图/报表/KPI 动态查询消费，若宿主表漏配 RLS 或动态查询裸连，将绕过隔离主防线。本节把 RLS 显式拉齐到**所有承载 `ext` 的 `org_id` 业务表**，并约束动态查询通道。

- **ext 宿主表全量 ENABLE RLS**：凡承载 `ext JSONB` 的 `org_id` 业务表——`asset` / `vendor` / `policy`（含 `policy_version`）/ `audit_plan` / `assessment` / `obligation` / `reg_filing` / `reg_inquiry` 等——**一律显式 `ENABLE ROW LEVEL SECURITY` 并建 `p_org_iso` 策略**（同上模板，`org_id = ANY(app.visible_orgs)`），不得遗漏；新增 ext 宿主表纳入此清单为强制项（写成建表 checklist 与 §十一 测试断言）。
- **自定义查询统一走数据访问层**：自定义视图 / 自定义报表 / KPI 的**动态查询**（D1-2 §22.1、custom_view_def/custom_report_def/kpi_def）**必须经统一数据访问层注入 `visibleOrgs`，或运行在已 `SET app.visible_orgs` 的 RLS 会话**中；
- **禁裸连接 / 禁超级用户连接**：动态查询**禁止使用裸数据库连接或具 `BYPASSRLS`/超级用户权限的连接**执行（否则 RLS 失效）；运行账号须为受 RLS 约束的普通应用账号。该约束写成**测试断言**——以未注入 `app.visible_orgs` 的会话执行自定义查询应返回空集或拒绝，以超级用户连接执行应被流程禁止（详见 §十一）。
- **生成列同隔离**：`is_aggregatable=true` 的 ext 键物化为 PostgreSQL 生成列（`GENERATED ALWAYS AS (ext->>'key')`，D1-2 §22.1）后，所在行仍受同表 RLS 约束，M9 看板物化视图刷新读这些列时同样在 RLS 会话内执行。

### 5.2 隔离边界（需求 7.3，设计阶段定死）
- 隔离维度 = **子公司（org_type=SUBSIDIARY）** 为主，部门为辅。
- 跨子公司访问：默认**全部拒绝**；唯集团管理层/只读审阅角色例外。
- 汇总统计（M9 看板）：集团层可见聚合，但明细按需脱敏（§7.3）。
- **集团级共享制度/知识的显式可见规则（D5）**：`org_type=GROUP` 且 `level=集团级` 的共享制度/知识库内容对全集团子公司**显式可见**——在 RLS 策略、`kb_chunk` 过滤、M5 检索三处**同步放宽**为 `org_id = ANY(app.visible_orgs) OR (org_type=GROUP AND level=集团级)`，避免集团统一发布的制度被组织隔离误拦。放宽仅限「集团级共享」标记内容，不波及子公司私有数据。

### 5.3 越权防护测试要求
- 水平越权（A 子公司访问 B 子公司数据）、垂直越权（低权限角色越权操作）均须专项测试，**默认期望：拒绝 + 留日志**（详见 §十一、D3-3）。

## 六、AI 问答数据隔离与越权防护（M5）⭐安全红线

> 需求 7.3 明确："本模块核心难点是权限控制而非问答技术本身"。

### 6.1 检索阶段强制过滤
- M5 在向量/全文**检索阶段**即按 `visibleOrgs` 过滤 `kb_chunk`、评估/审计记录（D1-2 §14）；**不在生成后再过滤**（生成后过滤无法防止上下文泄露）。
- 权限判断**复用 M8**，绝不在 AI 模块内另写一套（需求 7.3）。
- 多轮对话上下文同样受会话的 `visibleOrgs` 约束，不缓存越权数据。

#### 6.1.1·补（功能链路补全 · D5）检索隔离可验证挂载点 + 集团共享放宽

- **过滤与近邻同一 SQL（先过滤后近邻）**：pgvector 检索把组织过滤与 `top_k` 近邻写在**同一条 SQL** 里——`WHERE org_id = ANY(:visibleOrgs) [OR 集团级共享放宽] ORDER BY embedding <=> :q LIMIT :top_k`，**先按 `visibleOrgs` 过滤、再取近邻**，杜绝「先全库近邻再过滤」导致的候选泄露与召回失真。
- **服务端强制注入、缺失即拒**：`visibleOrgs` 由服务端从权限快照（§3.3.1）注入到该 SQL，**禁止前端传入**；服务端若检出 `visibleOrgs` 缺失/为空即**拒绝检索**（不退化为全库查询）。该挂载点写成可验证断言（§十一 AI 越权专项）。
- **集团级共享可见规则同步**：M5 检索的 `WHERE` 与 §5.2 集团共享放宽、`kb_chunk` 过滤**三处一致**——集团级共享制度/知识（`org_type=GROUP AND level=集团级`）显式可见，子公司私有内容仍严格隔离。
- **权限判断复用 M8**：上述过滤的 `visibleOrgs` 来源与 §6.1「复用 M8」一致，AI 模块内不另写隔离逻辑。

### 6.2 引用溯源与防幻觉
- 回答须标注来源（具体制度条款编号/评估记录编号），无依据不生成（需求 7.2/5）。
- 知识库未覆盖时明确告知而非编造（需求 3.4）。
- 准确性验证机制（抽样人工复核），避免幻觉导致合规误导（需求 7.4）。

### 6.3 访问日志
- 涉敏感数据的问答记 `operation_log`：谁、何时、查询了什么、命中哪些记录（需求 7.4），支持事后审计。

### 6.4 模型侧安全
- 优先内网部署模型（vLLM/Ollama）或内网 API，敏感数据**不出域**（由 §10.1「数据不出域」可审计强制开关落地，默认开）；该开关为**条件式互斥**分支——**开=外部端点直接拒绝（白名单不参与）；关=允许外部端点但出站必须命中白名单否则拒绝**（详见 §10.1·补 D1，不再用「开启时…白名单」并列措辞）；
- 提示注入（prompt injection）防护：检索结果与用户输入分隔，限制工具/数据访问范围在 `visibleOrgs` 内。

| 备份 | 备份文件加密存储 |

### 7.3 脱敏（需求 14.1）

### 7.4·补（V1.9 对齐 · PIPL / ISO 27701 / PCI DSS）个人信息（PI）与持卡人数据（CHD）专项

针对个人信息（PI）与持卡人数据（CHD）的强化控制，对齐 PIPL / ISO 27701/27018 / PCI DSS：
- **字段级加密与脱敏**：对 PI / CHD 字段（如证件号、卡号 PAN）做字段级加密存储与场景化脱敏展示（按角色与用途控制可见粒度）；卡号等遵循 PCI DSS 仅留必要字段、禁止存储敏感认证数据（CVV 等）。
- **跨境 / 出境访问审计**：海外子公司（多域，§3.1.1）涉及 PI 跨境处理或访问时，单独记录跨境访问审计（访问人/域/数据/用途/时间），支撑 PIPL 跨境合规与 ISO 27018 公有云个人数据留痕。
- **CDE 隔离**：持卡人数据环境（CDE）在网络与数据层与一般业务隔离（沿用 §五组织隔离 + RLS 兜底，并在 CDE 维度额外收敛访问面），最小化进入 CDE 的账号与组件。
- **处理记录（ROPA）**：登记个人信息处理活动记录（处理目的、数据类别、接收方、跨境情况、留存期），对齐 ISO 27701 ROPA 与 PIPL 要求，可导出供监管核查。

## 八、审计日志与不可篡改留痕

### 8.1 操作审计日志（需求 13.3）
- 记录"谁、何时、对什么数据、做了什么操作"；覆盖敏感数据访问、配置变更、权限变更、导出。
- **不可篡改**：`operation_log` 仅追加；应用账号无 `UPDATE/DELETE` 权限（D1-2 §13）；可选 WORM 存储/外部归档。

### 8.2 证据留痕防篡改（需求 11.2.2/11.4）
- 入库即生成 `sha256`；写入即只读（`immutable`）；数据库层回收对 `evidence` 的改删权限 + 触发器拦截。
- 不仅依赖应用逻辑（需求 11.4 明确）。
- 留存期 = 各监管口径最大值；到期人工确认后归档（不自动清除）。
- **`retention_until` 重算规则（只增不减）**：`retention_until = max(各关联监管/法规条款保留期, 入库日 + 平台下限)`；当证据新增关联条款或条款保留期变更时**重算且只增不减**（已延长不回退），保证最严口径生效。重算触发点：证据↔法规/义务映射变更、关联条款保留期参数变更。

### 8.3 通知留痕
- M10 全部发送记录入留痕（送达对象/时间/通道/内容/结果），作为"是否及时通知"的监管举证（需求 12.2.5）。

### 8.4·补（V1.9 对齐）平台操作审计日志管理

提供**操作审计日志管理**统一查询/治理界面（系统管理），面向安全/合规/审计角色：
- **检索与导出**：按 `actor（操作人/含 domain_id）/ 动作 / 对象（数据类型+对象 ID）/ 时间范围` 多维检索；命中结果可导出（导出动作本身入审计）。
- **不可篡改**：底层 `operation_log` 仅追加、应用账号无 `UPDATE/DELETE`（§8.1），管理界面只读不可编辑，保证检索结果与原始留痕一致。
- **越权 / 特权未审查明细下钻**：将 §5.3/§十一的越权拦截记录、§4.7 中未在最近一轮 UAR 内复核的特权账号、跨子公司访问明细以专题视图呈现，支持逐条下钻到 `operation_log` 原始记录，供合规复查与监管举证。

### 8.5·补（V1.9 对齐）可信时间戳 / 哈希链 / WORM

强化留痕「不可篡改 + 时点可信」：
- **可信时间戳（TSA）**：对证据入库、关键审计/审批动作、UAR 签认证据等加盖 RFC 3161 可信时间戳（对接内部或合规 TSA 服务），证明「某内容在某时刻已存在且未被篡改」。
- **哈希链（hash-chain）**：`operation_log` / `evidence` 记录采用链式哈希（每条含前一条哈希），任一历史记录被改动即破坏链完整性，便于完整性校验与举证。
- **WORM 存储**：审计日志与证据可归档至 WORM（一次写入多次读取）存储/外部归档，从存储层杜绝改删（与 §8.1、D4-3 衔接）。

#### 8.5.1·补（功能链路补全 · A3/C13）哈希链落地规则与 TSA 覆盖范围

- **哈希链列与分链口径**：`operation_log` 与 `evidence` 各增 `curr_hash CHAR(64)` / `prev_hash CHAR(64)`；**按 `org_id` 分链**（每个子公司一条独立链，互不串扰，便于按域校验与举证），`curr_hash = sha256(本行规范化内容 ‖ prev_hash)`。
- **串行化写入 + 断链告警**：同一 `org_id` 链的写入**串行化**（取该链尾哈希→算 `curr_hash`→落库，加并发控制/链锁避免并发写造成分叉）；新写入的 `prev_hash` 与库内链尾不一致即判**断链并告警**。
- **离线重算校验作业**：周期性**离线重算校验作业**逐链重放 `curr_hash`，与库内值比对，发现任一历史记录被改/删/回插即告警并定位断点；校验结果与时点可由 TSA 佐证。
- **TSA 覆盖范围（集中存放）**：可信时间戳集中存于 `trusted_timestamp(对象类型, 对象id, tsa_token, ts_authority, at)`，由各模块引用——覆盖 **`evidence` 入库、`operation_log` 关键动作、`access_review` 签认（§4.7 `signed_hash`）、审批动作、`compliance_dossier` 封卷（`sealed_sha256`）**；`evidence` 自身的 `tsa_token`/`ts_authority` 列（D1-2 §22.3）与该集中表口径一致。

## 十、外部集成凭证安全（需求 13.3、§十五）
- AD 服务账号、短信平台密钥、企微/飞书 Webhook、SIEM/日志/漏洞接口凭证 **统一密钥管理**（Vault/KMS 或加密配置 + 启动注入），**不明文落配置文件或代码**（需求附录集成通用要求）。
- 凭证最小权限、定期轮换、访问审计。
- Webhook 地址等敏感配置在管理后台脱敏显示。

### 10.1·补（V1.9 对齐 · CR-004）模型接入与代码仓库出站安全

**模型接入与管理（系统管理 → 「模型接入」独立菜单）：**
- **模型端点凭证入密钥库**：外部/内网模型端点的 API Key、RAG/embedding 端点凭证统一入密钥库（Vault/KMS），**不落明文**；配置变更入操作审计（§8.1）。
- **「数据不出域」开关为可审计的强制策略**：默认开启；开关状态及变更入审计，作为可核查的强制策略而非提示。

**§10.1·补（功能链路补全 · D1）「数据不出域」= 条件式互斥分支 + test 端点共用出站网关：**
- **条件式互斥（删并列措辞）**：
  - **开 → 强制仅内网，外部端点直接拒绝**：指向外部模型端点的请求一律拒绝，**白名单不参与判定**（开关优先级最高）。
  - **关 → 允许外部端点，但出站必须命中白名单否则拒绝**：外部出站请求须命中**出站域名/IP 白名单**，未命中即拒绝，防 SSRF 打内网（与 §九 SSRF 行一致）。
  - 不再使用「开启时…白名单」的并列措辞——开与关是互斥分支，白名单仅在「关」分支生效。
- **test 端点与正式调用共用同一出站网关**：`/ai/models/{id}/test`（连通性测试）与模型正式调用**走同一出站网关**，**先过「数据不出域」判定、再过白名单**，杜绝用 test 端点绕过策略外联。
- **防 SSRF 旁路（DNS 解析后 IP 复核）**：对 `base_url` 做 **DNS 解析后对解析得到的 IP 再次复核**（不只校验域名字符串），**禁内网网段 / 保留地址 / 回环 / 链路本地**（含解析后指向内网的情况），堵住「域名在白名单但解析到内网 IP」的 SSRF 旁路；解析→复核→连接全程同一网关执行，防 DNS rebinding。
- **「数据不出域」为全平台红线**：开关=开 时，不仅模型调用，**反馈与建议出站（§10.1 代码仓库）的出站目标也强制切内网 Git（禁 GitHub 外网）**，全平台出站统一受此红线约束。

**代码仓库（反馈与建议出站，CR-004）：**
- **仓库令牌入密钥管理**：GitHub Token 等代码仓库凭证入密钥库，不落明文；最小权限、可轮换、访问审计。
- **出站白名单 + 共用出站网关**：推送目标（GitHub / 一键切换内网 Git）端点纳入出站白名单，禁止任意外联；推送与上方模型调用**共用同一出站网关**——先过「数据不出域」判定、再过白名单，并对目标 `base_url` 做 DNS 解析后 IP 复核防 SSRF（同 §10.1·补 D1）。**「数据不出域」=开 时强制切内网 Git、禁 GitHub 外网**。
- **反馈出站脱敏 + 管理员审批**：对外推送前**强制自动脱敏**（移除/打码 PII、密钥、内部 IP、子公司可识别信息），并经**管理员审批**后方可出站（默认不自动直推）；脱敏内容与审批、推送动作全程**出站审计**留痕（落 `egress_audit(actor, at, repo_target, content_hash, approver_id)` 或复用 `operation_log`，`feedback_repo_sync.egress_audit_id` 指向有效目标）。

## 十一、安全测试与验收要求（联动 D3-3）
1. **AI 数据隔离越权专项**：A 子公司账号尝试经问答获取 B 子公司评估数据 → 全部应拒绝并留日志（需求 7.3）。
2. **权限四元组边界**：角色×组织组合矩阵全覆盖，垂直/水平越权全部期望拒绝。
3. **留痕不可篡改验证**：尝试改/删 `evidence`、`operation_log` → 应被数据库层拦截。
4. **加密验证**：传输 TLS、敏感存储加密、备份加密。
5. **OWASP 扫描 + 渗透测试**：上线前通过，输出报告。
6. **凭证安全核查**：配置/代码中无明文密钥。

**§十一·补（功能链路补全）新增可验证断言：**
7. **数据不出域条件式互斥（D1）**：开关=开 时指向外部端点（含 `/ai/models/{id}/test`）一律拒绝、白名单不参与；开关=关 时未命中白名单拒绝、命中放行；`base_url` 解析到内网/保留地址一律拒绝（test 与正式调用同网关）。
8. **M5 检索隔离可验证（D5）**：服务端未注入 `visibleOrgs` 时检索拒绝（不退化全库）；A 子公司会话检索不召回 B 子公司 `kb_chunk`；过滤与 `top_k` 同一 SQL「先过滤后近邻」；集团级共享内容（`org_type=GROUP AND level=集团级`）对子公司可见。
9. **ext 宿主表全量 RLS + 自定义查询隔离（D7）**：以未 `SET app.visible_orgs` 的会话执行自定义视图/报表/KPI 动态查询返回空集或拒绝；超级用户/`BYPASSRLS` 连接执行被流程禁止；逐一核对 `asset/vendor/policy/audit_plan/obligation/reg_filing` 等 ext 宿主表均 `ENABLE ROW LEVEL SECURITY`。
10. **UAR 回收不双写 AD（D2）**：UAR 判「回收」仅改 `user_role_org`/置 `platform_disabled`，`app_user.status` 不变；下次 AD 同步不覆盖 `platform_disabled`；鉴权按 `(AD status=DISABLED) OR platform_disabled` 生效。
11. **单域故障降级（A2）**：模拟某域域控故障——该域禁新登录、已有会话维持且 TTL 收紧、敏感写（评估提交/审批/制度发布/导出）被拒；恢复后优先回放 `DISABLED`。
12. **哈希链完整性（A3）**：篡改/回插 `operation_log` 或 `evidence` 历史记录后，离线重算校验作业告警并定位断点；按 `org_id` 分链校验互不串扰。

## 十二、安全运营
- 监控：登录异常、越权拦截、敏感数据高频访问、通知失败、爬虫异常（SSRF 迹象）→ 阈值告警（复用 M10）。
- 审计复查：定期抽查 `operation_log`，尤其只读审阅角色与集团层的跨子公司访问。
- 应急响应：安全事件分级、处置、留痕；与 D4-3 应急预案衔接。
