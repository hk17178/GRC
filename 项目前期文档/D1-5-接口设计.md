## 一、接口分类
1. **内部 API**：前端 ↔ 后端（REST）。
2. **外部集成接口**：平台 ↔ AD/SIEM/日志/漏洞/短信/企微/飞书/邮件/法规网站/信用接口。
3. **领域事件**：模块间解耦（发布/订阅，供 M10/M9 消费）。

## 二、内部 API 规范

### 2.1 风格与约定
| 项 | 约定 |
|---|---|
| 风格 | RESTful，资源化 URL；`/api/v1/...`（**版本化**，便于演进，D1-1 §12） |
| 认证 | 会话/Token（基于 AD 登录）；每请求携带身份 |
| 鉴权与隔离 | 服务端强制校验 RBAC 四元组；自动注入 `visibleOrgs` 隔离上下文（D1-3 §5），前端不可绕过 |
| 分页 | `page`/`size`，返回 `total` |
| 统一响应 | `{ code, message, data, traceId }` |
| 错误码 | 业务错误码分段（如 4xxxx 业务、5xxxx 系统）；不泄露敏感信息 |
| 幂等 | 写操作支持幂等键（防重复提交） |
| 审计 | 敏感操作记 `operation_log` |

```

GET   /api/v1/policies / {id}/attestations           制度与签署确认
POST  /api/v1/remediations/{id}/verify               整改验证
GET   /api/v1/dashboard/overview?org=&period=        看板聚合
POST  /api/v1/ai/chat                                 AI 问答（流式；隔离过滤）

```

#### 2.2 补（V1.9 对齐 · CR-004）模型接入管理 + 建议与反馈内部 API

> 以下内部 API 服务于"模型接入与管理"（系统管理独立菜单）与"建议与反馈"（M-FB）两模块；UI 不出现实现术语。

**模型接入与管理（CR-004-2）**
```
GET   /api/v1/ai/models                               模型/端点列表（对话/embedding 分类）
POST  /api/v1/ai/models                               新增模型/端点配置（API Key 入密钥库，不落明文）
PUT   /api/v1/ai/models/{id}                           修改模型/端点配置
DELETE /api/v1/ai/models/{id}                          删除模型/端点配置
POST  /api/v1/ai/models/{id}/test                      连通性测试（按钮触发；返回时延/可用性）
GET   /api/v1/ai/rag-config                            RAG 参数（召回数/相似度阈值/切片/索引刷新策略）
PUT   /api/v1/ai/rag-config                            修改 RAG 参数
GET   /api/v1/ai/prompts                               系统提示词模板与提示注入防护开关
PUT   /api/v1/ai/prompts                               维护提示词模板/防护开关
GET   /api/v1/ai/data-residency                        「数据不出域」开关状态（默认开）
PUT   /api/v1/ai/data-residency                        切换「数据不出域」开关
```
- 约定：API Key/服务地址入密钥管理，响应**不回显明文**；所有配置变更写入操作审计；「数据不出域」=**条件式互斥**——开关=**开** 时仅允许内网模型/内网 API，外部端点**直接拒绝**（白名单不参与）；开关=**关** 时允许外部端点但出站**必须命中白名单**否则拒绝（防 SSRF，D1-3 §9/§6.4；测试与正式调用同网关，见 §2.3 补）。

**建议与反馈（M-FB · CR-004-1）**
```
POST  /api/v1/feedback                                提交反馈（BUG/建议；自动带上下文）
GET   /api/v1/feedback?type=&module=&status=&page=    反馈列表（隔离过滤）
GET   /api/v1/feedback/{id}                            反馈详情（含 AI 可行性初评，人工复核为准）
PUT   /api/v1/feedback/{id}/category                  人工复核分类/处置（采纳/观察/婉拒）
POST  /api/v1/feedback/{id}/desensitize               生成脱敏结果（PII/密钥/内部 IP/可识别信息）
POST  /api/v1/feedback/{id}/approve-sync              管理员审批后推送代码仓库（出站审计）
GET   /api/v1/feedback/sync-logs                       出站推送审计日志
```
- 约定：对外推送默认**不自动直出**，须脱敏 + 管理员审批后出站；每次推送写出站审计（操作人/时间/内容哈希）；仓库凭证入密钥库，可一键切内网 Git。

### 2.3 AI 问答接口（特别约定）
- 流式返回（首字 ≤5s，需求 13.1）；
- 服务端在**检索阶段**按 `visibleOrgs` 过滤（安全红线，D1-3 §6），不在生成后过滤；
- 响应含**引用来源**（制度条款编号/记录编号）；涉敏感访问记日志。

#### 2.3 补（出站网关说明 · D1-3 §6.4/§9）`/ai/models/{id}/test` 与正式调用同网关

> `/ai/models/{id}/test`（连通性测试）与运行期正式推理调用**共用同一出站网关**，链路顺序固定：**①「数据不出域」判定 → ②白名单判定 → ③SSRF 防护 → 放行/拒绝**。测试按钮不绕过任何一道闸门。

- **数据不出域=条件式互斥（D1-3 §6.4）**：
  - 开关=**开**：强制仅内网端点；任何外部端点**直接拒绝**（白名单不参与判定）。
  - 开关=**关**：允许外部端点，但出站**必须命中白名单**否则拒绝。
- **SSRF 旁路堵截**：对 `base_url` 做 DNS 解析后**对解析 IP 复核**，禁内网/保留地址段（防 DNS 重绑定旁路）；测试与正式调用同一校验。
- **审计**：测试与正式调用的出站均记 `egress_audit`/`operation_log`（actor/at/repo_target/content_hash/approver_id）；配置变更（含开关切换）入操作审计且响应不回显密钥明文。

### 2.4 补（功能链路补全 · 留痕/卷宗/合规清单接口 · 对应 E7）

> 落实"可建链"的留痕审计、电子卷宗封存与合规清单正反向查询接口；哈希链与 TSA 详见 D1-2 C13、D1-3 §10。

**留痕审计与证据（防篡改哈希链 + TSA）**
```
GET   /api/v1/audit-logs?actor=&object=&from=&to=&page=   操作日志检索（检索动作本身回写 operation_log）
GET   /api/v1/audit-logs/export?...                       导出（导出动作回写 operation_log，记 content_hash）
POST  /api/v1/evidence                                    入库证据（算 sha256 + 申请 TSA + 链 prev_hash/curr_hash，按 org_id 分链串行写）
GET   /api/v1/evidence/{id}/verify                        单证据校验（重算哈希 + 校验链 prev_hash + 验签 tsa_token）
```
- 约定：`operation_log` 与 `evidence` 均含 `curr_hash/prev_hash`（按 `org_id` 分链、串行化写入）；断链/校验失败触发告警（离线重算校验作业）；`retention_until` 重算=max(各关联条款保留期, 入库日+下限)**只增不减**。

**电子卷宗（封存/导出/校验）**
```
POST  /api/v1/dossiers/{id}/seal                          封存卷宗（生成 sealed_sha256 + TSA 时间戳）
GET   /api/v1/dossiers/{id}/export                        导出（含清单 + 逐项 item_sha256 + sealed_sha256）
GET   /api/v1/dossiers/{id}/verify                        校验（逐项重算 item_sha256 与封存 sealed_sha256 比对）
```

**合规清单正反向查询（M11/合规义务清单 · 对应 C1）**
```
GET   /api/v1/obligations?org=&status=&regulation=&page=  义务清单（satisfaction_status=MET/PARTIAL/GAP，隔离过滤）
GET   /api/v1/obligations/{id}                            义务详情（含派生满足状态与重算依据）
GET   /api/v1/obligations/{id}/trace-down                 正向追溯：法规↓义务↓制度(obligation_policy_map)↓控制(obligation_control_map)↓证据
GET   /api/v1/evidence/{id}/trace-up                      反向追溯：证据↑义务↑法规(经 evidence_regulation_map.obligation_id)
POST  /api/v1/obligations/{id}/recompute                  触发满足状态重算（证据/控件测试/审计发现/映射变更后）
```
- 约定：满足状态为**派生值**——关联 control 的 `control_test_result` 全通过且证据未过期→MET；部分→PARTIAL；缺→GAP；存在未关闭且 `severity≥高` 的关联 `audit_finding`→降为 GAP/PARTIAL。重算结果变化时产 `OBLIGATION_STATUS_CHANGED`/`OBLIGATION_GAP_DETECTED` 事件（见 §四·补2）。

## 三、外部集成接口（按需求 §15）

> 通用要求（适用全部）：标准化接口（REST/Webhook）封装 + 异常（超时/认证失败/格式异常）容错与告警 + 凭证安全存储（不明文）+ 采集类明确频率与格式，由相关系统运维团队共同确认接口规范。

### 3.1 AD（Active Directory）— 用户/组织同步
| 项 | 内容 |
|---|---|
| 用途 | 同步用户账号、组织架构、部门归属（身份与权限基础） |
| 方向 | AD → 平台 |
| 协议 | LDAP/LDAPS（或 AD 同步 API） |
| 数据 | 用户(username/display/email/ad_object_id/部门)、组织树 |
| 模式 | 全量初始化 + **增量同步**（基于变更游标/ad_object_id） |
| 频率 | **待定**（设计阶段定值，需求 8.3/10.4）；离职/调岗时效需明确 |
| 异常 | 同步失败告警；不覆盖式更新（保留审计） |
| 安全 | 服务账号最小权限 + 凭证入密钥库 |
| **多域控（CR-003）** | **支持配置 N 个身份源（各子公司/海外独立 AD 域/林）**，每个域独立 LDAP(S) 地址（可多控制器）、服务账号、Base DN、同步频率，绑定对应 `org_id`；单域不可用不影响其余域。认证按 `DOMAIN\user` / `user@domain` 路由到对应域控；账号唯一性为"域内唯一"。详见 D1-3 §3.1.1、D1-2 §四。 |

### 3.2 SIEM / 日志平台 / 漏洞管理 — KRI 数据采集
| 项 | 内容 |
|---|---|
| 用途 | 采集 KRI 指标（日志留存天数、漏洞修复时效、接口异常率等，需求 4.2.5） |
| 方向 | 外部 → 平台 |
| 模式 | 平台定时**拉取**（REST/查询）或外部**推送**（按系统能力） |
| 频率/格式 | **待与运维共同确认**（需求 4.2.5/§15 集成通用要求） |
| 异常 | 采集失败/数据缺失告警；指标缺失不误判 |

### 3.3 短信网关（公司自有）— 紧急级通知
| 项 | 内容 |
|---|---|
| 方向 | 平台 → 短信平台 |
| 用途 | 紧急级通知（成本敏感，由规则引擎控制是否启用） |
| 约束 | **调用额度限制**需提前确认（需求 12.4），防超额；失败重试 |

### 3.4 企业微信 / 飞书 — Webhook 群机器人

## 四、领域事件接口（模块解耦）

- 生产者发布领域事件到事件总线（Redis Stream），消费者（M10 通知、M9 看板）订阅。
- 事件 Schema（统一信封）：
```json
{
  "eventId": "uuid",
  "type": "REMEDIATION_OVERDUE",          // 评估完成/整改逾期/法规更新/制度发布/KRI预警…
  "sourceModule": "M2",
  "orgId": 120,
  "occurredAt": "2026-06-23T08:00:00Z",
  "payload": { "taskId": 392, "assignee": 11, "overdueDays": 2 }
}
```
- **新增事件类型或新增订阅者，不改动生产者**（D1-1 §4.3，不牵一发动全身）。
- 通知路由结合 M6 组织 + M8 权限，确保不跨子公司广播（需求 12.2.2）。

#### 四·补1（功能链路补全 · 事件来源统一为调度器 · 对应 E1/B）

> **时间驱动事件的生产者统一为"调度/到期扫描"内核**（独立 Worker 或应用内分布式定时 Quartz+ShedLock，D1-1/D1-4 §调度内核），周期扫描各表 `due_date/deadline/expiry/plan_start_date/reminder_days/reuse_window` 等字段产出。**业务模块不再自产时间事件**——尤其 `COMPLIANCE_DIGEST_DUE`/`FEEDBACK_DIGEST_DUE` 的生产者改为**调度器**（不再由 M10/M-FB 自产自销，消除"既产又消"口径）。

**调度器产出的时间驱动事件（统一来源=调度器）**

| 事件类型 | 扫描字段/对象 | 幂等台账 | 消费 |
|---|---|---|---|
| `REG_FILING_DUE` | reg_filing.statutory_deadline / reminder_days | 按(对象,阈值)去重 | M10 场景 / M9 |
| `REG_INQUIRY_DUE` | reg_inquiry.reply_deadline | 按(对象,阈值)去重 | M10 场景 / M9 |
| `EXT_AUDIT_PLAN_APPROACHING` | audit_plan.plan_start_date / reminder_days | `audit_plan_reminder_log(audit_plan_id,reminder_day)` UNIQUE | M10 场景 |
| `EXTERNAL_RESPONSE_DUE` | 外审对外回函 reply_deadline（**时限提醒**，与三段状态事件分开） | 按(对象,阈值)去重 | M10 场景 |
| `CERTIFICATION_EXPIRING` | 认证证书 expiry / reminder_days | 按(对象,阈值)去重 | M10 场景 |
| `ACCESS_REVIEW_DUE` | access_review.period / due | 按(对象,阈值)去重 | M10 场景 |

## 五、防腐层与容错（通用机制）
- 每个外部系统一个**适配器**，实现统一内部接口；外部改版/替换只改适配器。
- 统一容错：超时控制、重试（指数退避）、熔断、降级；失败多次告警管理员（复用 M10）。
- 限流：对外部接口（短信额度、企微/飞书频率）限流，避免被限流封禁。
- 凭证：统一密钥管理，不明文（D1-3 §10）。

---

## 六、接口安全
- 内部 API：认证 + RBAC 四元组 + 隔离上下文注入；输入校验；输出脱敏。
- 集成接口：凭证安全、最小权限、传输 TLS、调用审计。
- 爬虫：目标白名单防 SSRF。

---

## 七、评审重点
1. 外部接口的**频率/额度/格式**待确认项清单及责任人（AD/SIEM/短信/企微等）。
2. 内部 API 的隔离上下文注入与版本化策略。
3. 领域事件 Schema 与总线选型（Redis Stream 起步，可换 RabbitMQ/Kafka）。
4. AI 接口检索阶段过滤的接口契约。

> 配套：D1-4 详细设计将细化各模块完整 API 清单；D4-1 安装/部署将把外部接口参数纳入 Web 向导配置项。
