# GRC 平台 · 表单引擎专项设计文档

**文档编号：** D1-6 ｜ **版本：** v1.9（草案） ｜ **日期：** 2026-06-23
**依据：** 需求 V1.9（§4.2.2/4.2.3/4.4、§5.2.2）、D1-1 架构、D1-2 数据模型 §7
**状态：** 待评审

> 表单引擎是**全项目最高难点与最大工作量来源**（可行性评估）。它服务 M2 风险评估、M3 审计检查表、M7 供应商评估——三者**共用同一引擎**，不另建第二套配置能力。

---

## 一、设计目标与核心矛盾

### 1.1 目标
让运营人员（信息安全/合规/内审）**不写代码**即可配置出符合各体系要求的评估/审计表单、打分逻辑与报告格式，并支持新体系平滑纳入。

### 1.2 核心矛盾（需求 4.4，必须正面回应）
> "表单引擎若做得过于通用，容易退化为万能表单生成器而牺牲易用性（运营人员不知道如何配出一份合规的等保测评表）。"

**对策（双轨并行，缺一不可）：**
1. **通用配置能力**：元数据驱动，结构/打分/报告皆可配；
2. **内置体系脚手架**：预置 8 套体系的可用模板，运营从"克隆调整"起步而非"空白画起"。

### 1.3 设计原则
- 结构配置层 与 打分/报告映射层 **两层解耦**（互不牵连）；
- 模板**版本化 + 启动快照**（进行中评估不受后续修订影响）；
- 控件库**复用优先**（跨体系一次测试多处用）；
- 配置治理**集中管控、分级开放**（合规底线不被业务无意破坏）。

## 二、总体架构（元数据驱动）

```
                 ┌─────────────── 模板（template）──────────────┐
                 │  framework 体系   usage 用途(评估/审计)         │
                 └───────────────────┬─────────────────────────┘
        ┌────────────────────────────┴────────────────────────────┐
        │                  模板版本 template_version                │
        │       version_no · effective_date · status               │
        │  ┌─ 结构配置层 ─┐        ┌─ 打分/报告映射层 ─┐             │
        │  │ section 层级 │        │ scoring_model     │             │
        │  │ item 题目    │◄──解耦──┤ report_format     │             │
        │  │ 题型/校验/跳题│        │ item.weight       │             │
        │  │ item→控件引用 │        │ item.report_key   │             │
        │  └──────┬───────┘        └───────────────────┘             │
        └─────────┼──────────────────────────────────────────────────┘
                  │ 引用
         ┌────────▼─────────┐   control_framework_map（跨体系映射）
         │ control_library   │◄──────────────────────────────────────
         │ 统一控件库(控制点) │
         └──────────────────┘
  运行期：渲染器读「版本快照」动态生成表单/报告；答案存 assessment_answer(JSONB)+抽取打分列
```

**关键点**：模板=数据；引擎=通用渲染/打分/出报告器。**新增体系/调整模板 = 改数据，不改引擎代码。**

## 三、结构配置层（Structure）

### 3.1 层级（域 → 控制项 → 具体问题）
- `template_section` 自引用，深度与命名（如"控制点编号"）**可按体系配置**（需求 4.2.2）。
  - 等保三级：按"安全通用要求 + 安全扩展要求"分类 + 控制点编号；
  - ISO 27001：资产-威胁-脆弱性-可能性×影响 四要素；
  - 供应商：合同条款 + 履约表现（无"控制点"概念）。

### 3.2 题型（item_type）
单选 / 多选 / 打分题（如 0–5 级成熟度）/ 文本说明 / 矩阵题 / 附件上传（佐证）。

### 3.3 校验与跳题
- 必填校验规则；
- 跳题逻辑 `skip_logic`（如某题答"不适用"→自动跳过其关联后续问题）。

### 3.4 多级填写与提交流程
- 流程节点（如 系统负责人填写 → 部门负责人确认）由 **M8 审批流引擎（Flowable）** 实现，引擎只定义"需要几级、谁审"，不自研流程。

## 四、打分与报告映射层（Scoring & Report）

### 4.1 打分模型（scoring_model JSON，模板版本级）
- **默认定量**：可能性 × 影响 = 风险值，便于跨对象/跨周期横向对比与看板聚合（需求 4.2.2/V1.7）。
- **叠加定性**：供应商等难量化维度，在定量基础上叠加定性判断项（如合同履约），非完全切换。
- **可选方式**：定性成熟度等级（0–5）、简单合规判定（是/否/不适用）保留供特定场景选用。
- 每题/每题类**权重**（item.weight）。
- **风险分级口径（五级，统一标准）**：风险值 → 区间映射至**极低/低/中/高/极高**（`VERY_LOW/LOW/MID/HIGH/VERY_HIGH`），与 D1-2 `risk_level`、看板/标签/分布图、M9 聚合完全一致（见 CR-001-7）；**不再使用三级（高/中/低）口径**。`levelMatrix` 示例区间见下，体系模板可按需调整阈值但级数固定为五级。

```json
// scoring_model 示例
{
  "method": "QUANTITATIVE",
  "formula": "likelihood * impact",
  "levelMatrix": [
    {"max":3,"level":"VERY_LOW"},
    {"max":6,"level":"LOW"},
    {"max":12,"level":"MID"},
    {"max":20,"level":"HIGH"},
    {"max":25,"level":"VERY_HIGH"}
  ],
  "qualitativeOverlay": [{"itemRef":"sla_perf","weight":0.2}]
}
```

### 4.2 报告章节映射（report_format + item.report_section_key）
- 每题得分**映射到报告的目标章节**（如某控制项得分归入"异常处置"章节而非"网络安全"，需求 4.2.2）。
- `report_format` 固化章节顺序/措辞规范（随模板版本固化，确保同体系不同周期产出格式一致，便于监管比对）。

### 4.3 两层解耦的价值
- 调整报告格式 → 只改 `report_format`/`report_section_key`，**不动表单题目**；
- 调整表单字段 → 不破坏既有打分逻辑（需求 4.2.2/4.4）。

## 五、模板版本化与继承

| 能力 | 机制 |
|---|---|
| 版本号 + 生效日期 | `template_version` |
| 启动快照 | 评估启动即绑定 `template_version_id`，后续模板修订不影响进行中评估（需求 4.2.2） |
| 克隆继承 | 基于既有版本/模板克隆后局部调整，而非空白画起（脚手架的落地方式） |
| 发布前测试 | `status=DRAFT` 可预览/测试填写，验证通过才置 `EFFECTIVE`（需求 4.2.2 配置治理） |

## 六、内置体系脚手架（需求 V1.6）

预置可用模板：**ISO 27001、等保三级、PBOC、PCI DSS、ISO 27701、ISO 27018、ISO 9001、供应商风险评估**；并支持**平台管理员自定义新增体系分类**，新体系可从零搭建或克隆调整——公司未来新增认证/监管要求时**无需开发改动即可纳入**。

> 脚手架是"易用性"的来源：运营从一份已合规的模板起步，避免不知如何配出合规的等保测评表。

---

## 七、统一控件库 ⭐（需求 V1.7）

### 7.1 颗粒度
按**"具体控制点"**抽象（如"特权账号定期审查"），而非笼统的"控制域"（如"访问控制"）。

### 7.2 跨体系映射与复用
- 体系模板**引用**控件库条目，而非各自重复定义题目；
- 同一控件被多体系评估后，支持**"一次测试结果，多体系复用"**（control_test_result + 复用有效期），减少重复填表与重复举证（需求 4.2.2）。
- `control_framework_map` 维护控件在各体系下的条款编号映射。

### 7.3 成本权衡与维护工具（需求 4.4/V1.7）
- "具体控制点"颗粒度带来**更高初始建设与维护投入**；
- 配套**控件维护工具**：控件批量管理、跨体系映射的批量维护、影响分析（改一个控件→提示受影响的模板/体系），降低长期维护成本。

### 7.4 落地建议（呼应可行性评估）
- **接受"先重复、后抽象"**：一期先做"模板克隆 + 基础结构配置"，统一控件库的跨体系复用作为**二期**，用真实体系数据驱动控件库演进，不一开始追求完美映射。

## 八、配置治理（集中管控、分级开放，需求 V1.7/10.2）

| 操作 | 权限角色 | 管控 |
|---|---|---|
| 模板新建、结构性修改（题目层级、打分逻辑、报告章节映射） | **表单/模板结构配置管理员**（归口信息安全部） | 纳入 M8 审批，避免合规底线被业务无意破坏 |
| 模板内局部参数（权重微调、提醒文案、个别选项） | **表单/模板局部参数调整人员**（合规/内审等业务部门） | M8 控制可调范围 |
| 已发布模板的任何修改 | 上述角色 | **必须经预览/测试填写验证后才正式生效**，避免改动导致进行中评估无法提交或打分出错 |

## 九、与 M3 审计 / M7 供应商的复用

- **统一引擎，按 `template.usage` 区分**（ASSESSMENT / AUDIT）：同一套模板管理后台，按"用途"字段区分，不另建独立配置入口（需求 5.2.2）。
- 审计场景差异化补充（在通用引擎之上）：
  - 检查项**引用 M1 制度条款**作为依据，报告自动带出条款编号（需求 5.2.2）；
  - 检查项与严重度评级关联（判不合规时建议严重度，供审计人员参考调整，不强制覆盖人工判断）；
  - 补充审计专有脚手架（等保测评检查表、内部合规审计检查表）。
- M7 供应商：供应商专用模板（定量叠加定性），复用整改与跟踪机制。

> 结论：模板版本管理、权限治理、预览测试等机制**只实现一次**，杜绝两套配置能力逐渐能力不一致（需求 5.4）。

## 十、报告自动生成

- 输入：评估答案 + scoring_model + report_format + item.report_section_key；
- 输出：风险结论、风险点清单、等级分布统计图，按目标体系章节结构组织（需求 4.2.3）；
- 人工复核签批后 `EFFECTIVE`（不完全依赖自动化结论）；
- 导出 Word/PDF（异步生成，§D1-1）。

## 十一、运行期渲染与执行

- **前端动态渲染**：读模板版本快照元数据，生成填写界面（题型/校验/跳题）；多主题令牌驱动样式。
- **填写**：答案存 `assessment_answer.value`(JSONB)，佐证附件存 MinIO；打分实时/提交时计算并抽取到列。
- **填写流程**：经 M8 多级提交。
- **进行中保护**：会话中断自动保存草稿（需求 13.2）。

## 十二、关键技术风险与对策

| 风险 | 对策 |
|---|---|
| 过度通用 → 退化为万能表单生成器 | 通用配置 + 内置脚手架双轨；不可偏废（需求 4.4） |
| 两层耦合致后期维护困难 | 结构与打分/报告分表存储、分别演进 |
| 已发布模板被误改致评估中断 | 修改权限分级 + 发布前预览测试强制 |
| 控件库颗粒度维护成本高 | 控件维护工具 + 批量映射 + 影响分析；先重复后抽象 |
| 报告格式漂移 | report_format 随版本固化 |

## 十四、验收标准（关键）
1. 不写代码可配出一份**可用的等保三级测评表**并完成评估、出报告；
2. 模板改报告格式不影响表单；改表单不破坏打分；
3. 进行中评估不受模板升级影响（版本固化）；
4. 同一控件在两个体系间复用测试结果；
5. 结构性修改与局部参数调整权限分离且生效前强制测试；
6. 新增一个自定义体系并克隆出可用模板，全程零开发改动。

## 十四·补、风险建模与五级矩阵增补（CR-001 / CR-002）

**1. 资产-威胁-脆弱性(A-T-V)建模**：评估表单支持以「资产 → 威胁源 → 脆弱性 → 可能性 → 影响」四要素建模（落地 ISO 27005 / 等保风险评估方法），打分以 A-T-V 组合为最小单元，汇总为风险项。

**2. 风险处置与残余风险**：评估报告新增——固有风险等级(`inherent_level`) → 处置决策(`treatment_decision`：降低/接受/转移/规避) → 残余风险等级(`residual_level`) → 管理层接受签认(`risk_acceptance`，含签认人/时间/留痕)。报告映射新增上述区块。

**3. 五级矩阵**：`levelMatrix` 由 3 级改为**五级**——极低/低/中/高/极高(`VERY_LOW/LOW/MID/HIGH/VERY_HIGH`)，风险值→区间映射统一口径，与看板/标签/分布图、M9 聚合一致（见 CR-001-7）。

**4. 综合风险指数可解释**：指数计算需输出**构成明细**（各风险域得分与权重）、**环比归因**（较上期变化来源）、**计算口径说明**，支持从指数下钻到风险项。

**5. 供应商评估复用**：`assessment` 新增 `kind`(准入/年度定期/续约/退出/事件复评)，target=vendor，复用同一打分/报告/工单闭环；技术安全与 DPA 维度作为供应商专属控件补充（详见 §十四·补2）。

## 十四·补2、供应商专属控件与 A-T-V 结构层建模（V1.9 对齐 · CR-001 / CR-002）

### 1. 供应商专属控件（技术安全 + DPA 维度）
在通用控件库之上，为 M7 供应商评估补充一组**专属控件分类**，统一纳入控件库（控制点颗粒度），供供应商模板引用、并可被多体系/多供应商复用：

| 维度 | 专属控件（控制点）示例 |
|---|---|
| **技术安全** | 加密传输与存储、漏洞与补丁管理、访问与身份管理、日志与监测、备份与可恢复性、安全事件通报时限 |
| **数据保护 / DPA** | **数据驻留**（处理/存储地域与跨境约束）、**PCI 范围**（是否触及持卡人数据环境 CDE 及合规证明）、**再委托**（分包/次级处理者审批与穿透约束）、**最小权限**（数据访问按需授权、离场销毁）、数据处理协议(DPA)条款落实、个人信息删除/返还义务 |

> 上述维度作为**定量叠加定性**模型的一部分：技术安全可定量打分，DPA 类多为合规判定（是/否/不适用）+ 佐证附件；缺失关键项可触发准入阻断或风险升级。

### 2. A-T-V 作为结构层建模，与"域 → 控制项 → 题"层级共存
- **A-T-V（资产-威胁-脆弱性）是结构配置层的一种建模方式**，落地 ISO 27005 / 等保风险评估方法：以「资产 → 威胁源 → 脆弱性 → 可能性 × 影响」为最小打分单元，汇总为风险项。
- 它**不取代** §三的「域 → 控制项 → 具体问题」层级，而是**与其共存**：
  - 合规符合性类评估（等保测评检查表、ISO 27001 控制项核查、供应商条款核查）采用「域 → 控制项 → 题」层级；
  - 风险评估类对象采用 A-T-V 四要素建模；
  - 二者由 `template.framework / usage` 与 `template_section` 的层级定义区分，**同一引擎、同一控件库**承载，互不排斥，可在同一评估对象内混合引用（如供应商评估既有条款核查项，又有针对关键资产的 A-T-V 风险项）。

## 十四·补3、表单引擎与全局自定义字段/视图能力的关系（V1.9 对齐 · CR-001-2）

CR-001-2 将"高自由度可配置"全局化（自定义字段 / 视图筛选 / 看板 / 报表 / 工作流 / KPI / 通知场景），本节厘清表单引擎与该能力的边界与协作。

### 1. 控件库 vs 自定义字段（边界）
| 维度 | 表单引擎·控件库（控制点） | 全局·自定义字段 |
|---|---|---|
| 定位 | 评估/审计**题目与控制点**，承载打分逻辑与跨体系复用 | 在**评估对象/供应商/审计等业务实体**上扩展业务属性（如供应商联系人、合同到期日、自定义标签） |
| 是否参与打分 | 是（item.weight / scoring_model） | 默认否（业务字段，用于展示/筛选/分组） |
| 存储 | `assessment_answer.value`(JSONB) + 抽取打分列 | 业务实体的 **JSONB 扩展列 + 字段元数据登记**（CR-001-2 / D1-2） |
| 治理 | §八 配置治理（结构性归口管理员 + 发布前测试） | 沿用同一"集中管控、分级开放"思路，纳入 M8 权限与变更留痕 |

> 原则：**打分相关 → 走控件库题目；纯业务属性 → 走自定义字段。** 避免把业务字段塞进控件库导致打分模型污染，也避免把控制点降级为无打分语义的自定义字段。

### 2. 评估对象自定义业务字段在表单中的引用
- 评估对象上登记的**自定义业务字段**（经字段元数据登记后）可被表单**只读引用 / 预填**（如自动带出"系统等级""数据分类""合同到期日"），减少重复录入；
- 引用方式：`item` 支持绑定一个**字段引用源**（`field_ref`，指向业务实体的自定义字段键），渲染期由引擎解析取值；此类引用项**不进入打分**，仅作上下文展示或作为跳题/校验条件输入。

### 3. 视图 / 看板 / 报表的协作
- 表单引擎产出的评估结果（含五级风险等级、残余风险、A-T-V 风险项）作为**数据源**，由全局**自定义视图筛选 / 看板 / 报表**能力消费；
- 报告章节固化（§四 report_format）解决"同体系跨周期格式一致、便于监管比对"；自定义报表则面向**运营自助的灵活汇总分析**，二者并行不冲突。

## 十四·补4、levelMatrix 平台统一基线 + 求值/连续性/覆盖校验（功能链路补全 · A1 / G1）

> 落地决策 A1：风险五级 `levelMatrix` 由"模板可调阈值"上收为**平台级统一基线**，模板**不可改阈值**，只能引用。目的：M9 跨子公司聚合按 `risk_level` 计数语义一致、不再失真。**原 §4.1 / 十四·补 中"体系模板可按需调整阈值"的措辞作废**，以本节为准。

### 1. 平台统一基线（不可由模板改写）
- `levelMatrix` 不再随 `scoring_model`（模板版本级）下发可改阈值，而是由平台维护**唯一基线**，挂在平台级配置（建议 D1-2 新增 `risk_level_matrix(version, level, max_value, min_value, effective_from, status)` 或集中常量表），五档固定为 `VERY_LOW/LOW/MID/HIGH/VERY_HIGH`：

```json
// 平台统一基线（risk_score 取值域 1..25，likelihood×impact，1..5 × 1..5）
// 模板 scoring_model 仅 引用 不 覆盖：{"levelMatrixRef":"PLATFORM_BASELINE_v1"}
[
  {"level":"VERY_LOW",  "min":1,  "max":3},
  {"level":"LOW",       "min":4,  "max":6},
  {"level":"MID",       "min":7,  "max":12},
  {"level":"HIGH",      "min":13, "max":20},
  {"level":"VERY_HIGH", "min":21, "max":25}
]
```

- 模板侧 `scoring_model` 只保留 `levelMatrixRef`（指向平台基线版本），**移除内嵌 `levelMatrix` 数组的可编辑性**；引擎求值时一律读平台基线，模板无权传入自有区间。
- 既有 §4.1 JSON 示例中的 `levelMatrix` 内嵌数组，运行期仅作**渲染说明展示**，不参与求值与校验，求值以平台基线为准（避免"改一半"歧义）。

### 2. 求值规则（归档判定）
- 输入 `risk_score`（由 §十四·补5 A-T-V 链路或题目加权得出，值域 1..25）。
- 归档：取**满足 `min ≤ risk_score ≤ max` 的唯一档**；等价实现为"按 `max` 升序，命中首个 `risk_score ≤ max` 的档"——因基线连续无洞，两种判法结果一致。
- `risk_score` 落在 `max` 边界（如 =6）归**本档**（LOW），不上溢到下一档（`value ≤ max 归本档`）。

### 3. 基线保存/变更时的连续性与覆盖校验（拒绝有洞/重叠/非单调）
平台维护基线时，保存前强制校验，任一不满足即**拒绝保存**并定位冲突档：
1. **五档齐全**：必须恰为 `VERY_LOW/LOW/MID/HIGH/VERY_HIGH` 五档，不多不少；
2. **单调递增**：按档序 `max` 严格升序（`VERY_LOW.max < LOW.max < … < VERY_HIGH.max`）；
3. **相邻连续无缝**：`next.min = prev.max + 1`（整数域）或 `next.min = prev.max`（连续实数域，左闭右开），**禁止空洞**（如 prev.max=6、next.min=8 缺 7）；
4. **无重叠**：任意两档区间交集为空（由 2、3 蕴含，单列断言便于定位）；
5. **全量程覆盖**：`第一档.min = 取值域下限(1)`、`末档.max = 取值域上限(25)`，覆盖整个 `risk_score` 值域，任何合法 `risk_score` 必落且仅落一档。
- 校验失败信息需指出"哪两档之间有洞/重叠/逆序"，便于运营自查。
- 配套测试断言对应 D3-2 **TC-M2-104（平台统一阈值基线）**：模板不可改阈值；并新增基线连续性/覆盖校验用例（有洞/重叠/缺档/逆序四类负向）。

### 4. 对 M9 聚合的语义保证
- 全平台所有评估对象的 `risk_level` 由同一基线派生 → M9 跨子公司"极低/低/中/高/极高"计数口径一致，热力矩阵/分布图/聚合不因模板私改阈值而失真（呼应 D1-7 §5.1 读 `risk_domain` 字典与本节基线协同）。

## 十四·补5、A-T-V 承载与打分链路 + risk_finding 物化（功能链路补全 · C3 / C4 / G1）

> 落地决策 C4 / C3：A-T-V 四要素的**结构承载表**与**风险点物化表**补齐，使"A-T-V → risk_score → 五级 → 风险点（残余/处置/接受）"链路在结构上可建链。`inherent/treatment/residual/acceptance` 由 assessment 级**下沉到 risk_finding 级**。

### 1. A-T-V 结构承载表 asset_threat_vuln（C4，建议 D1-2 §7 展开建表）

```sql
-- A-T-V 最小打分单元（资产-威胁源-脆弱性-可能性×影响）
CREATE TABLE asset_threat_vuln (
  id BIGINT PRIMARY KEY,
  org_id BIGINT NOT NULL,
  assessment_id BIGINT NOT NULL REFERENCES assessment(id),
  asset_id BIGINT NOT NULL,
  threat VARCHAR(256) NOT NULL,                  -- 威胁源
  vulnerability VARCHAR(256) NOT NULL,           -- 脆弱性
  likelihood SMALLINT NOT NULL,                  -- 可能性 1..5
  impact SMALLINT NOT NULL,                      -- 影响 1..5
  risk_score INT GENERATED ALWAYS AS (likelihood * impact) STORED,  -- 抽取列 = L×I，值域 1..25
  risk_level VARCHAR(12)                          -- 五级映射（按平台 levelMatrix 基线派生）
  -- + 审计字段（created_*/updated_*）
);
-- RLS：org_id 纳入（D7），visibleOrgs 注入
```

- `risk_score` 为**生成列**（`GENERATED ALWAYS AS (likelihood*impact)`），引擎不手填；`risk_level` 由 §十四·补4 平台基线对 `risk_score` 求值得出。
- A-T-V 行**物化为 risk_finding**（见下），报告"风险点清单"基于 risk_finding。

### 2. 风险点物化表 risk_finding（C3，承载残余/处置/接受）

```sql
CREATE TABLE risk_finding (
  id BIGINT PRIMARY KEY,
  org_id BIGINT NOT NULL,
  assessment_id BIGINT NOT NULL REFERENCES assessment(id),
  atv_id BIGINT REFERENCES asset_threat_vuln(id),    -- 可空：A-T-V 来源；条款核查项物化时为空
  title VARCHAR(256) NOT NULL,
  inherent_level VARCHAR(12),                         -- 固有风险等级（下沉至风险点级）
  treatment_decision VARCHAR(12),                     -- MITIGATE/ACCEPT/TRANSFER/AVOID
  residual_level VARCHAR(12),                         -- 残余风险等级
  risk_acceptance JSONB,                              -- {approver, at, evidence_id} 管理层接受签认
  risk_score INT,
  risk_level VARCHAR(12),                             -- 五级（按平台基线）
  status VARCHAR(16)                                  -- OPEN→...→DONE→VERIFIED（含关闭门控，见 §F2）
  -- + 审计字段
);
```

- **下沉语义**：`inherent_level / treatment_decision / residual_level / risk_acceptance` 从原 §十四·补2 的 assessment/报告级**下沉到 risk_finding 级**——每个风险点独立处置与接受，报告区块逐风险点呈现，不再整表一刀切。
- **整改关联**：`remediation_task` 在 `source_type=ASSESSMENT` 时 `source_id` 统一指向 `risk_finding.id`；报告"已转整改/已闭环"基于 `risk_finding ↔ remediation_task` 关系派生（呼应 D1-4）。
- **关闭门控（F2，引擎/状态机强约束）**：`residual_level ∈ {HIGH, VERY_HIGH}` 的风险点，**必须存在有效 `risk_acceptance`（approver/at/evidence_id 齐全）** 才能从 `DONE → VERIFIED`；否则阻断关闭。对应 D3-2 **TC-M2-106（未接受不可关闭）**。

### 3. 混合评估对象的汇总算法（C4，写明）
同一 `assessment` 可混合三类对象，汇总到同一评估结论：
1. **条款核查项**（域→控制项→题）：按 `item.weight` 加权得分 → 映射 risk_level；可物化为 risk_finding（`atv_id` 为空）。
2. **A-T-V 风险项**：每行 `risk_score = L×I` → risk_level → 物化为 risk_finding（`atv_id` 非空）。
3. **定性叠加项**（`qualitativeOverlay`）：在定量基础上按 `weight` 叠加调整（如供应商 SLA 履约）。
- **汇总规则**：评估级综合 = 各 risk_finding 的 `risk_score` 按"取最大档"或"加权聚合"二选一（模板 `scoring_model.aggregation ∈ {MAX_LEVEL, WEIGHTED}` 声明，默认 `MAX_LEVEL` 以安全优先：综合等级不低于任一高危风险点档位）；综合等级仍由平台基线求值，确保与 M9 口径一致。

## 十四·补6、控件测试结果复用端到端机制（功能链路补全 · C11）

> 落地决策 C11：补齐 `control_test_result` 的**生产者 → 复用查找 → 失效**完整链路，使 §7.2"一次测试结果，多体系复用"在工程上闭环；跨子公司不复用（org_id 隔离优先）。

### 1. 生产者（落库）
- 评估提交时，对每个答案 `assessment_answer`，按 `answer.control_id`（D1-2 §7 已有冗余列）落一条 `control_test_result(org_id, asset_id, control_id, result, tested_at, evidence_id, reuse_window_days)`；
- `result` 取值与 D1-2 一致（符合/部分不符/不符合）；`reuse_window_days` 由模板/控件配置给出复用有效期。

### 2. 复用查找（预填）
- 新评估渲染/填写期，对引用了控件 `control_id` 的题目，按键 `(org_id, asset_id, control_id)` 查最近一条 `control_test_result`；
- **命中条件**：`tested_at + reuse_window_days ≥ now()`（在复用窗口内）且证据未失效 → **预填**该题结果与佐证，标注"复用自 YYYY-MM-DD 的测试结果"，允许人工覆盖。
- **跨子公司不复用**：查找键强制含 `org_id`，**不跨 org_id 命中**（org_id 隔离优先于复用收益）。

### 3. 失效（级联）
以下任一发生即令对应 `control_test_result` 失效（不再被复用预填）：
1. **超窗**：`tested_at + reuse_window_days < now()`；
2. **控件变更**：被引用控件 `control_library` 条目内容/版本变更；
3. **证据失效**：关联 `evidence` 失效（撤回/过期/校验失败）→ 级联失效。
- 失效后新评估不再预填，需重新测试落新结果。
- 配套测试呼应 D3-2 控件复用相关用例（命中预填 / 超窗不预填 / 跨子公司不命中 / 证据失效级联）。

## 十四·补7、EFFECTIVE 版本只读 + 启动快照防击穿（功能链路补全 · F2 启动快照）

> 落地：模板"已发布版本就地 UPDATE"会击穿进行中评估的启动快照（§5 承诺的"进行中评估不受后续修订影响"）。补齐**版本只读 + 修订走新版本**的结构约束。

### 1. 版本只读（EFFECTIVE 不可就地改）
- `template_version.status = EFFECTIVE` 后，其结构（`template_section/template_item`）、打分（`template_item_scoring`，见 §十四·补8）、报告映射（`report_format`）**全部只读**，禁止就地 `UPDATE`；
- 任何修订必须**克隆出新 `template_version`（status=DRAFT）** 调整 → 预览/测试 → 置 EFFECTIVE，老版本保持不变（呼应 §五 克隆继承、§八 发布前测试）。
- 工程约束建议：对 EFFECTIVE 版本的结构/打分/报告子表写操作在数据访问层拦截（或 DB 触发器/行级约束），违例报错；操作留痕入 `operation_log`。

### 2. 启动快照（进行中评估锚定版本）
- 评估启动即绑定 `assessment.template_version_id` → 渲染器、打分器、报告器**一律读该快照版本**；
- 因 EFFECTIVE 只读，进行中评估的题目/权重/报告映射**不被任何后续修订击穿**；新版本仅对"启动晚于新版本生效"的新评估生效。
- 与 §十四·补8 `template_item_scoring` 协同：改打分=新版本的新 scoring 行，不回写老版本行，进行中评估的打分稳定。

## 十四·补8、template_item_scoring 真分表（功能链路补全 · A4 / G1）

> 落地决策 A4：新增 `template_item_scoring` **独立映射表**，把"打分/报告映射"从结构表 `template_item` 物理拆出，使 §4.3"改报告不动表单 / 改表单不破打分"在**结构上**成立（而非仅靠约定）。

### 1. 建表（建议 D1-2 §7 展开）

```sql
-- 打分/报告映射独立分表（与结构表 template_item 解耦）
CREATE TABLE template_item_scoring (
  id BIGINT PRIMARY KEY,
  item_id BIGINT NOT NULL REFERENCES template_item(id),  -- 结构题目
  version_id BIGINT NOT NULL REFERENCES template_version(id),
  weight NUMERIC(6,3),                                    -- 题目权重（原 item.weight 迁出）
  report_section_key VARCHAR(64),                         -- 报告章节映射（原 item.report_section_key 迁出）
  severity_suggest VARCHAR(12),                           -- 严重度建议（审计场景，供人工参考）
  UNIQUE(item_id, version_id)
);
```

### 2. 解耦在结构上成立的两条路径
- **改报告映射不触碰结构表**：调整 `report_section_key` / `severity_suggest` 只写 `template_item_scoring`，`template_item`（题型/校验/跳题/层级）**零改动** → 报告格式调整不影响表单结构。
- **改表单结构不破打分**：调整 `template_item`（增删题、改题型/跳题）时，`template_item_scoring` 通过 `item_id` 关联，权重/映射独立维护；删题时其 scoring 行随之清理，留存题不受影响 → 表单字段调整不破坏既有打分逻辑。
- 原 §四中 `item.weight / item.report_section_key` 的内嵌语义，迁移为 `template_item_scoring.weight / report_section_key`（结构表 `template_item` 不再承载打分/报告列）；§4.1 / §4.2 表述据此理解，求值与映射一律读分表。
- 与 §十四·补7 协同：scoring 行随 `version_id` 版本化，EFFECTIVE 只读，修订走新版本 → 启动快照下打分稳定。
