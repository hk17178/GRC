# GRC 平台 · 表单引擎专项设计文档

**文档编号：** D1-6 ｜ **版本：** v1.0（草案） ｜ **日期：** 2026-06-23
**依据：** 需求 V1.9（§4.2.2/4.2.3/4.4、§5.2.2）、D1-1 架构、D1-2 数据模型 §7
**状态：** 待评审

> 表单引擎是**全项目最高难点与最大工作量来源**（可行性评估）。它服务 M2 风险评估、M3 审计检查表、M7 供应商评估——三者**共用同一引擎**，不另建第二套配置能力。

---

## 一、设计目标与核心矛盾

### 1.1 目标
让运营人员（信息安全/合规/内审）**不写代码**即可配置出符合各体系要求的评估/审计表单、打分逻辑与报告格式，并支持新体系平滑纳入。

### 1.2 核心矛盾（需求 4.4，必须正面回应）
> "表单引擎若做得过于通用，容易退化为万能表单生成器而牺牲易用性（运营人员不知道如何配出一份合规的等保测评表）。"

**对策（双轨并行，缺一不可）：**
1. **通用配置能力**：元数据驱动，结构/打分/报告皆可配；
2. **内置体系脚手架**：预置 8 套体系的可用模板，运营从"克隆调整"起步而非"空白画起"。

### 1.3 设计原则
- 结构配置层 与 打分/报告映射层 **两层解耦**（互不牵连）；
- 模板**版本化 + 启动快照**（进行中评估不受后续修订影响）；
- 控件库**复用优先**（跨体系一次测试多处用）；
- 配置治理**集中管控、分级开放**（合规底线不被业务无意破坏）。

## 二、总体架构（元数据驱动）

```
                 ┌─────────────── 模板（template）──────────────┐
                 │  framework 体系   usage 用途(评估/审计)         │
                 └───────────────────┬─────────────────────────┘
        ┌────────────────────────────┴────────────────────────────┐
        │                  模板版本 template_version                │
        │       version_no · effective_date · status               │
        │  ┌─ 结构配置层 ─┐        ┌─ 打分/报告映射层 ─┐             │
        │  │ section 层级 │        │ scoring_model     │             │
        │  │ item 题目    │◄──解耦──┤ report_format     │             │
        │  │ 题型/校验/跳题│        │ item.weight       │             │
        │  │ item→控件引用 │        │ item.report_key   │             │
        │  └──────┬───────┘        └───────────────────┘             │
        └─────────┼──────────────────────────────────────────────────┘
                  │ 引用
         ┌────────▼─────────┐   control_framework_map（跨体系映射）
         │ control_library   │◄──────────────────────────────────────
         │ 统一控件库(控制点) │
         └──────────────────┘
  运行期：渲染器读「版本快照」动态生成表单/报告；答案存 assessment_answer(JSONB)+抽取打分列
```

**关键点**：模板=数据；引擎=通用渲染/打分/出报告器。**新增体系/调整模板 = 改数据，不改引擎代码。**

## 三、结构配置层（Structure）

### 3.1 层级（域 → 控制项 → 具体问题）
- `template_section` 自引用，深度与命名（如"控制点编号"）**可按体系配置**（需求 4.2.2）。
  - 等保三级：按"安全通用要求 + 安全扩展要求"分类 + 控制点编号；
  - ISO 27001：资产-威胁-脆弱性-可能性×影响 四要素；
  - 供应商：合同条款 + 履约表现（无"控制点"概念）。

### 3.2 题型（item_type）
单选 / 多选 / 打分题（如 0–5 级成熟度）/ 文本说明 / 矩阵题 / 附件上传（佐证）。

### 3.3 校验与跳题
- 必填校验规则；
- 跳题逻辑 `skip_logic`（如某题答"不适用"→自动跳过其关联后续问题）。

### 3.4 多级填写与提交流程
- 流程节点（如 系统负责人填写 → 部门负责人确认）由 **M8 审批流引擎（Flowable）** 实现，引擎只定义"需要几级、谁审"，不自研流程。

## 四、打分与报告映射层（Scoring & Report）

### 4.1 打分模型（scoring_model JSON，模板版本级）
- **默认定量**：可能性 × 影响 = 风险值，便于跨对象/跨周期横向对比与看板聚合（需求 4.2.2/V1.7）。
- **叠加定性**：供应商等难量化维度，在定量基础上叠加定性判断项（如合同履约），非完全切换。
- **可选方式**：定性成熟度等级（0–5）、简单合规判定（是/否/不适用）保留供特定场景选用。
- 每题/每题类**权重**（item.weight）。
- **风险分级口径（五级，统一标准）**：风险值 → 区间映射至**极低/低/中/高/极高**（`VERY_LOW/LOW/MID/HIGH/VERY_HIGH`），与 D1-2 `risk_level`、看板/标签/分布图、M9 聚合完全一致（见 CR-001-7）；**不再使用三级（高/中/低）口径**。`levelMatrix` 示例区间见下，体系模板可按需调整阈值但级数固定为五级。

```json
// scoring_model 示例
{
  "method": "QUANTITATIVE",
  "formula": "likelihood * impact",
  "levelMatrix": [
    {"max":3,"level":"VERY_LOW"},
    {"max":6,"level":"LOW"},
    {"max":12,"level":"MID"},
    {"max":20,"level":"HIGH"},
    {"max":25,"level":"VERY_HIGH"}
  ],
  "qualitativeOverlay": [{"itemRef":"sla_perf","weight":0.2}]
}
```

### 4.2 报告章节映射（report_format + item.report_section_key）
- 每题得分**映射到报告的目标章节**（如某控制项得分归入"异常处置"章节而非"网络安全"，需求 4.2.2）。
- `report_format` 固化章节顺序/措辞规范（随模板版本固化，确保同体系不同周期产出格式一致，便于监管比对）。

### 4.3 两层解耦的价值
- 调整报告格式 → 只改 `report_format`/`report_section_key`，**不动表单题目**；
- 调整表单字段 → 不破坏既有打分逻辑（需求 4.2.2/4.4）。

## 五、模板版本化与继承

| 能力 | 机制 |
|---|---|
| 版本号 + 生效日期 | `template_version` |
| 启动快照 | 评估启动即绑定 `template_version_id`，后续模板修订不影响进行中评估（需求 4.2.2） |
| 克隆继承 | 基于既有版本/模板克隆后局部调整，而非空白画起（脚手架的落地方式） |
| 发布前测试 | `status=DRAFT` 可预览/测试填写，验证通过才置 `EFFECTIVE`（需求 4.2.2 配置治理） |

## 六、内置体系脚手架（需求 V1.6）

预置可用模板：**ISO 27001、等保三级、PBOC、PCI DSS、ISO 27701、ISO 27018、ISO 9001、供应商风险评估**；并支持**平台管理员自定义新增体系分类**，新体系可从零搭建或克隆调整——公司未来新增认证/监管要求时**无需开发改动即可纳入**。

> 脚手架是"易用性"的来源：运营从一份已合规的模板起步，避免不知如何配出合规的等保测评表。

---

## 七、统一控件库 ⭐（需求 V1.7）

### 7.1 颗粒度
按**"具体控制点"**抽象（如"特权账号定期审查"），而非笼统的"控制域"（如"访问控制"）。

### 7.2 跨体系映射与复用
- 体系模板**引用**控件库条目，而非各自重复定义题目；
- 同一控件被多体系评估后，支持**"一次测试结果，多体系复用"**（control_test_result + 复用有效期），减少重复填表与重复举证（需求 4.2.2）。
- `control_framework_map` 维护控件在各体系下的条款编号映射。

### 7.3 成本权衡与维护工具（需求 4.4/V1.7）
- "具体控制点"颗粒度带来**更高初始建设与维护投入**；
- 配套**控件维护工具**：控件批量管理、跨体系映射的批量维护、影响分析（改一个控件→提示受影响的模板/体系），降低长期维护成本。

### 7.4 落地建议（呼应可行性评估）
- **接受"先重复、后抽象"**：一期先做"模板克隆 + 基础结构配置"，统一控件库的跨体系复用作为**二期**，用真实体系数据驱动控件库演进，不一开始追求完美映射。

## 八、配置治理（集中管控、分级开放，需求 V1.7/10.2）

| 操作 | 权限角色 | 管控 |
|---|---|---|
| 模板新建、结构性修改（题目层级、打分逻辑、报告章节映射） | **表单/模板结构配置管理员**（归口信息安全部） | 纳入 M8 审批，避免合规底线被业务无意破坏 |
| 模板内局部参数（权重微调、提醒文案、个别选项） | **表单/模板局部参数调整人员**（合规/内审等业务部门） | M8 控制可调范围 |
| 已发布模板的任何修改 | 上述角色 | **必须经预览/测试填写验证后才正式生效**，避免改动导致进行中评估无法提交或打分出错 |

## 九、与 M3 审计 / M7 供应商的复用

- **统一引擎，按 `template.usage` 区分**（ASSESSMENT / AUDIT）：同一套模板管理后台，按"用途"字段区分，不另建独立配置入口（需求 5.2.2）。
- 审计场景差异化补充（在通用引擎之上）：
  - 检查项**引用 M1 制度条款**作为依据，报告自动带出条款编号（需求 5.2.2）；
  - 检查项与严重度评级关联（判不合规时建议严重度，供审计人员参考调整，不强制覆盖人工判断）；
  - 补充审计专有脚手架（等保测评检查表、内部合规审计检查表）。
- M7 供应商：供应商专用模板（定量叠加定性），复用整改与跟踪机制。

> 结论：模板版本管理、权限治理、预览测试等机制**只实现一次**，杜绝两套配置能力逐渐能力不一致（需求 5.4）。

## 十、报告自动生成

- 输入：评估答案 + scoring_model + report_format + item.report_section_key；
- 输出：风险结论、风险点清单、等级分布统计图，按目标体系章节结构组织（需求 4.2.3）；
- 人工复核签批后 `EFFECTIVE`（不完全依赖自动化结论）；
- 导出 Word/PDF（异步生成，§D1-1）。

## 十一、运行期渲染与执行

- **前端动态渲染**：读模板版本快照元数据，生成填写界面（题型/校验/跳题）；多主题令牌驱动样式。
- **填写**：答案存 `assessment_answer.value`(JSONB)，佐证附件存 MinIO；打分实时/提交时计算并抽取到列。
- **填写流程**：经 M8 多级提交。
- **进行中保护**：会话中断自动保存草稿（需求 13.2）。

## 十二、关键技术风险与对策

| 风险 | 对策 |
|---|---|
| 过度通用 → 退化为万能表单生成器 | 通用配置 + 内置脚手架双轨；不可偏废（需求 4.4） |
| 两层耦合致后期维护困难 | 结构与打分/报告分表存储、分别演进 |
| 已发布模板被误改致评估中断 | 修改权限分级 + 发布前预览测试强制 |
| 控件库颗粒度维护成本高 | 控件维护工具 + 批量映射 + 影响分析；先重复后抽象 |
| 报告格式漂移 | report_format 随版本固化 |

## 十四、验收标准（关键）
1. 不写代码可配出一份**可用的等保三级测评表**并完成评估、出报告；
2. 模板改报告格式不影响表单；改表单不破坏打分；
3. 进行中评估不受模板升级影响（版本固化）；
4. 同一控件在两个体系间复用测试结果；
5. 结构性修改与局部参数调整权限分离且生效前强制测试；
6. 新增一个自定义体系并克隆出可用模板，全程零开发改动。

## 十四·补、风险建模与五级矩阵增补（CR-001 / CR-002）

**1. 资产-威胁-脆弱性(A-T-V)建模**：评估表单支持以「资产 → 威胁源 → 脆弱性 → 可能性 → 影响」四要素建模（落地 ISO 27005 / 等保风险评估方法），打分以 A-T-V 组合为最小单元，汇总为风险项。

**2. 风险处置与残余风险**：评估报告新增——固有风险等级(`inherent_level`) → 处置决策(`treatment_decision`：降低/接受/转移/规避) → 残余风险等级(`residual_level`) → 管理层接受签认(`risk_acceptance`，含签认人/时间/留痕)。报告映射新增上述区块。

**3. 五级矩阵**：`levelMatrix` 由 3 级改为**五级**——极低/低/中/高/极高(`VERY_LOW/LOW/MID/HIGH/VERY_HIGH`)，风险值→区间映射统一口径，与看板/标签/分布图、M9 聚合一致（见 CR-001-7）。

**4. 综合风险指数可解释**：指数计算需输出**构成明细**（各风险域得分与权重）、**环比归因**（较上期变化来源）、**计算口径说明**，支持从指数下钻到风险项。

**5. 供应商评估复用**：`assessment` 新增 `kind`(准入/年度定期/续约/退出/事件复评)，target=vendor，复用同一打分/报告/工单闭环；技术安全与 DPA 维度作为供应商专属控件补充（详见 §十四·补2）。

## 十四·补2、供应商专属控件与 A-T-V 结构层建模（V1.9 对齐 · CR-001 / CR-002）

### 1. 供应商专属控件（技术安全 + DPA 维度）
在通用控件库之上，为 M7 供应商评估补充一组**专属控件分类**，统一纳入控件库（控制点颗粒度），供供应商模板引用、并可被多体系/多供应商复用：

| 维度 | 专属控件（控制点）示例 |
|---|---|
| **技术安全** | 加密传输与存储、漏洞与补丁管理、访问与身份管理、日志与监测、备份与可恢复性、安全事件通报时限 |
| **数据保护 / DPA** | **数据驻留**（处理/存储地域与跨境约束）、**PCI 范围**（是否触及持卡人数据环境 CDE 及合规证明）、**再委托**（分包/次级处理者审批与穿透约束）、**最小权限**（数据访问按需授权、离场销毁）、数据处理协议(DPA)条款落实、个人信息删除/返还义务 |

> 上述维度作为**定量叠加定性**模型的一部分：技术安全可定量打分，DPA 类多为合规判定（是/否/不适用）+ 佐证附件；缺失关键项可触发准入阻断或风险升级。

### 2. A-T-V 作为结构层建模，与"域 → 控制项 → 题"层级共存
- **A-T-V（资产-威胁-脆弱性）是结构配置层的一种建模方式**，落地 ISO 27005 / 等保风险评估方法：以「资产 → 威胁源 → 脆弱性 → 可能性 × 影响」为最小打分单元，汇总为风险项。
- 它**不取代** §三的「域 → 控制项 → 具体问题」层级，而是**与其共存**：
  - 合规符合性类评估（等保测评检查表、ISO 27001 控制项核查、供应商条款核查）采用「域 → 控制项 → 题」层级；
  - 风险评估类对象采用 A-T-V 四要素建模；
  - 二者由 `template.framework / usage` 与 `template_section` 的层级定义区分，**同一引擎、同一控件库**承载，互不排斥，可在同一评估对象内混合引用（如供应商评估既有条款核查项，又有针对关键资产的 A-T-V 风险项）。

## 十四·补3、表单引擎与全局自定义字段/视图能力的关系（V1.9 对齐 · CR-001-2）

CR-001-2 将"高自由度可配置"全局化（自定义字段 / 视图筛选 / 看板 / 报表 / 工作流 / KPI / 通知场景），本节厘清表单引擎与该能力的边界与协作。

### 1. 控件库 vs 自定义字段（边界）
| 维度 | 表单引擎·控件库（控制点） | 全局·自定义字段 |
|---|---|---|
| 定位 | 评估/审计**题目与控制点**，承载打分逻辑与跨体系复用 | 在**评估对象/供应商/审计等业务实体**上扩展业务属性（如供应商联系人、合同到期日、自定义标签） |
| 是否参与打分 | 是（item.weight / scoring_model） | 默认否（业务字段，用于展示/筛选/分组） |
| 存储 | `assessment_answer.value`(JSONB) + 抽取打分列 | 业务实体的 **JSONB 扩展列 + 字段元数据登记**（CR-001-2 / D1-2） |
| 治理 | §八 配置治理（结构性归口管理员 + 发布前测试） | 沿用同一"集中管控、分级开放"思路，纳入 M8 权限与变更留痕 |

> 原则：**打分相关 → 走控件库题目；纯业务属性 → 走自定义字段。** 避免把业务字段塞进控件库导致打分模型污染，也避免把控制点降级为无打分语义的自定义字段。

### 2. 评估对象自定义业务字段在表单中的引用
- 评估对象上登记的**自定义业务字段**（经字段元数据登记后）可被表单**只读引用 / 预填**（如自动带出"系统等级""数据分类""合同到期日"），减少重复录入；
- 引用方式：`item` 支持绑定一个**字段引用源**（`field_ref`，指向业务实体的自定义字段键），渲染期由引擎解析取值；此类引用项**不进入打分**，仅作上下文展示或作为跳题/校验条件输入。

### 3. 视图 / 看板 / 报表的协作
- 表单引擎产出的评估结果（含五级风险等级、残余风险、A-T-V 风险项）作为**数据源**，由全局**自定义视图筛选 / 看板 / 报表**能力消费；
- 报告章节固化（§四 report_format）解决"同体系跨周期格式一致、便于监管比对"；自定义报表则面向**运营自助的灵活汇总分析**，二者并行不冲突。

3. 配置治理的角色与发布前测试机制。
4. 与 M3/M7 复用的边界（usage 区分 + 差异化补充）。
