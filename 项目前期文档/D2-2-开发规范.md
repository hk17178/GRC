# GRC 平台 · 开发规范

**文档编号：** D2-2 ｜ **版本：** v1.0（草案） ｜ **日期：** 2026-06-23
**依据：** D1-1 架构 ｜ **状态：** 待评审

---

## 一、目的
统一工程实践，保障代码质量、可维护性与"模块不牵一发动全身"在代码层落地。

## 二、技术栈基线
前端 Vue 3 + TypeScript + Vite；后端 Spring Boot 3（Java 17）+ Flowable；DB PostgreSQL 16（pgvector）；Redis；MinIO。

## 三、代码组织与模块边界 ⭐
- 后端按模块分包：`com.mandao.grc.<module>`（m1policy、m2risk、m8auth、core 等）；横向内核在 `core`。
- **模块只暴露 `api` 子包（接口/DTO）**，内部 `domain/infra` 私有。
- **依赖规则（CI 强制）**：业务模块→core 允许；业务模块↔业务模块禁止直连；禁止环依赖。用 **ArchUnit / Spring Modulith** 在构建期校验，越界即失败。
- 跨模块协作只走 接口 或 领域事件。

## 四、编码规范
- 命名：类 PascalCase、方法/变量 camelCase、常量 UPPER_SNAKE；表/列 snake_case。
- 分层：Controller→Application→Domain→Infrastructure，禁止跨层穿透。
- 异常：统一异常 + 错误码；不吞异常；对外不泄露堆栈。
- 隔离上下文：数据访问统一经拦截器注入 visibleOrgs，**禁止业务代码手写 org 过滤**或裸 SQL 绕过。
- 配置：业务可变项走配置/元数据，不硬编码（需求 13.5）。
- 日志：结构化；敏感数据脱敏；关键操作写 operation_log；强制写操作审计的动作清单与 audit trail 约束见 §九补。
- 多域身份：用户唯一键统一 `(domain_id, username)`，**禁止以全局 username 作主键/外键**（见 §四补·CR-003）。
- 注释/文档：公共接口需 Javadoc/TSDoc；复杂逻辑注明意图。

## 五、前端规范
- 组件基于设计令牌（tokens.css 多主题），不写死颜色/圆角/阴影，统一引用语义令牌。
- 状态管理 Pinia；路由懒加载；接口统一封装（鉴权/错误/分页）。
- 可访问性：遵循设计规范无障碍约定（prefers-reduced-motion 等）。
- 国际化：**所有界面文案一律走 i18n 资源键，禁止在组件/模板中硬编码中英文字面量**；日期/数字/时区一律经本地化工具格式化（详见 §五补·i18n）。

## 六、Git 与分支策略
- 主干 `main`（可发布）+ 特性分支 `feature/*` + 修复 `fix/*`。
- 提交信息：`类型: 摘要`（feat/fix/docs/refactor/test/chore）。
- **合并需 PR + Code Review 通过 + CI 绿**。
- 禁止直接推 main；禁止提交密钥/敏感配置（.gitignore + 扫描）。

## 七、CI/CD 流水线
构建 → 单元测试 → **模块依赖检查** → SAST/SCA 扫描 → 集成测试 → 制品打包（镜像）→ 部署测试环境。
- 安全门禁：高危漏洞阻断；越权专项与上线安全测试通过方可发布（D2-1 §8）。

## 八、数据库变更
- 用 **Flyway** 版本化迁移；脚本评审后入库；向前兼容（升级不破坏进行中评估）。
- 禁止手改生产库结构。

## 九、测试要求
- 关键逻辑（打分、隔离、状态机、规则）单元测试覆盖；
- 接口/集成测试纳入流水线；
- 提交即跑冒烟。

## 十、评审重点
模块依赖 CI 校验、隔离上下文强制、配置化落地是否在规范中可执行。

## 四·补（V1.9 对齐 · CR-003 多域身份）
- **用户唯一键统一为 `(domain_id, username)`**：username 在全平台不唯一，**禁止以全局 username 作主键/外键**；所有引用用户的外键须携带 `domain_id`（或指向用户内部代理主键 `user_id`，再由 `user_id`→`(domain_id, username)` 映射）。
- **identity_source 与服务账号密钥按域隔离**：每个 `identity_source`（各子公司/海外独立 AD 域/林）的连接配置、绑定服务账号及凭证按域独立存放、独立轮换，入密钥库不落明文；**禁止跨域共用一套服务账号或密钥**。
- **认证路由与 org_id 域映射在基础设施层实现，禁止在业务模块硬编码**：`DOMAIN\user`、`user@domain` 解析、域→`identity_source`→`org_id` 的映射归属内核/基础设施层（core / auth 基础设施），业务模块只消费已解析的主体上下文，不得感知或写死具体域名、域控地址、映射关系。
- **不依赖域间信任**：各域认证彼此独立；单域故障仅影响该域登录，并以权限快照维持已登录会话的鉴权，业务代码不得假设全域在线。

## 五·补（V1.9 对齐 · i18n）
- **文案外置语言包**：中英文文案集中维护于语言包资源文件，组件经资源键引用；新增文案必须同时补齐中英两套键，缺键 CI 校验。
- **禁止硬编码**：模板/脚本中不得出现裸中英文字面量（含枚举展示名、提示语、表头、按钮文案）；后端返回面向用户的文案亦走错误码 + 资源键，不在代码内拼接最终展示串。
- **本地化工具统一处理**：日期、时间、数字、货币、时区一律经统一本地化工具格式化与解析，禁止手写 `format`/字符串拼接；时区按用户偏好或租户配置渲染，存储统一 UTC。
- 顶栏语言切换、内容可选双语等表现层能力由资源键 + 本地化工具支撑，业务逻辑不感知当前语种。

## 九·补（V1.9 对齐 · 操作审计日志 / audit trail）
- **强制写操作审计日志的动作清单**（命中以下任一动作必须落审计，缺失即评审不通过）：
  · 配置变更（含自定义字段/视图/看板/报表/工作流/KPI/通知场景等可配置项发布）
  · 数据导出（任何导出/下载敏感或批量数据）
  · 权限变更（角色授予/回收、数据可见范围调整、域映射变更）
  · 制度变更（制度发布/废止/版本变更）
  · 阈值变更（风险评分阈值、KRI/预警阈值、报送时限等参数调整）
  · AI 配置变更（模型/端点/API Key/RAG/提示词/"数据不出域"开关等变更）
  · 反馈出站（建议与反馈 M-FB 经审批后向外部 Git 推送的出站动作）
- **audit trail 独立存储**：审计日志与业务库逻辑/物理隔离独立表（或独立存储），不与业务数据混表。
- **追加写、不可更新/删除**：审计记录仅 append，禁止 UPDATE/DELETE；以约束/触发器或专用追加写通道保证不可篡改，保留链路可校验。
- **统一切面埋点**：审计落库经统一 AOP 切面/拦截器自动埋点（操作主体、`(domain_id,username)`、org_id、动作、对象、前后值摘要、时间、来源 IP），**禁止各业务模块各自手写散落的审计写入**，避免漏埋与口径不一致。

## 五·补2（V1.9 对齐 · CR-001-2 自定义字段）
- **自定义字段统一经 JSONB 扩展列 + 字段元数据登记**：高自由度自定义字段写入约定的 JSONB 扩展列，并在字段元数据登记表登记（键、类型、校验、可见性、是否检索等）；**禁止为业务自定义需求随意新增物理列**或临时改表。
- 自定义字段的筛选/看板/报表/校验均基于元数据驱动；发布前预览、分级开放与变更留痕按 CR-001-2 集中管控。
