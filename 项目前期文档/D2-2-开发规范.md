# GRC 平台 · 开发规范

**文档编号：** D2-2 ｜ **版本：** v1.0（草案） ｜ **日期：** 2026-06-23
**依据：** D1-1 架构 ｜ **状态：** 待评审

---

## 一、目的
统一工程实践，保障代码质量、可维护性与"模块不牵一发动全身"在代码层落地。

## 二、技术栈基线
前端 Vue 3 + TypeScript + Vite；后端 Spring Boot 3（Java 17）+ Flowable；DB PostgreSQL 16（pgvector）；Redis；MinIO。

## 三、代码组织与模块边界 ⭐
- 后端按模块分包：`com.mandao.grc.<module>`（m1policy、m2risk、m8auth、core 等）；横向内核在 `core`。
- **模块只暴露 `api` 子包（接口/DTO）**，内部 `domain/infra` 私有。
- **依赖规则（CI 强制）**：业务模块→core 允许；业务模块↔业务模块禁止直连；禁止环依赖。用 **ArchUnit / Spring Modulith** 在构建期校验，越界即失败。
- 跨模块协作只走 接口 或 领域事件。

## 四、编码规范
- 命名：类 PascalCase、方法/变量 camelCase、常量 UPPER_SNAKE；表/列 snake_case。
- 分层：Controller→Application→Domain→Infrastructure，禁止跨层穿透。
- 异常：统一异常 + 错误码；不吞异常；对外不泄露堆栈。
- 隔离上下文：数据访问统一经拦截器注入 visibleOrgs，**禁止业务代码手写 org 过滤**或裸 SQL 绕过。
- 配置：业务可变项走配置/元数据，不硬编码（需求 13.5）。
- 日志：结构化；敏感数据脱敏；关键操作写 operation_log。
- 注释/文档：公共接口需 Javadoc/TSDoc；复杂逻辑注明意图。

## 五、前端规范
- 组件基于设计令牌（tokens.css 多主题），不写死颜色/圆角/阴影，统一引用语义令牌。
- 状态管理 Pinia；路由懒加载；接口统一封装（鉴权/错误/分页）。
- 可访问性：遵循设计规范无障碍约定（prefers-reduced-motion 等）。

## 六、Git 与分支策略
- 主干 `main`（可发布）+ 特性分支 `feature/*` + 修复 `fix/*`。
- 提交信息：`类型: 摘要`（feat/fix/docs/refactor/test/chore）。
- **合并需 PR + Code Review 通过 + CI 绿**。
- 禁止直接推 main；禁止提交密钥/敏感配置（.gitignore + 扫描）。

## 七、CI/CD 流水线
构建 → 单元测试 → **模块依赖检查** → SAST/SCA 扫描 → 集成测试 → 制品打包（镜像）→ 部署测试环境。
- 安全门禁：高危漏洞阻断；越权专项与上线安全测试通过方可发布（D2-1 §8）。

## 八、数据库变更
- 用 **Flyway** 版本化迁移；脚本评审后入库；向前兼容（升级不破坏进行中评估）。
- 禁止手改生产库结构。

## 九、测试要求
- 关键逻辑（打分、隔离、状态机、规则）单元测试覆盖；
- 接口/集成测试纳入流水线；
- 提交即跑冒烟。

## 十、评审重点
模块依赖 CI 校验、隔离上下文强制、配置化落地是否在规范中可执行。
