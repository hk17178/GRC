## 二、分项交底（展开）

### H-01 调度/到期扫描内核（P0，一期）
- **来源**：D1-1 §5.12（独立工作进程，只产时间事件不做分发决策）、D1-4 调度章（产出 11 个时间事件）。
- **待细化**：① 扫描周期（建议按事件类别分级：法定时限类高频 5–15min，周期摘要类按 cron）；② **分布式锁/单实例触发**（多实例下用 ShedLock/数据库咨询锁，避免重复发事件）；③ **幂等台账**已设计 `audit_plan_reminder_log(audit_plan_id,reminder_day) UNIQUE`，需为其它阈值事件补同型台账或统一 `reminder_dispatch_log(object_type,object_id,event_type,threshold)`；④ 漏扫补偿（停机后重启回扫窗口）。
- **建议**：Quartz + ShedLock；扫描 SQL 走索引（见 H-12 对 `*_due/plan_start_date/reuse_window` 建索引）；事件只产不消，落总线后由 M10/M9 消费。
- **验收**：到期/临近/逾期/周期摘要事件**不漏不重**；TC-M3-104 幂等用例通过；单实例故障切换不重复发。

### H-02 防篡改哈希链（P0，一期）
- **来源**：D1-2 §13（operation_log/evidence 增 `curr_hash/prev_hash`，按 `org_id` 分链）、D1-3 §8（A3 决策）、`trusted_timestamp` 表。
- **待细化**：① **入库串行化**——同一 `org_id` 链内写入需串行以保证 `prev_hash` 正确（建议每 org 一个写入序列/行锁，评估高并发下的吞吐）；② **断链检测告警通道**（定时校验发现 hash 不连续 → 经 M10 紧急级告警安全负责人）；③ **离线重算校验作业**频率（建议每日 + 监管核查前手动触发）与失败处置；④ genesis（每链首条）规约。
- **验收**：TC-SEC-002 链完整性校验通过；人为篡改任一行 → 重算即发现并告警；跨子公司不串链、不互相泄漏。

### H-03 留存期重算（P1，二期）
- **来源**：D1-2 §十八、§23（`evidence_regulation_map` 变更触发 `retention_until` 重算）。
- **待细化**：`retention_until = max(各关联条款保留期, 入库日+口径下限)`，**只增不减**；映射新增/变更时重算；建"监管口径→保留年限"参数表（可配，避免硬编码）。
- **验收**：延长保留期的映射变更后 `retention_until` 正确增大且不回退；到期归档前二次校验。

### H-04 / H-05 / H-06 高自由度可配置（P1–P2，二期；建议并入 D1-8 专项）
- **来源**：D1-2 §22（custom_field_def / custom_view_def / custom_report_def / kpi_def / process_binding / dashboard_widget_config）、D1-1 §4.4、D1-3 §5.1·D7。
- **待细化**：
  - **H-04 生成列**：`is_aggregatable=true` 的 JSONB 键 → 宿主表 `GENERATED ALWAYS AS (ext->>'key')` 生成列的**运行期 DDL 编译**与发布前预览确认；ext 键名作用域前缀（`grp__key`/`sub_{orgId}__key`）防集团/子公司碰撞的落地校验。
  - **H-05 动态查询编排器**：自定义视图/报表/KPI 的查询构造**必须经统一数据访问层注入 `visibleOrgs`**，或运行在已 `SET app.visible_orgs` 的 RLS 会话；**禁裸连接/超级用户/`BYPASSRLS`**（见 D3-2 TC-SEC-102）。
  - **H-06 自定义工作流**：`process_binding(object_type,org_id,condition→process_definition_key,version)`，单据发起时**固化流程版本快照**（与 Flowable 部署版本绑定，后续改流程不影响在途单据）。
- **验收**：自定义字段可聚合进 M9 看板；自定义查询**无任何路径**绕过 `visibleOrgs`（红线断言用例必过）；自定义流程改版不影响在途审批。
- **建议**：H-04/05/06 体量大，建议整体并入文档体系规划已立项的 **D1-8《可配置性/低代码专项设计》**统一展开，本文仅交底其与隔离/治理的红线约束。

### H-07 风险评估·混合打分权重来源（P1，一/二期）
- **来源**：D1-6 §4（A-T-V 承载 + 汇总算法默认 `MAX_LEVEL`，`WEIGHTED` 为分支）、D1-2 §七。
- **待细化**：当一个评估对象同时含 条款核查项 + A-T-V 项 + 定性叠加项时，汇入同一 `assessment.risk_score` 的算法：默认 `MAX_LEVEL`（取最高残余等级，安全优先）已闭合；**`WEIGHTED` 分支的权重谁配、配在哪张表**（建议 `scoring_model` JSON 内增 `mix_weights`，或模板级配置）需在开发期定。
- **验收**：默认 `MAX_LEVEL` 行为确定；启用 `WEIGHTED` 时权重来源唯一、可追溯。

### H-08 各模块详设（P1，各期）

## 三、草稿 DDL → 正式建库脚本交底（H-12，DBA，建库期）

> D1-2 的全部 `CREATE TABLE` 为**设计稿**（重在字段/关系/约束的逻辑正确），正式迁移脚本须补：

1. **索引**：隔离锚点 `org_id`（所有业务表）；调度扫描字段 `reg_filing.statutory_deadline`、`audit_plan.plan_start_date`、`access_review.period`、`control_test_result.reuse_window 到期`、各 `*_deadline/due_date`；外键列；`kb_chunk` 的 pgvector HNSW（已在 §14）。
2. **RLS 策略**：对所有承载 `ext` JSONB 与敏感数据的 `org_id` 业务表 `ENABLE ROW LEVEL SECURITY` + `CREATE POLICY ... USING (org_id = ANY(current_setting('app.visible_orgs')::bigint[]))`（清单见 D1-3 §5.1·D7）。
3. **唯一/检查约束**：`app_user UNIQUE(domain_id,username)`；`risk_index_weight` 同 `(org_id,period)` 的 `SUM(weight)=1`（用触发器或应用层校验）；`audit_plan_reminder_log UNIQUE(audit_plan_id,reminder_day)`；`notification_escalation UNIQUE(scene_id,level_order)`。
4. **生成列**：A-T-V `risk_score = likelihood*impact`（`GENERATED ALWAYS AS`）；自定义可聚合字段生成列（H-04）。
5. **默认值/枚举**：状态枚举默认初值；审计字段（created/updated/by）统一；`platform_disabled DEFAULT false`。
6. **分区（按需）**：`operation_log`/`evidence`/`notif_log` 等高增长表评估按时间或 `org_id` 分区。
7. **迁移版本化**：Flyway/Liquibase 管理；与 D2-3 配置管理对齐。

## 四、重点链路时序补充（交底示范）

### 4.1 时间事件链：调度内核 → 事件 → 通知（H-01）
```
调度/到期扫描内核(定时, 分布式锁)
  └─扫描 reg_filing.statutory_deadline / audit_plan.plan_start_date+reminder_days / ...
     └─命中阈值 且 reminder_dispatch_log 无此(对象,阈值)记录
        ├─写 reminder_dispatch_log(幂等台账)
        └─发布时间事件(REG_FILING_DUE / EXT_AUDIT_PLAN_APPROACHING / ...) → 事件总线
                                                          │
        M10 通知中心 ◄─订阅─┤  按 notification_scene 匹配(事件→角色/层级/通道)
        M9 看板      ◄─订阅─┘  → 降噪/升级链 notification_escalation → 多通道发送 → 留痕
```
- 关键不变量：**生产者只有调度器**（业务模块不自发时间事件）；**幂等靠台账不靠时间窗**；单实例故障切换不重复。

### 4.2 举证链满足状态重算（C1 / H-03 关联）
```
触发源(任一)：control_test_result 变更 / evidence 失效或新增 / audit_finding 关闭或新增高危 / obligation_*_map 变更
  └─重算 obligation.satisfaction_status：
        关联 control 全通过且有未过期证据 → MET
        部分满足 → PARTIAL
        缺证据/控件未过 或 存在未关闭且 severity≥高 的关联 audit_finding → GAP
  └─状态变化 → 发 OBLIGATION_STATUS_CHANGED / OBLIGATION_GAP_DETECTED → M9看板/M10通知
正向查询：法规条款 ↓ obligation ↓ policy/control ↓ evidence（一条法规看到底）
反向取证：evidence ↑ evidence_regulation_map ↑ 法规条款（按条款反向取证）
```

### 4.3 数据不出域出站网关（C10 / D1）
```
任一 AI 出站调用(正式 或 /ai/models/{id}/test) 与 M-FB 仓库出站
  └─统一出站网关：
        if 数据不出域开关 = 开:
            目标为外部端点 → 直接拒绝(白名单不参与)；M-FB 出站目标强制内网 Git
        else (关):
            目标须命中出站白名单 → 否则拒绝
        对 base_url DNS 解析后 IP 复核：禁内网/保留地址(防 SSRF/DNS rebinding)
        放行调用 → 记出站审计(egress_audit / operation_log)
```
