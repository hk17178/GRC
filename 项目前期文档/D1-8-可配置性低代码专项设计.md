> **定位**：本文是"高自由度可配置/反硬编码"（质量目标 Q8）的**专项落地设计**，把 D1-9 交底的 H-04~H-06（自定义字段生成列、动态查询编排器、流程版本快照）展开到可实现层，并统一 7 类自定义能力的**治理模型与隔离红线**。功能审计中唯一"结构闭合、运行期待写"的 C8 链路由本文闭合。
>
> **第一性约束（不可破）**：自由度再高，也**绝不能绕过组织隔离**。所有自定义查询/视图/报表/KPI 必须经统一数据访问层注入 `visibleOrgs` 或运行在已 `SET app.visible_orgs` 的 RLS 会话；禁裸连接/超级用户/`BYPASSRLS`。

## 一、可配置面总览（7 类）

| # | 可配置能力 | 承载元数据表（D1-2 §22） | 运行期机制 | 隔离要点 |
|---|---|---|---|---|
| 1 | 自定义字段 | `custom_field_def` + 宿主表 `ext JSONB` | 元数据登记 → ext 读写 → 可聚合键编译生成列 | 随宿主 `org_id` 隔离；`is_sensitive` 脱敏 |
| 2 | 自定义列表视图/筛选 | `custom_view_def` | 动态查询编排器（白名单字段+注入 visibleOrgs） | 查询必经统一访问层 |
| 3 | 自定义看板/大屏 | `dashboard_widget_config` | 组件库装配 + 预览沙箱 | 组件数据源受 M8 |
| 4 | 自定义报表 | `custom_report_def` | 报表模板引擎（数据源经统一访问层） | 同上 |
| 5 | 自定义 KPI | `kpi_def` | KPI 求值引擎（formula DSL） | 口径与 M9/risk_index 一致 |
| 6 | 自定义工作流 | `process_binding` + Flowable | 发起时固化流程版本快照 | 审批仍走 M8 四元组 |
| 7 | 自定义通知场景 | `notif_scene_def` → `notification_scene` | 场景库装配（事件→角色/层级/通道） | 不跨子公司广播 |

> 此外 i18n 语言包亦为配置（外置语言资源），不在本文 7 类低代码范围，见 D1-1 §4.4。

## 二、统一治理模型（集中管控、分级开放）

> 7 类能力共用同一套治理骨架，避免每类各写一套。

### 2.1 分级开放（scope_level）
- 元数据表统一带 `scope_level ∈ {GROUP, SUBSIDIARY}`：**结构性/底线项归口集团管理员（GROUP）**，**局部项可下放子公司（SUBSIDIARY）**。
- 集团级定义对全集团生效；子公司级仅本子公司生效且**不得覆盖**集团已登记的同名项（见 §3.3 作用域前缀）。
- 谁能配 = 由 M8 权限控制（结构配置管理员 vs 局部参数调整人员，沿用 D1-3 §8 配置治理）。

### 2.2 三态生命周期（status）
```
DRAFT ──提交──► PREVIEW ──发布(管理员)──► EFFECTIVE
  ▲                                         │
  └──────────────回滚/下线◄──────────────────┘
```
- `DRAFT`：编辑中，不影响生产。
- `PREVIEW`：**发布前预览沙箱**——仅作用于配置者本人会话（或指定测试租户），不写生产视图/不进生产看板；用于验证配置正确。
- `EFFECTIVE`：正式生效；**已生效定义不可就地改**，修订须新建版本（`version`）→ 沙箱预览 → 再发布（与表单模板版本固化同理，防"改坏在用配置"）。

### 2.3 变更留痕与回滚
- 任何 DRAFT→EFFECTIVE / 回滚 / 下线 动作**全部写 `operation_log`**（操作人/前后值/审批人），纳入哈希链（D1-3 §8）。
- 支持回滚到上一 `EFFECTIVE` 版本；自定义看板/视图改坏可一键回滚，不污染生产驾驶舱。

### 2.4 发布前预览（PREVIEW 沙箱）落地
- 沙箱口径：PREVIEW 态定义在**配置者本人会话变量**下渲染（如 `app.preview_defs`），生产查询不读 PREVIEW 态；测试通过点"发布"才置 EFFECTIVE。
- 强制门禁：所有 EFFECTIVE 前必须经过一次 PREVIEW 验证（CI/界面强制），呼应需求 4.2.2 配置治理。

## 三、自定义字段机制（H-04）

### 3.1 登记 → 存储
- `custom_field_def(object_type, field_key, data_type, validation, is_sensitive, is_aggregatable, scope_level, status)` 登记；数据落宿主表统一的 `ext JSONB` 扩展列，键名与登记一致。
- 读写经**统一封装**（不裸操作 ext）：写时按 `validation` 校验、按 `data_type` 强类型化；读时按 `is_sensitive` 脱敏。

### 3.2 可聚合字段 → 生成列编译（运行期 DDL）
> 解决"JSONB 不可直接进 M9 聚合看板"。当 `is_aggregatable=true`：
```
管理员将字段置 EFFECTIVE 且 is_aggregatable=true
  └─平台生成 DDL：ALTER TABLE <host> ADD COLUMN g_<key>
        <type> GENERATED ALWAYS AS ((ext->>'<scoped_key>')::<type>) STORED;
  └─该生成列纳入 M9 物化视图刷新清单（自动）
  └─发布前预览确认该 DDL（避免误加列）；执行入 operation_log
```
- 生成列只读、随 ext 自动派生；M9 看板/KPI 可直接聚合。
- 失败处置：DDL 失败回退该字段为非聚合并告警。

### 3.3 作用域前缀防碰撞（隔离关键）
- `ext` 键名带作用域前缀：集团级 `grp__<key>`、子公司级 `sub_{orgId}__<key>`。
- 校验：子公司**不得**登记与集团 `grp__` 同名键；预览阶段做冲突检测；同一 JSONB 键空间内集团/子公司物理隔离。

## 四、自定义列表视图 / 筛选（H-05，隔离红线核心）

### 4.1 动态查询编排器
- `custom_view_def.definition`(JSON) 描述：可见列、筛选条件、排序、分组——**均为声明式**，不存裸 SQL。
- 编排器把声明式定义编译为参数化查询，**强制**：
  1. 字段**白名单**：只允许该 `object_type` 已登记字段（标准列 + `custom_field_def` 键），杜绝注入任意列/子查询；
  2. **注入 `org_id = ANY(visibleOrgs)`**（与统一数据访问层同一拦截路径）；
  3. 运行在已 `SET app.visible_orgs` 的会话（RLS 兜底）。
- **禁止**：裸 JDBC 连接、超级用户连接、`BYPASSRLS` 角色执行自定义查询（CI/代码评审红线）。

### 4.2 隔离断言
- 任意自定义视图/筛选/导出**无路径**可越过 `visibleOrgs`（对应 D3-2 **TC-SEC-102**：构造跨子公司条件的自定义视图，期望 0 行越界 + RLS 兜底）。

## 五、自定义看板 / 大屏（dashboard_widget_config）
- 组件库（指标卡/图表/信息流）+ 布局；用户/角色级上屏（合规态势大屏"添加组件"即此能力的原型）。
- 组件数据源一律走标准聚合接口（受 M8 + visibleOrgs），不自取数。
- 三态 + 版本 + scope_level（§2）；PREVIEW 沙箱仅本人会话，发布后进生产；改坏可回滚。

## 六、自定义报表（custom_report_def）
- `definition` 描述报表结构/数据集/参数；数据集经统一访问层（visibleOrgs）。
- 大批量导出异步（复用报表导出 Worker，D1-1）；导出动作入 `operation_log`。

## 七、自定义 KPI（kpi_def）
- `formula`(JSON DSL) 描述指标计算（聚合函数 + 维度 + 过滤）；求值引擎在**已注入 visibleOrgs** 的会话执行。
- **口径一致性红线**：KPI 涉及风险等级/风险域时，**必须复用平台统一 levelMatrix 与 `risk_domain` 字典**（与 M9 聚合、`risk_index_*` 同口径），不得自定义阈值，避免与统一口径冲突（呼应五级=平台统一阈值决策）。

## 八、自定义工作流（H-06，process_binding + Flowable）
- `process_binding(object_type, org_id, condition → process_definition_key, version)`：按对象类型+组织+条件绑定一个 Flowable 流程定义。
- **版本快照固化**：单据**发起时**记录 `process_definition_key + version` 到单据上，后续改流程定义**不影响在途单据**（与表单模板版本固化同理）。
- 审批节点仍走 M8 四元组与职责分离（不可自批）；自定义流程不绕过权限。

## 九、自定义通知场景（notif_scene_def → notification_scene）
- 设计态登记 `notif_scene_def`，运行态装配为 `notification_scene`（事件集→角色/层级→模板→通道→org_scope）+ `notification_escalation` 升级链。
- 新增场景**无需改码**，从场景库装配；M10 消费；不跨子公司广播（联动 M6/M8）。

## 十、隔离与安全红线（汇总，最高优先）

| 红线 | 落地 | 验证 |
|---|---|---|
| 自定义查询不破隔离 | 编排器强制注入 visibleOrgs + 字段白名单 + RLS 会话；禁裸连/超级用户/BYPASSRLS | D3-2 TC-SEC-102 |
| ext 宿主表全量 RLS | asset/vendor/policy/audit_plan 等带 org_id 业务表 ENABLE RLS（D1-3 §5.1·D7） | RLS 兜底用例 |
| 敏感自定义字段脱敏 | `is_sensitive` 键导出/问答按级别脱敏（D1-3 §7.3） | 脱敏用例 |
| 集团/子公司键不碰撞 | 作用域前缀 grp__/sub_{orgId}__ + 预览冲突检测 | §3.3 |
| 配置变更可追溯/可回滚 | 三态 + version + operation_log（哈希链） + 回滚 | §2.3 |
| 自定义 KPI 口径一致 | 复用统一 levelMatrix / risk_domain，不得自定义阈值 | §七 |

## 十一、类与编排（交底示范）

### 11.1 配置治理核心类（概念）
```
ConfigDefinition «抽象»            ← custom_field_def / custom_view_def / custom_report_def
  + orgId, scopeLevel, status, version, definition(JSON)        / kpi_def / notif_scene_def / process_binding
  + submit() / preview() / publish() / rollback()   (三态迁移，写 operation_log)
        ▲
        ├─ CustomFieldDef ──► GeneratedColumnCompiler（is_aggregatable→DDL）
        ├─ CustomViewDef ───► QueryOrchestrator（白名单+visibleOrgs注入）
        ├─ KpiDef ──────────► KpiEvaluator（formula DSL，复用统一口径）
        └─ ProcessBinding ──► FlowableBinder（发起时固化版本快照）

ConfigGovernanceService
  + assertScopeAndPermission(user, def)   // M8 + scope_level
  + previewSandbox(def, session)          // PREVIEW 仅本人会话
  + publishWithAudit(def)                 // EFFECTIVE + operation_log(哈希链)
```

### 11.2 配置发布时序（DRAFT→EFFECTIVE）
```
配置者→ConfigGovernanceService.submit(def)         [置 DRAFT]
配置者→preview(def)        → 沙箱仅本人会话渲染      [置 PREVIEW，不进生产]
配置者→验证通过→publish(def)
   ├─assertScopeAndPermission（M8 + scope_level，越权拒）
   ├─若 CustomFieldDef.is_aggregatable → GeneratedColumnCompiler 生成列 DDL（预览已确认）
   ├─置 EFFECTIVE + 新 version
   └─写 operation_log（前后值/操作人，入哈希链）
后续修订→不可改 EFFECTIVE，新建 version→PREVIEW→publish
```

### 11.3 自定义视图查询执行时序（隔离红线）
```
前端→GET /custom-views/{id}/data?filters=...
后端→QueryOrchestrator.build(viewDef, filters)
   ├─字段白名单校验（仅该 object_type 已登记列；拒任意列/子查询）
   ├─编译为参数化查询，注入 org_id = ANY(:visibleOrgs)
   └─在已 SET app.visible_orgs 的连接执行（RLS 兜底）
   └─结果按 is_sensitive 脱敏后返回
   ✗ 任一环缺 visibleOrgs / 走裸连接 → 拒绝执行（红线）
```

## 十三、评审重点
1. 自定义查询**无任何路径**绕过 `visibleOrgs`（隔离第一性）是否在编排器与 RLS 双层强制。
2. 生成列编译（H-04）与作用域前缀防碰撞是否覆盖集团/子公司同名场景。
3. 三态 + 版本固化 + PREVIEW 沙箱是否真正做到"改坏不污染生产、可回滚"。
4. 自定义 KPI 口径与平台统一 levelMatrix / risk_domain 一致，不另立阈值。
5. 自定义工作流版本快照对在途单据的隔离。
