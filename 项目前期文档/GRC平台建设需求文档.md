## 一、项目概述

### 1.1 建设背景

随着公司业务规模扩大及监管要求趋严（含人民银行非银行支付机构监管、ISO 27001、等保三级、个人信息保护法等多体系要求），公司在制度管理、风险评估、内部审计、法规跟踪及合规问答等方面的工作量持续上升，现有依赖人工与离散表格（Excel/Word/邮件）的管理方式已难以支撑，存在以下突出问题：

- 制度文件分散在不同部门、不同版本并存，缺乏统一检索与问答入口；
- 风险评估依赖人工填表、邮件流转，缺乏标准化打分模型与闭环跟踪机制；
- 内部审计的计划、实施、发现、整改分散管理，难以追溯审计历史与举证；
- 法律法规更新依赖人工关注，存在遗漏风险，且未与内部制度建立映射关系；
- 关键事件（法规更新、评估/审计进展、整改逾期）依赖人工口头或邮件提醒，时效性差、易遗漏，且无统一记录；
- 管理层缺乏统一的风险与合规态势视图，决策依赖人工汇总。

为此，拟建设统一的 GRC（Governance, Risk, Compliance）平台，整合制度管理、风险评估、审计管理、法规跟踪、智能问答与多通道通知能力，实现治理-风险-合规工作的标准化、流程化与数据化。

### 1.2 建设目标

- 建立公司统一的制度知识库，支持全文检索与智能问答；
- 建立标准化的风险评估闭环流程，从自评估到报告输出到整改跟踪全流程线上化；
- 建立内部审计全生命周期管理能力，支持审计发现与整改的留痕与举证；
- 建立法规跟踪与映射机制，确保法规更新能够及时识别并落地到制度修订；
- 建立基于大模型的合规智能问答能力，支持跨子公司、跨数据源的风险与合规查询（在权限受控前提下）；
- 建立统一的多通道通知机制，确保法规更新、评估/审计进展、整改逾期等关键事件能够及时、准确、可追溯地送达相关责任人；
- 为管理层提供风险与合规态势的统一看板视图。

### 1.3 适用范围

本平台适用于公司总部及各子公司的信息安全、合规、内审、风控、法务等部门，覆盖公司全部制度体系文件、信息系统资产、第三方供应商及业务流程的风险评估与审计活动。

### 1.4 名词术语

| 术语 | 说明 |
|---|---|
| GRC | Governance, Risk, Compliance，治理-风险-合规一体化管理 |
| 制度知识库 | 经结构化处理、支持全文检索与AI问答的制度文件集合 |
| 风险评估闭环 | 从自评表单填写、风险打分、报告生成到整改计划分配与验证的完整流程 |
| 资产/对象台账 | 统一登记的评估对象清单，包括信息系统、业务流程、第三方供应商等 |
| RAG | Retrieval-Augmented Generation，检索增强生成，AI问答的底层技术路径之一 |
| 数据隔离 | 确保不同子公司/部门数据在查询、问答场景下按权限严格隔离，不可越权访问 |
| 通知通道 | 用于送达通知的具体渠道，本平台首批采用企业微信机器人、飞书机器人、邮件、公司自有短信平台 |
| Webhook 机器人 | 企业微信、飞书等群机器人的一种接入方式，仅支持单向推送，不支持接收用户交互 |
| RBAC | Role-Based Access Control，基于角色的访问控制 |
| 统一控件库 | 将跨体系共性的控制项抽象为可复用原子控件的集合，供各体系评估/审计模板引用 |
| KRI | Key Risk Indicator，关键风险指标，用于阈值持续监控的量化指标 |
| 制度签署确认（Policy Attestation） | 制度发布后，要求相关人员在线确认已阅读知悉并留痕的机制 |
| AD | Active Directory，微软目录服务，本平台的用户身份与组织信息同步来源 |
| SLA | Service Level Agreement，服务等级协议，用于供应商履约水平的约定与跟踪 |

## 二、需求范围总览（V1.9 重写）

本平台需求划分为 **业务模块 + 横向支撑模块 + 系统管理模块** 三类。其中 M1（制度）、M2（风险评估）、M3（审计管理）、M4（法规跟踪）、M5（AI 智能问答）、M11（监管事项）为业务模块；M6（组织/资产台账）、M8（统一权限与审批流引擎）、M9（看板与留痕）、M10（统一通知中心）为横向支撑模块；M7（供应商）为业务模块并复用 M2 引擎；系统管理模块新增「模型接入与管理」与「建议与反馈（M-FB）」两项。

> V1.9 变更：① M3 由「内部审计」升级为「**审计管理**」，统一覆盖 内审 / 外审 / 监管检查 / 认证审计（以 `audit_type` 区分）；② 新增一级业务模块「**监管事项（M11）**」；③ 系统管理新增「**模型接入与管理**」「**建议与反馈（M-FB）**」两模块。

| 编号 | 模块名称 | 与其它模块的关系 |
|---|---|---|
| M1 | 制度体系文件管理与知识库问答 | 引用 M6 组织架构判定文件适用范围；问答权限受 M8 约束 |
| M2 | 风险评估闭环管理 | 评估对象引用 M6 台账；审批流程依托 M8；产生的事件供 M10 通知中心消费 |
| M3 | **审计管理（内审 / 外审 / 监管检查 / 认证审计）** | 检查表配置复用 M2 表单引擎；评估对象引用 M6 台账；证据存入 M9；外审对外回函闭环与认证有效期跟踪由本模块承担；产生的事件供 M10 消费 |
| M4 | 法律法规跟踪与采集 | 与 M1 制度条款建立映射关系；采集后 AI 自动分类与结构化排版；合规义务清单贯通 M1/M3/M9；产生的事件供 M10 消费 |
| M5 | AI 大模型智能问答 | 数据源覆盖 M1、M4、M2/M3/M11；权限隔离依托 M8；模型端点与 RAG 参数由「模型接入与管理」配置 |
| M6 | 组织架构与资产/对象台账 | 为 M1、M2、M3、M5、M10、M11 提供统一的组织与对象数据基础；承载资产合规属性与 ROPA |
| M7 | 第三方/供应商管理 | 复用 M2 的表单引擎与打分模型；供应商对象纳入 M6 台账；含技术安全 + DPA 维度与评估闭环 |
| M8 | 统一权限与审批流引擎 | 为各模块提供统一的权限判断与审批流支撑；含 UAR 闭环、SoD 冲突检测、权限变更审计；支持多域控身份集成 |
| M9 | 管理看板与证据留痕库 | 汇总 M2、M3、M11 的数据；下钻 / 趋势 / 跨子公司 benchmark；合规卷宗导出与端到端举证链；存储 M10 通知发送记录与平台操作审计日志 |
| M10 | 统一通知中心 | 被 M2、M3、M4、M11、M-FB 共用，承接各模块触发的通知事件，统一路由至各通知通道；含通知场景库、升级链、管理层聚合简报、站内待办 |
| M11 | **监管事项管理（新增）** | 报送日历（法定时限预警）/ 监管问询 / 处罚约谈台账 / 重大事件报送 / 年度合规计划；事件供 M10 消费，证据存入 M9 |

| M-FB（系统管理） | **建议与反馈（新增）** | 系统管理菜单 + AI 助手双入口收集 BUG / 功能建议，格式化 → 分类 → AI 可行性初评（人工复核为准）→ 脱敏审批后同步代码仓库 → M10 周期分析推送 |

>
> **菜单结构与顺序（CR-001-5）：**
> - **外部审计与内部审计为两个独立一级菜单（不合并）**：M3 在数据层为统一「审计管理」模块、以 `audit_type` 区分内审/外审/监管检查/认证，但**界面始终呈现为「外部审计」「内部审计」两个菜单**（外部审计涵盖外审/监管检查/认证）。
> - **业务菜单顺序**（由上到下）：**外部审计 → 内部审计 → 风险评估 → 法规跟踪 → 监管事项 → 制度体系 → AI 智能问答 → 第三方供应商**。
> - **系统管理菜单顺序**（由上到下）：**通知中心 → 模型接入 → 权限与审批 → 看板与留痕 → 建议与反馈 → 系统设置**。
> - 「合规态势总览」页签标题简化为「**合规态势**」；产品 UI 不出现实现/设计术语类说明文字（如四元组、Flowable、M2/M10 等内部代号、"复用工单机制"等流程叙述），此类属设计文档语言，不进产品界面。

## 三、模块一：制度体系文件管理与知识库问答

### 3.1 需求描述

实现公司所有制度体系文件（管理制度、操作规程、标准、模板等）的统一管理，包括发布后的员工签署确认闭环，并基于该文件库构建可供机器人问答使用的知识库。

### 3.2 功能需求

**3.2.1 文件全生命周期管理**

- 支持制度文件的起草、评审、签批、发布、修订、废止全流程线上化，并保留完整版本历史；
- 支持按制度体系分类（如 ISO 27001、等保三级、PIPL、PBOC 等）及组织层级（集团级/子公司级/部门级）建立目录结构；
- 支持制度文件之间的引用关系标注（如下位文件引用上位制度条款）；
- 支持生效日期、复审周期、责任部门、责任人等元数据管理，并在复审到期前自动提醒（提醒经 M10 通知中心送达）。

**3.2.2 制度签署确认（Policy Attestation）**

制度发布或修订后，"是否所有相关人员已阅读并确认"本身是合规证据的一部分（如 PIPL、ISO 27001 审计中常需证明员工已签署保密协议/隐私政策），制度管理不应止于"发布"，需补充确认闭环：

- 强制确认范围限定于核心制度（如员工行为准则、保密协议、隐私政策等全员适用且涉及监管强制要求的制度），技术规范、操作细则等专业性制度按需配置确认范围而非默认强制，避免"确认疲劳"导致流于形式；制度发布时支持按制度类型及其是否属于核心制度，配置对应的强制确认角色/部门/人员范围；
- 支持员工在线确认操作（已阅读并知悉），确认记录（确认人、确认时间、确认时的制度版本号）写入 M9 证据留痕库，作为留存证据；
- 逾期未确认的人员及统计数据经 M10 通知中心提醒本人及其上级，支持按部门/子公司汇总未确认率，供 M9 管理看板展示；
- 制度修订后，若变更内容触发重新确认条件（如核心条款变更），需支持重新发起确认流程，区别于仅元数据调整的小幅修订。

**3.2.3 知识库构建**

- 现有制度文件已全部为可编辑电子文档，知识库构建以直接解析现有文档结构为主，无需建设 OCR 识别能力；
- 支持对制度文件进行结构化切片（按章节/条款切分，保留条款编号与上下文，避免断章取义）；
- 支持多版本并存检索，问答时默认仅检索当前生效版本，历史版本需显式声明查询；
- 支持文件元数据（适用范围、生效状态、所属体系）作为检索过滤条件。

**3.2.4 全文检索与问答**

- 支持关键词全文检索及语义检索；
- 支持自然语言问答，回答需引用具体制度名称及条款编号，避免无依据的生成内容；
- 问答结果不在知识库覆盖范围内时，应明确告知用户而非编造答案。

### 3.3 非功能需求

- 制度文件检索响应时间应在 3 秒以内（基于公司当前文件规模评估）；
- 知识库内容更新（制度发布/修订后）应在合理时间窗口内（如 1 小时内）完成索引刷新；
- 废止文件应可配置为问答时不再引用，但保留查询入口供审计追溯。

### 3.4 关键风险与设计要点

- 切片策略是核心难点：简单按字数切分会破坏条款上下文，导致问答断章取义，引发合规风险，需按章节/条款边界切分；
- 需明确版本冲突处理规则：当新旧版本同时存在时，问答必须默认引用现行有效版本；
- 制度文件可能涉及内部敏感信息（如安全架构细节），知识库问答权限需与 M8 权限引擎联动；
- 签署确认范围若配置过宽（如所有制度强制全员确认），会导致员工产生"确认疲劳"而流于形式，需按制度类型与适用范围合理设置确认对象，而非默认全员强制。

## 四、模块二：风险评估闭环管理

### 4.1 需求描述

实现内部各类架构（信息系统、业务流程、第三方供应商等）风险评估的全流程线上化管理，覆盖自评估表单填写、自动化报告输出、整改计划安排与分配、整改验证闭环。

### 4.2 功能需求

**4.2.1 评估对象与评估计划**

- 评估对象统一引用 M6 资产/对象台账，支持按系统、流程、供应商、子公司等维度发起评估；
- 支持周期性评估计划（如年度等保测评、季度供应商评估）及临时性专项评估的创建与排期；
- 评估计划启动、阶段推进、临近截止等节点需触发 M10 通知中心提醒相关责任人。

**4.2.2 自评估表单引擎（模板可配置化）**

不同体系/法规要求对应的风险评估，差异不仅体现在题目本身，更体现在评估维度划分、打分逻辑与报告产出格式的结构性不同（如等保三级测评按"安全通用要求+安全扩展要求"分类并有明确控制点编号；ISO 27001 通常采用资产-威胁-脆弱性-可能性×影响的四要素模型；供应商风险评估则可能以合同条款+履约表现为主，不存在"控制点"概念）。因此表单引擎需支持**结构配置**与**打分/报告映射配置**两层分离的可配置能力，而非单一硬编码表单：

*表单结构配置层：*

- 支持题目层级结构自定义（域 → 控制项 → 具体问题），层级深度与命名（如"控制点编号"）可按体系要求配置；
- 支持多种题目类型：单选、多选、打分题（如 0-5 级成熟度）、文本说明、矩阵题、附件上传；
- 支持必填校验规则及跳题逻辑配置（如某题答"不适用"则自动跳过其关联的后续问题）；
- 支持表单填写过程中的附件上传（截图、证明材料）作为佐证；
- 支持表单的多级填写与提交流程配置（如系统负责人填写 → 部门负责人确认），流程节点经 M8 审批流引擎实现。

*打分与报告映射配置层：*

- 支持每道题/每个题目类别的权重配置；
- 打分方式默认采用定量模型（可能性 × 影响 = 风险值），便于跨评估对象、跨周期的横向对比统计及 M9 看板的聚合展示；对于供应商评估等部分维度难以量化的场景，在定量模型基础上叠加定性判断项（如合同履约情况）作为补充，而非完全切换为定性模式；定性成熟度等级（如 0-5 分）、简单合规判定（是/否/不适用）作为可选打分方式保留，供特定场景按需选用；
- 支持打分结果到报告章节的映射配置（如某控制项得分自动归入报告"异常处置"章节而非"网络安全"章节），确保自动化报告产出符合目标体系的章节结构要求；
- 结构配置与打分/报告映射配置两层解耦，避免后续调整报告格式时牵连表单本身，或调整表单字段时破坏既有打分逻辑。

*模板版本化与继承：*

- 模板需具备版本号与生效日期，已基于某版本启动的评估不受后续模板修订影响，沿用启动时的版本直至完成；
- 内置以下体系的基础模板作为脚手架：ISO 27001、等保三级、PBOC、PCI DSS、ISO 27701、ISO 27018、ISO 9001、供应商风险评估，支持用户基于既有模板克隆后做局部调整，而非每次从空白表单画起，兼顾配置灵活性与使用易用性；
- 支持平台管理员自定义添加新的体系分类（而非仅限于上述内置体系），新增体系后可从零搭建模板或基于已有模板克隆调整，确保公司未来新增认证体系（如新的行业认证、新的监管要求）时无需依赖开发改动即可纳入平台管理。

*统一控件库：*

不同体系的控制项之间存在大量重叠（如"访问控制""日志留存"等要求在 ISO 27001、等保三级、PBOC 体系中均存在但措辞不同），若各体系模板各自独立定义题目，会导致同一控制点在不同模板中重复定义、评估时重复测试同一对象。表单引擎需在结构配置层之上支持**统一控件库**能力：

- 将跨体系共性的控制项抽象为可复用的原子控件，抽象颗粒度按"具体控制点"而非笼统的"控制域"进行（如抽象为"特权账号定期审查"而非笼统的"访问控制"），体系模板引用控件库条目而非直接定义题目；
- 同一控件被多个体系评估后，支持"一次测试结果，多体系复用"，减少重复填表与重复举证的工作量；
- 控件库需支持按具体控制点颗粒度维护跨体系映射关系，确保新增体系模板时能够最大化复用既有控件；该颗粒度选择会带来更高的初始建设与维护投入，需配套设计控件维护工具降低长期维护成本。

*模板配置治理：*

- 采用"集中管控、分级开放"模式：模板的新建与结构性修改（题目层级、打分逻辑、报告章节映射等）归口信息安全部管理，纳入 M8 权限与审批流管控，避免合规底线被业务部门无意中破坏；
- 模板内的局部参数（如权重微调、提醒文案、个别题目的选项调整）开放给合规、内审等业务部门自助调整，调整权限同样由 M8 权限模型控制具体的可调整范围；
- 已发布模板的任何修改（无论是结构性修改还是局部参数调整）均需经"预览/测试填写"环节验证后才能正式生效，避免改动导致进行中的评估表单无法提交或打分逻辑出错。

**4.2.3 风险打分与报告自动生成**

- 报告自动生成基于 4.2.2 中配置的打分模型与报告章节映射规则，输出风险结论、风险点清单、等级分布统计图，无需为不同体系单独开发报告生成逻辑；
- 报告需支持人工复核与签批后才正式生效，避免完全依赖自动化结论；
- 报告产出格式（章节顺序、措辞规范）随模板版本固化，确保同一体系在不同评估周期产出格式一致，便于监管比对历史报告。

**4.2.4 整改计划与闭环跟踪**

- 风险点可一键转化为整改任务，支持责任人分配、计划完成时间、优先级设置；
- 整改任务状态跟踪（待处理/处理中/已完成/已逾期/已验证），任务分配、状态变更、逾期等事件经 M10 通知中心推送至责任人及其上级；
- 整改完成需上传证明材料并经发起人验证后才能关闭，验证记录写入 M9 证据留痕库；
- 支持整改任务的工单化管理，与 M3 审计发现共用同一整改跟踪机制，避免重复建设。

**4.2.5 关键风险指标（KRI）持续监控**

现有评估机制为"周期制"（年度/季度发起一次评估），存在的局限是风险在两次评估周期之间发生劣化时无法及时发现。需补充"持续监控制"作为周期性评估的补充：

- 支持配置关键风险指标（如日志留存天数、漏洞修复时效、第三方接口调用异常率等可系统化采集的指标）及对应阈值；
- 指标数据来源于公司已有的 SIEM、日志平台、漏洞管理系统的接口对接，而非依赖人工填报；
- 指标超过阈值时不必等待下次评估周期，直接触发 M10 通知中心预警相关责任人，预警记录纳入 M9 证据留痕库。

### 4.3 非功能需求

- 打分模型与等级矩阵需支持后台可配置调整，不应硬编码在程序逻辑中；风险等级统一采用**五级**（极低 / 低 / 中 / 高 / 极高，VERY_LOW / LOW / MID / HIGH / VERY_HIGH），`levelMatrix`（风险值→五级区间）后台可配，M9 聚合口径与之统一；
- 报告生成应支持导出为 Word/PDF 格式，满足对外提交监管或审计的格式要求；
- 整改任务的逾期统计需支持按部门、按子公司聚合，供 M9 看板调用，并作为 M10 通知分级规则的触发依据。

### 4.4 关键风险与设计要点

- "自动化输出报告结论"的价值高度依赖打分规则的结构化程度，若评估标准模糊，自动化将退化为简单模板填充；
- 整改计划分配建议复用工单类系统的成熟模型（状态机、提醒机制），不建议重新设计一套独立逻辑；
- 表单引擎若做得"过于通用"，容易退化为万能表单生成器而牺牲易用性（运营人员不知道如何配出一份合规的等保测评表），需以"通用配置能力 + 内置体系脚手架"两者结合解决，不可偏废一方；
- 表单结构配置与打分/报告映射配置必须分层解耦，避免两者耦合导致后续维护困难；
- 模板配置入口的修改权限与发布前测试机制需一并设计，避免已发布模板被误改导致评估中断；
- 控件库按"具体控制点"颗粒度抽象，相比按"控制域"抽象有更高的初始建设与维护投入，需配套设计控件维护工具及跨体系映射关系的批量管理能力，降低长期维护成本；
- KRI 持续监控依赖与 SIEM、日志平台、漏洞管理系统的接口对接，具体对接的指标范围及接口规范需在设计阶段与相关系统的运维团队共同确认。

### 4.5 风险评估·补（V1.9 对齐 · CR-001-7 / CR-002）

**4.5.1 风险等级五级化（CR-001-7）**

- 风险评估等级由原 3 级（高 / 中 / 低）统一改为**五级（由低到高）：极低、低、中、高、极高（VERY_LOW / LOW / MID / HIGH / VERY_HIGH）**。原 3 级为既往实现选择，需求未硬性固定级数，现按业务要求固化为五级；
- 打分模型 `levelMatrix`（风险值→五级区间）、`risk_level` 枚举、看板/标签/分布图的等级口径同步统一；M9 聚合口径与本五级口径一致，避免不同子公司口径不一致导致汇总失真（呼应 11.4）。

**4.5.2 资产-威胁-脆弱性（A-T-V）关联建模**

- 评估支持「资产-威胁-脆弱性」关联建模，将 ISO 27001 / 等保的四要素（资产-威胁-脆弱性-可能性×影响）在填报中落地：每条风险由「资产 + 威胁 + 脆弱性」三元关联生成，可能性与影响在此关联上打分得出风险值；
- A-T-V 模型与 4.2.2 表单引擎结构配置兼容，作为风险评估的可选建模方式，供 ISO/等保类评估按需启用。

**4.5.3 风险处置计划 + 残余风险 + 管理层接受签字**

- 评估报告新增**风险处置决策**维度，每条风险点可选处置策略：**降低 / 接受 / 转移 / 规避**；
- 记录处置后的**残余风险**等级（按五级口径），并对「接受」类残余风险要求**管理层接受签字留痕**（接受人、接受时间、接受时的风险版本），写入 M9 证据留痕库作为举证材料。

**4.5.4 综合风险指数可解释 / 可下钻**

- 综合风险指数须可解释、可下钻：展示**指数构成**（各风险域贡献）、**各风险域权重**、**环比归因**（较上期升降的来源）、**计算口径说明**；
- 看板/热力矩阵单元格支持下钻至明细，并叠加趋势箭头与时间对比（呼应 §九 M9）。

## 五、模块三：审计管理（内审 / 外审 / 监管检查 / 认证审计）

### 5.1 需求描述

实现公司各类审计活动的在线实施与管理。本模块由原「内部审计」升级为统一「**审计管理**」，以**审计类型（`audit_type`）**区分四类：**内部审计 / 外部审计（ISO、PCI、ISO 27701 等认证机构审计）/ 监管检查（PBOC、网信办等）/ 认证审计**，统一管理审计计划、检查表执行、问题发现、整改跟踪、对外回函与闭环、认证有效期及报告输出。

> V1.9 升级（CR-001-1 / CR-002）：除内审外，新增外审 / 监管检查 / 认证审计的全闭环管理，含外审对外回函三段漏斗、认证有效期跟踪、等保测评全过程视图。详见 5.5「审计管理·补」。

### 5.2 功能需求

**5.2.1 审计计划与执行**

- 支持年度审计计划的制定与排期，关联 M6 审计对象台账；
- 支持现场/线上审计过程中的问题发现记录，包括问题描述、依据条款、严重程度评级、证据材料上传；
- 审计发现的整改复用 M2 的整改任务与工单机制，避免重复建设两套整改流程；
- 支持历史审计记录的检索与趋势分析（如同一问题是否反复出现）；
- 审计计划启动、发现项创建、整改逾期等关键节点经 M10 通知中心推送至相关责任人与审计负责人。

**5.2.2 审计检查表模板配置（复用 M2 表单引擎）**

不同体系/法规对应的审计检查表（内审/外审/监管检查/认证审计），同样存在维度划分、评分逻辑、报告格式的结构性差异（如等保测评检查表按控制点编号组织，PBOC 合规审计检查表的章节结构与等保不同），因此审计检查表的配置需求与 M2 风险评估表单的需求本质相同，**应直接复用 4.2.2 所述的表单引擎（结构配置层 + 打分/报告映射配置层、模板版本化与继承、统一控件库、内置体系脚手架、配置治理机制），而非为审计单独建设第二套表单配置能力**。在复用基础上，审计检查表场景需补充以下差异化要求：

- 支持检查表题目直接引用 M1 制度知识库中的具体条款作为检查依据，引用关系在生成审计报告时自动带出对应条款编号，避免人工逐条核对；
- 支持检查表题目与严重程度评级的关联配置（如某检查项判定不合规时自动建议的严重程度等级，供审计人员参考调整而非强制覆盖人工判断）；
- 内置脚手架在 4.2.2 通用体系模板基础上，补充审计场景特有的模板（如等保测评检查表、内部合规审计检查表），与风险评估模板共享同一套模板管理后台，按"用途"字段（风险评估/内部审计）区分而非另建独立配置入口。

**5.2.3 审计报告输出**

- 支持审计报告的自动生成（基于发现项汇总及检查表打分/映射规则），复用 4.2.3 所述的报告生成机制；
- 支持人工复核签批后正式生效；
- 报告产出格式随模板版本固化，确保同一审计类型在不同周期产出格式一致。

### 5.3 非功能需求

- 审计过程数据（检查表填写记录、发现项、整改记录）需具备防篡改的留痕能力，满足监管检查举证要求；
- 审计报告需支持按不同监管要求（如等保测评报告、PBOC 合规审计报告）输出对应格式。

### 5.4 关键风险与设计要点

- 审计与风险评估（M2）在流程本质上高度相似（检查表填写→发现问题→整改跟踪），需明确共用工作流引擎、表单引擎与整改模块，仅在模板内容与报告格式上做差异化，避免重复造两套配置能力；
- 若审计检查表配置与 M2 风险评估表单分别建设，将导致同一类"模板版本管理、权限治理、预览测试"机制被实现两次，且后续两边可能逐渐出现能力不一致，需在技术选型上明确统一；
- 审计发现的证据材料需要独立、可追溯的存储机制，与 M9 证据留痕库统一设计。

### 5.5 审计管理·补（V1.9 对齐 · CR-001-1 / CR-002）

**5.5.1 审计类型与组织隔离**

- 以 `audit_type` 区分四类审计：**内部审计 / 外部审计 / 监管检查 / 认证审计**；
- 登记外审任务时记录：认证体系、外部审计机构 / 监管机构、责任单位、周期；
- **组织隔离（关键）：** 各子公司管理本子公司各认证体系的外部审计，集团管理集团级认证；数据按组织严格隔离、权限分开（沿用 M8 权限模型 + 组织隔离）。

**5.5.2 外审对外回函三段漏斗（闭环）**

- 录入外部机构的审计发现（来源、问题、依据、严重度、责任单位），整改复用 M2 工单机制（责任人分配、状态跟踪、逾期升级）；
- 对外回函/闭环采用**三段漏斗**：**已提交外部机构 → 外方受理 → 外方确认关闭**，每段记录提交时间、回函状态与凭证，区别于仅内部关闭；
- 整改状态经 M10 通知责任人，期限临近 / 逾期自动提醒并可升级上级。

**5.5.3 认证有效期跟踪**

- 跟踪各认证体系证书的有效期与到期提醒，临近到期触发 M10 提醒，驱动复审 / 再认证审计排期。

**5.5.4 等保测评全过程视图**

- 在审计管理下提供**等保测评全过程**专项视图：测评进度、差距项、整改进度、到期/复测节点，覆盖测评从启动到出具结论的全过程。

**5.5.5 审计覆盖率与管理层回应**

- 提供审计覆盖率 / 盲区分析（哪些对象、控制点已被审计覆盖、哪些尚未覆盖）；
- 审计发现支持记录**管理层回应 / 接受风险**（管理层对审计发现的处置意见或风险接受留痕）。

## 六、模块四：法律法规跟踪与采集

### 6.1 需求描述

建立模块配置跟踪或采集指定权威网站（如中国人民银行、国家网信办、金融监管总局等）发布的最新法律法规、规范性文件或监管要求，并支持与内部制度的映射关联。

### 6.2 功能需求

- 支持配置需跟踪的权威网站/信息源清单，按更新频率自动检测新增内容；
- 新法规发布或既有法规条款修订后，自动生成变更摘要，并经 M10 通知中心推送至相关责任部门（如合规部、信息安全部）；
- 支持法规条款与 M1 内部制度条款的映射关系登记，便于判断法规更新后哪些制度条款需要修订（逐条合规核查能力）；
- 支持法规库的检索与历史版本管理。

### 6.3 技术路线

- 采用自建爬虫方式跟踪指定权威网站（如中国人民银行、国家网信办、金融监管总局等）发布的内容；
- 需考虑目标网站改版导致爬虫失效的维护成本，建立失效检测与告警机制（爬虫连续多次未抓取到预期内容时触发 M10 通知提醒运维人员排查），避免长期失效而未被察觉；
- 需评估目标网站是否存在反爬限制或使用条款限制，确保采集方式的合规与技术可持续性；
- 支持按网站结构变化快速调整采集规则的配置化能力，降低每次网站改版后的修复成本。

### 6.4 关键风险与设计要点

- 自建爬虫的核心风险在于网站改版导致的失效维护成本及潜在的访问限制问题，需建立常态化的爬虫健康监控机制，而非一次性开发后不再维护；
- 通知内容颗粒度需精细化：应说明"变更了什么"（条款级摘要）而非简单提示"有更新"，否则责任人仍需自行比对新旧版本，可结合 M5 AI 能力做变更点提取后再推送。

### 6.5 法规跟踪·补（V1.9 对齐 · CR-001-6 / CR-002）

**6.5.1 采集后 AI 自动分类与结构化排版（CR-001-6）**

- 新增追踪源后，采集内容**自动按体系/主题分类**并**结构化排版**到统一卡片流：结合 AI 分类与**条款级摘要**，降低人工整理成本；
- 自动分类结果支持人工校正，分类维度可配置（呼应高自由度可配置原则）。

**6.5.2 合规义务清单（Obligation Register · CR-002）**

- 法规跟踪下新增**合规义务清单**，建立端到端链路：**法规条款 → 合规义务 → 责任人 → 关联制度 / 控制点 → 证据**；
- 义务清单作为「一条法规看到底」的载体，向下贯通至 M2/M3 评估/审计与整改、M9 证据留痕，形成**端到端举证链**（法规→义务→制度→控制点→评估/审计→整改→证据）。

## 七、模块五：AI 大模型智能问答

### 7.1 需求描述

提供基于大模型的智能问答能力，支持就最新法律法规、各子公司风险评估记录、审计发现等内容进行自然语言问答。

### 7.2 功能需求

- 问答数据源覆盖 M1 制度知识库、M4 法规库、M2/M3 风险评估及审计记录；
- 支持引用溯源，回答需标注数据来源（具体制度条款、具体评估记录编号），不得无依据生成内容；
- 支持多轮对话与上下文理解，适配合规人员的复杂查询场景（如"对比A、B子公司近两次评估的高风险项变化"）。

### 7.3 权限与数据隔离要求（核心约束）

本模块的核心难点是权限控制而非问答技术本身，子公司风险评估记录通常涉及敏感经营数据，必须满足以下要求：

- 问答引擎在检索阶段必须按用户角色与所属组织进行数据过滤，杜绝跨子公司越权访问（如 A 子公司员工不可通过问答获取 B 子公司的风险评估数据）；
- 权限模型需与 M8 统一权限引擎对接，而非在 AI 问答模块内单独实现一套权限逻辑；
- 数据模型设计阶段必须确定组织维度的隔离边界，后期补救改造成本极高，需优先评审。

### 7.4 非功能需求

- 问答内容涉及敏感数据时，需有访问日志记录，支持事后审计谁查询了什么数据；
- 问答准确性需建立验证机制（如抽样人工复核），避免大模型幻觉导致的合规误导。

### 7.5 AI 问答·补（V1.9 对齐 · CR-002 / CR-004）

**7.5.1 生成可报送 / 可汇报材料**

- AI 问答能力扩展至**生成可报送/可汇报材料**（如监管报送稿、管理层合规简报、评估/审计汇总），生成结果须经人工复核后使用；
- 回答与生成材料须**标注数据时点**（数据截止时间），避免基于过期数据形成结论或对外报送。

**7.5.2 与「模型接入与管理」的关系**

- M5 的对话/embedding 模型、端点、RAG 参数（召回数、相似度阈值、索引刷新策略）、系统提示词、提示注入防护、「数据不出域」开关等，统一由系统管理下的「**模型接入与管理**」模块配置（详见 §十二·B）；
- M5 仅消费该模块的配置，不在问答模块内单独维护模型接入与密钥；模型调用范围受 M8 权限与组织隔离约束（数据访问限定在用户可见组织范围）。

## 八、模块六：组织架构与资产/对象台账

### 8.1 需求描述

建立统一的组织架构、信息系统、业务流程及评估对象清单，作为 M2 风险评估、M3 审计、M5 智能问答、M10 通知路由的共用数据底座，避免各模块各自维护对象列表导致数据无法关联。

### 8.2 功能需求

- 支持公司及各子公司组织架构的登记与维护，作为权限隔离（M8）与通知路由（M10）的判断基础；
- 组织架构中的用户身份信息（账号、所属部门、所属子公司）通过对接 AD（Active Directory）**多域控/多身份源**同步获取（各子公司含海外可各自运行独立 AD 域/林，详见 §13.7 多域控身份集成 · CR-003），不在本平台独立维护用户账号体系，也不与 OA 审批系统打通审批流程；
- 支持信息系统、业务流程、数据资产的统一台账登记，包括责任人、所属子公司、重要性等级等属性，该部分数据由平台内自行登记维护，暂不对接 CMDB；
- 台账数据作为 M2/M3 评估对象选择的唯一来源，确保评估记录可追溯至具体资产。

### 8.3 关键风险与设计要点

- 本模块是 M2、M3、M5、M8、M10 共用的数据基础，若数据模型设计存在缺陷（如组织层级、对象属性定义不完整），将导致依赖该模块的其它需求出现数据关联缺失或返工；
- 与 AD 的同步频率及账号变更（如离职、调岗）的同步时效需明确定义，避免权限引擎依据过期的组织信息做出错误的数据隔离判断。

### 8.4 组织/资产·补（V1.9 对齐 · CR-002）

**8.4.1 资产合规属性扩展**

- 资产/对象台账维度深化，新增合规属性：**数据分级、是否含个人信息、是否涉及跨境、等保定级（含定级结论 / 备案证明 / 测评周期）、CIA（保密性/完整性/可用性）评级、是否持卡人数据（PCI 范围）、所属网络区域**；
- 资产合规属性作为 M2 风险评估、M3 审计、监管报送的判定依据（如等保定级驱动等保测评，跨境/个人信息驱动 PIPL 合规义务）。

**8.4.2 个人信息处理活动记录（ROPA · PIPL 法定台账）**

- 新增**个人信息处理活动记录（ROPA）**作为 PIPL 法定台账，登记处理目的、处理方式、个人信息类别、接收方、跨境情况、保存期限等，可挂在资产/数据资产维度；
- ROPA 与资产合规属性、合规义务清单联动，支撑 PIPL 合规举证。

## 九、模块七：第三方/供应商管理

### 9.1 需求描述

实现第三方/供应商安全管理的统一规划，覆盖供应商全生命周期的风险评估、监控与跟踪。

### 9.2 功能需求

- 支持供应商全生命周期管理（准入评估、合同期内管理、续约评估、退出评估）；
- 支持供应商风险评估，复用 M2 的表单引擎与打分模型，区分供应商专用评估模板；
- 支持 SLA 跟踪与到期提醒，提醒经 M10 通知中心送达对应负责人；
- 支持事件触发的非计划性复评：供应商风险不应仅在准入或续约节点评估一次，需支持配置外部事件（如供应商被曝数据泄露、信用评级下调、监管处罚）触发非计划性复评任务，复评任务复用 M2 整改与跟踪机制；
- 外部事件来源支持人工登记（如安全团队获悉供应商负面舆情后手动发起复评），同时支持对接外部数据源（如企业信用查询接口）实现自动触发；
- 供应商对象纳入 M6 统一台账管理。

### 9.3 供应商管理·补（V1.9 对齐 · CR-002）

**9.3.1 技术安全 + DPA 评估维度**

- 供应商评估在原合同/履约维度外，扩展**技术安全评估维度**：数据驻留地、PCI 范围、供应商自身持有的认证、与本公司连接的最小权限；
- 扩展**合规维度**：**DPA（数据处理协议）、资质、跨境、持卡人数据、再委托**等，支撑 PIPL/PCI 等合规要求。

**9.3.2 评估闭环（准入/年度/续约/事件复评）**

- 供应商评估按**评估类型（kind）**区分：**首次准入 / 年度定期 / 续约 / 事件复评**，统一进入闭环：**评估 → 发现 → 整改 → 闭环漏斗**；
- 配套**定期评估计划**（如年度定期评估排期），到期触发 M10 提醒；事件复评沿用 9.2 的外部事件触发机制，复评同样进入上述闭环。

## 十、模块八：统一权限与审批流引擎

### 10.1 需求描述

建立贯穿 M1 至 M5、M10 的统一权限模型与多级审批流引擎，作为制度发布、评估报告确认、整改计划签批、通知接收范围判定等场景的共用基础设施，而非在各模块内分散实现。

### 10.2 角色体系

权限引擎的"基于角色（RBAC）"需要先有明确的角色清单作为依据，否则权限设计无据可依。结合公司组织架构（存在需要跨子公司汇总查看数据的集团层面角色）及各模块功能需求中出现的职责，按功能域梳理出以下角色体系，角色颗粒度采取细分原则——同一职责域内按"决策权限层级"与"具体业务范围"两个维度进一步拆分，而非粗放地用单一角色覆盖整个职责域：

> V1.9 说明：以下 20 个角色为**基础集**，角色清单**随新模块扩展不固定数量**（如监管事项 M11、模型接入与管理、建议与反馈 M-FB 等新模块的配置/处置角色按需新增）。角色支持自定义新增、停用与权限范围调整（呼应高自由度可配置原则）。新模块涉及的典型角色：监管事项处理人（M11）、模型接入管理员、反馈处置/出站审批管理员（M-FB）。

*平台与基础设施类角色：*

| 角色 | 职责范围 | 涉及模块 |
|---|---|---|
| 平台超级管理员 | 系统级配置（权限模型本身的维护、角色定义变更、平台全局参数），具备最高配置权限 | M8 |
| 通知规则管理员 | 配置 M10 通知规则、通道接入与降噪策略，不涉及业务数据本身 | M10 |
| 组织与台账管理员 | 维护 M6 组织架构、资产/对象台账数据及与 AD 的用户同步 | M6 |

*制度管理类角色：*

| 角色 | 职责范围 | 涉及模块 |
|---|---|---|
| 制度起草人 | 起草、修订制度文件草稿，提交评审，不具备发布权限 | M1 |
| 制度发布审批人 | 审批制度发布/修订/废止，配置签署确认范围 | M1、M8 |
| 普通员工（制度签署方） | 阅读制度、完成签署确认 | M1 |

*风险评估与整改类角色：*

### 10.4 功能需求

- 支持基于角色（RBAC）及组织归属的细粒度权限控制，权限判断需精确到"用户-角色-组织-数据"四元组，角色体系以 10.2 所列角色为基础；
- 支持同一用户同时拥有多个角色（如某人既是系统/业务责任人又是部门级评估审核人），权限取多角色权限的合并结果；
- 支持角色与组织归属的组合判断，确保同一角色在不同子公司下的数据可见范围相互隔离（如"系统/业务责任人"角色在 A 子公司与 B 子公司下分别只能看到本子公司数据），集团管理层角色除外；
- 用户账号与基础身份信息通过对接 AD（Active Directory）同步获取，权限引擎在此基础上叠加角色与数据范围的判断逻辑，不通过本平台独立维护用户账号体系；
- 支持可配置的多级审批流（如部门级评估审核人 → 信息安全负责人（评估侧） → 集团管理层），适配制度发布、风险评估报告确认、整改计划签批等不同审批场景；
- 审批流需支持加签、转办、超时提醒等常见审批操作（超时提醒经 M10 通知中心送达）；
- 权限引擎需作为 M5 智能问答数据隔离及 M10 通知路由的底层依据，确保"一套权限模型，全平台复用"。

### 10.5 关键风险与设计要点

- 角色体系按细分原则设计，需建立配套的角色管理界面支持角色的新增、停用、权限范围调整，避免角色数量增多后权限矩阵难以维护；
- 一人多角色叠加权限的合并规则（取并集还是按场景区分）需在设计阶段明确定义，避免权限判断出现歧义；
- 角色与组织归属的组合判断逻辑是 M5 智能问答数据隔离的直接依据，若该逻辑在设计阶段不清晰，将直接影响 M5 上线后的数据安全性；
- 与 AD 的用户同步需明确同步频率与异常处理机制（如 AD 中用户离职后，平台侧角色与权限需及时回收）。

### 10.6 权限引擎·补（V1.9 对齐 · CR-002 / CR-003）

**10.6.1 访问权限审阅（UAR）闭环**

- 支持**定期权限复核活动（UAR）**：按周期发起权限复核，记录谁复核、对每项权限的决定（保留 / 回收）、复核签字与证据，形成闭环；复核结果驱动权限调整并留痕。

**10.6.2 职责分离（SoD）冲突检测与权限变更审计**

- 支持**职责分离（SoD）冲突检测**：识别同一用户持有相互冲突的角色/权限组合并告警；
- 所有**权限变更**（授权、回收、角色调整）均纳入审计，写入平台操作审计日志（不可篡改）。

**10.6.3 越权 / 特权账号明细下钻**

- KRI「越权拦截 / 特权账号未审查」等指标支持**下钻到账号 / 系统 / 时间**明细，供安全负责人核查。

**10.6.4 多域控身份集成（CR-003，详见 §13.7）**

- 权限引擎支持**多域控（多身份源）**：账号唯一键由全局 `username` 改为 **(domain_id, username)** 复合唯一；登录采用 **DOMAIN\\user 或 user@domain** 认证路由；各子公司管理本域用户与角色，集团聚合跨域视图；**不依赖域间信任**；单域故障仅影响该域，其余域照常并维持权限快照。

## 十一、模块九：管理看板与证据留痕库

### 11.1 需求描述

为管理层提供风险与合规态势的聚合可视化视图，并建立审计发现、整改证明材料的独立留痕与证据存储机制，满足监管检查举证要求。

### 11.2 功能需求

**11.2.1 管理看板**

- 风险态势总览（按子公司、按业务线的风险等级分布）；
- 整改完成率、逾期率统计；
- 审计发现趋势分析（同类问题是否反复出现）；
- 通知送达情况统计（如法规更新通知的阅读/响应情况），供管理层评估通知机制有效性；
- 看板数据权限同样受 M8 权限引擎约束，管理层可见范围按其管理职责配置。

**11.2.2 证据留痕库**

- 审计发现、整改证明材料（截图、文档、邮件记录等）需独立存储，具备防篡改特性（如哈希校验、操作日志不可删除）；
- 证据材料需与具体评估记录/审计发现建立唯一关联，支持按记录追溯调取；
- M10 通知中心的发送记录（送达对象、时间、通道、内容、是否成功）同步写入留痕库，作为"是否及时通知相关责任人"的监管举证材料；
- 留痕库的保留期限按 PBOC、等保三级等不同监管口径中要求的最长期限执行（取各项要求的最大值），确保同一份记录能同时满足多个监管口径的留存要求，避免因体系差异导致部分记录提前清除而无法满足其中某一监管要求；具体期限数值需在各体系适用的监管条款中逐一核实后固化为平台配置参数。

### 11.3 非功能需求

- 看板数据应支持按时间维度（如季度、年度）和组织维度（子公司、部门）灵活筛选与下钻；
- 看板统计数据的刷新时效需明确（如准实时或按日汇总），并在界面标注数据截止时间，避免管理层基于过期数据决策；
- 证据留痕库的存储需考虑附件类材料（截图、文档）的容量增长，明确存储扩容方案与单文件大小限制；
- 证据材料的防篡改机制（如哈希校验）需在材料入库时即生成校验值，并支持事后完整性校验。

### 11.4 关键风险与设计要点

- 看板的聚合统计依赖 M2、M3 各模块数据的结构化程度，若上游数据口径不统一（如不同子公司风险等级定义不一致），将导致汇总数据失真，需在 M2 打分模型设计阶段即统一口径；
- 证据留痕库是监管检查举证的核心，其防篡改与不可删除特性需在技术方案中明确实现机制（如写入即只读、操作日志独立存储），不能仅依赖应用层逻辑控制；
- 留存期限到期后的数据处理方式（自动归档还是人工确认后清除）需明确，避免到期数据被自动清除后又因监管延期检查而无法提供。

### 11.5 看板与留痕·补（V1.9 对齐 · CR-002）

**11.5.1 下钻、趋势/基线与跨子公司 benchmark**

- 关键指标与 KRI 增**历史趋势（sparkline）+ 基线**，看板/热力矩阵单元格支持**下钻至明细**并叠加趋势箭头与时间对比；
- 风险等级口径统一为五级（呼应 4.5.1），消除子公司口径不一致导致的汇总失真；
- 支持**跨子公司同口径对比 benchmark**：按法规/控制点的子公司横向对比。

**11.5.2 合规卷宗导出与证据反向取证**

- 支持**合规卷宗一键导出**：按检查/项目打包导出，含**清单 + 校验值（sha256）**；
- 支持**证据 ↔ 法规条款反向取证**：由法规条款反查关联证据，与 §6.5.2 合规义务清单共同构成**端到端举证链**（法规→义务→制度→控制点→评估/审计→整改→证据）。

**11.5.3 平台操作审计日志与防篡改链**

- 建立**平台自身操作审计日志（audit trail）**：改阈值、导出、改制度、权限变更、对外推送等关键操作均留痕，**不可篡改**，提供管理界面查询；
- 证据留痕增**可信时间戳（TSA）/ 防篡改链**说明：材料入库即生成校验值与可信时间源标注，支持事后完整性校验。

**11.5.4 跨模块全文检索**

- 顶部全文检索须**可用且跨模块溯源**，覆盖法规 / 制度 / 评估 / 审计 / 整改 / 证据，检索结果可定位回源记录。

## 十二、模块十：统一通知中心

### 12.1 需求描述

建立统一的多通道通知中心，作为 M2（风险评估）、M3（内部审计）、M4（法规跟踪）等业务模块的共用横向基础设施，承接各类事件的通知触发、规则判定、分级降噪与多通道送达，避免各模块各自接入通知通道、各自维护通知逻辑导致的重复建设与不一致体验。原始需求场景包括：

1. 可配置多个通知通道（如企业微信机器人），用于通知监管法规更新；
2. 可配置指定机器人用于通知或跟踪风险评估与内部审计相关进展。

### 12.2 功能需求

**12.2.1 通知通道管理**

- 支持多通道接入与统一管理，首批覆盖以下四类通道：企业微信群机器人（Webhook）、飞书群机器人、邮件、公司自有短信平台（通过对接公司现有短信网关实现）；
- 所有通道均以单向推送为目标，不要求接收用户在通道内的交互式回复（如点击按钮处理任务），企业微信与飞书均采用 Webhook 群机器人接入方式即可满足需求，无需评估自建应用；
- 支持同一事件按规则同时分发至多个通道（如重要通知同时推送企业微信/飞书 + 落一份邮件作为留痕备份）；
- 支持通道的启用/停用、测试发送、连通性检测；
- 企业微信、飞书群机器人均需考虑各自平台的单条消息长度限制及推送频率限制，避免因超限被限流或封禁；机器人被移出群、Webhook 地址失效等异常情况需被检测并告警；
- 短信通道因存在发送成本，建议仅用于紧急级通知，由通知规则引擎按级别控制是否启用短信通道，避免不必要的成本支出。

**12.2.2 通知规则引擎**

- 通知触发条件（事件类型、严重级别、目标对象、对应通道）需支持后台可配置，不得硬编码在程序逻辑中；
- 典型规则示例："高风险整改项逾期 3 天 → 推送至该系统责任人企业微信/飞书 + 其直属上级邮件"；
- 规则配置需与 M6 组织架构、M8 权限模型联动，确保通知仅送达对应子公司、对应职责范围内的责任人，不发生跨子公司广播；
- 支持按事件来源模块（M2/M3/M4）分别配置独立的规则集，同时支持平台管理员统一管理全部规则。

**12.2.3 通知分级与降噪**

- 通知按紧急程度分级（如紧急/重要/普通），不同级别可配置不同的送达时效要求及通道组合（如紧急级可叠加短信通道，普通级仅用企业微信/飞书）；
- 支持免打扰时段配置（如非工作时间仅普通级通知延迟发送，紧急级通知不受限）；
- 支持同一时间窗口内同类通知的合并推送，避免短时间内重复事件导致同一群组被刷屏；
- 支持用户级通知订阅偏好设置（如个人可自主订阅/取消关注特定类型事件，紧急级通知不可取消订阅）。

**12.2.4 通知内容**

- 法规更新类通知应附带变更摘要（说明具体变更条款及内容，而非仅提示"有更新"），可结合 M5 AI 能力自动提取变更点；
- 评估/审计进展类通知应包含可定位的关键信息（如任务编号、所属系统、责任人、截止时间），支持点击跳转直达任务详情页（跳转为外链方式打开平台页面，而非在通知通道内完成操作）。

**12.2.5 发送可靠性与留痕**

- 通知发送失败（如 Webhook 超时、网络异常、短信网关调用失败）需具备重试机制，多次失败后告警通知管理员；
- 所有通知记录（触发事件、目标对象、通道、内容、发送时间、发送结果）需完整留痕，写入 M9 证据留痕库，作为"是否及时通知相关责任人"的监管举证材料。

### 12.3 非功能需求

- 通知触发到送达的时延应控制在合理范围内（如紧急级通知 1 分钟内送达），具体时效指标需结合各通道实际能力确认；
- 通知规则引擎需支持平台运营人员自助配置新规则，无需研发介入；
- 通知发送记录的查询与统计能力需支撑 M9 看板的"通知送达情况"展示。

### 12.4 关键风险与设计要点

- 本模块不应被理解为单一的"发消息"功能，而需要通道管理、规则引擎、分级降噪三层设计，否则后续新增通知场景（如新增一个事件类型）将需要修改代码而非配置规则；
- 必须与 M6（组织架构）、M8（权限模型）联动，否则存在通知误发、跨子公司越权可见的风险；
- 短信通道涉及调用公司自有短信平台的发送接口与额度，需提前确认接口规范及调用额度限制，避免高频通知场景下超出额度。

### 12.5 通知中心·补（V1.9 对齐 · CR-001-4 / CR-002）

**12.5.1 通知场景库 + 升级链（CR-001-4）**

- 接收人涉及各种角色与管理层级，通知内容与场景丰富且难以一次穷尽，故按**可配置场景库**实现：**场景 → 触发条件 → 接收角色/层级（精细路由）→ 可编辑消息模板（含变量、跳转）**，新增场景无需改码；
- 代表场景：整改期限临近 → 责任人；整改逾期 → 责任人 + 上级（**升级链**）；内/外审或评估发现概况 → 子公司负责人；法规变化 → 该子公司合规对接人；KRI 预警；制度待签署；**管理层合规简报（聚合，不逐条打扰）**；反馈与建议周期分析（见 §十二·B）；
- 通道仍为锁定的 4 类（企微 / 飞书 / 邮件 / 短信，**单向推送**），本项丰富的是内容与触达精细度，并不改变通道单向性质。

**12.5.2 站内待办中心（含 SLA / 法定时限）**

- 新增**站内待办中心**作为站内触达：聚合各模块待办（评估、整改、签署、审计、监管报送等）；
- 待办支持 **SLA / 法定时限单列预警**（法定时限项单独高亮排序）、**批量处理**与**委派/代理**。

## 十二·A、模块十一：监管事项管理（M11）【新增 · V1.9 · CR-002】

### M11.1 模块目标

将合规对外的监管互动统一台账化、日历化、闭环化，覆盖**报送日历（法定时限预警）/ 监管问询 / 处罚约谈台账 / 重大事件报送 / 年度合规计划**，确保法定报送不逾期、监管互动可追溯。

### M11.2 功能需求

- **报送日历（法定时限预警）：** 维护各项法定/约定报送事项及其法定时限，按时限自动预警（经 M10 站内待办与通知，法定时限单列高亮排序）；
- **监管问询：** 登记监管机构问询事项、答复要求、答复内容与时限，跟踪答复闭环；
- **处罚 / 约谈台账：** 登记监管处罚、约谈事项、整改要求与整改闭环；
- **重大事件监管报送：** 支持重大事件（如支付机构重大事件须按时限报央行）的监管报送登记与时限跟踪；
- **年度合规计划：** 计划 → 执行 → 年末对照，跟踪年度合规工作完成情况。

### M11.3 边界

- **完整安全事件管理（IR / SOC）不在本平台**，IR/SOC 主体在安全运营侧；本平台仅承接「**重大事件的监管报送**」（合规职责）；
- 监管事项产生的事件供 M10 消费，证据/报送记录存入 M9 留痕库，受 M8 权限与组织隔离约束。

## 十二·B、新增模块：模型接入与管理 / 建议与反馈（M-FB）【新增 · V1.9 · CR-004】

> 本章两模块归属**系统管理**。「模型接入与管理」为独立一级菜单；「建议与反馈（M-FB）」为系统管理菜单 + AI 助手双入口。

### B.1 模型接入与管理（系统管理 · 独立菜单）

**B.1.1 模块目标**

为 M5 智能问答及全平台 AI 能力提供可视化的模型接入治理入口，确保模型可插拔、密钥安全、敏感数据不出域。

**B.1.2 功能需求**

- **模型 / 端点：** 本地（vLLM/Ollama）或内网 API；**对话模型与 embedding 模型分别配置**；多模型可切换（可插拔）；
- **凭证：** API Key / 服务地址，**入密钥管理库，不明文展示、不落明文**；
- **RAG 参数：** 召回数、相似度阈值、切片/索引刷新策略；
- **提示词与防护：** 系统提示词模板、提示注入防护开关、工具/数据访问范围限定在用户可见组织范围（`visibleOrgs`）；
- **「数据不出域」开关（默认开）：** 开启时仅允许内网模型/内网 API，**禁止外部模型端点**；
- **连通性测试 + 调用审计：** 测试按钮、AI 调用量/时延/失败监控；配置变更入操作审计；
- **安全：** 外部模型端点须经**出站白名单**（防 SSRF）。

**B.1.3 边界**

- 本模块仅做模型接入与治理配置，问答能力本体在 M5；密钥统一入密钥库，配置变更全程留痕。

### B.2 建议与反馈（M-FB · 系统管理 + AI 助手双入口）

**B.2.1 模块目标**

统一收集、格式化、分类、可行性初评全体使用者的 **BUG 反馈与功能建议**，并按治理要求脱敏审批后对外/对内同步与周期分析。

**B.2.2 功能需求**

- **双入口收集（入站）：** ① 系统管理「建议与反馈」菜单站内提交表单（类型、标题、描述、截图、所属模块、紧急度，自动带上下文：当前页面/版本/角色/子公司）；② M5 AI 助手内直接「提建议 / 报 BUG」，结构化后落入同一反馈库。说明：现有 4 类通知通道为单向推送，本模块新增的是**入站采集**，不改变通道单向性质；
- **格式化 + 分类：** 自动格式化为统一结构（标题/复现步骤/期望/实际/环境/附件）；一级分类 = BUG / 功能建议（可扩展）；二级 = 模块归属、严重度/价值度；
- **AI 可行性初评：** 规则 + LLM 给出价值/成本/风险初判与建议处置（采纳/观察/婉拒），**人工复核为准**（AI 建议不替代人工裁决）；
- **同步到代码仓库「反馈与建议」目录（GitHub · 脱敏 + 管理员审批后推送）：** 对外推送前强制 ① **自动脱敏**（移除/打码 PII、密钥、内部 IP、截图敏感区、子公司可识别信息）；② **管理员审批**（默认不自动直推）；③ **出站审计**（操作人/时间/内容哈希写入平台操作审计日志）；凭证（GitHub Token）入密钥库不落明文；可**一键切换为内网 Git**目标；
- **周期分析推送：** M10 通知场景库新增「**反馈与建议周期分析**」场景，按周期聚合（新增/分类占比/高价值建议 Top/高频 BUG/处置进度）生成结论，推送**指定管理员 + 企业微信机器人**（沿用管理层聚合简报模式）。

**B.2.3 边界**

- AI 分类/可行性评估为**辅助**，最终采纳由人工裁决；对外推送默认经**脱敏 + 审批**，不自动直出；对外出站数据外泄风险纳入风险登记并配套缓解。

## 十三、全局非功能需求

前述各模块的非功能需求为该模块特有的要求，本章统一约定适用于全平台的通用非功能需求。

### 13.1 性能要求

- 常规页面操作（列表加载、表单打开、查询）响应时间应在 3 秒以内；
- 制度知识库检索响应时间应在 3 秒以内，AI 问答首字返回时间应在合理范围内（如 5 秒以内），具体指标在设计阶段结合模型部署方式确定；
- 系统应支持公司当前及未来三年预期的用户规模与数据量，并发用户数及数据量预估需在设计阶段结合实际组织规模确认；
- 报告与文件导出（Word/PDF）应在合理时间内完成，大批量导出支持异步生成并通过 M10 通知用户下载。

### 13.2 可用性与可靠性

- 平台作为合规管理的日常工作系统，应具备较高的服务可用性，目标可用性指标（如 99.5%）在设计阶段确认；
- 关键操作（评估提交、审计发现记录、整改验证、签署确认）需有防误操作与数据自动保存机制，避免因会话中断丢失填写内容；
- 系统需具备完整的运行日志与错误日志，支持故障的快速定位与排查。

### 13.3 安全要求

- 所有用户访问需经身份认证（基于 AD 账号），认证方式支持与公司现有单点登录（SSO）体系对接（若有）；
- 敏感数据（子公司风险评估数据、证据材料）在传输与存储环节均需加密；
- 系统需记录关键操作的审计日志（谁、何时、对什么数据做了什么操作），审计日志本身不可篡改，留存期限符合监管要求；
- 系统需具备防范常见 Web 安全风险（如越权访问、注入、敏感信息泄露）的能力，上线前应通过安全测试；
- AI 问答的数据隔离作为重点安全约束，需单独进行越权访问测试（详见 M5、M8）。

### 13.4 兼容性与易用性

- 平台为 Web 应用，需兼容公司主流浏览器版本；
- 界面交互需符合一般业务系统的操作习惯，降低合规、内审、业务部门人员的学习成本；
- 配置类功能（表单模板、通知规则、权限角色）面向运营人员，应提供可视化配置界面，避免要求运营人员编写代码或脚本。

### 13.5 可维护性与可扩展性

本节统一约定平台应对未来变化的扩展性要求，各模块已分别落实的可配置能力（如表单模板自定义、通知规则可配置、体系脚手架可扩展、角色可自定义）不再重复，此处约定全局原则：

- 业务可变部分（评估模板、审计检查表、打分模型、通知规则、法规跟踪源、权限角色、内置体系分类）均应以配置方式实现，支持运营人员自助调整而非依赖研发改动代码；
- 系统应采用模块化架构，M6、M8、M10 等横向支撑能力以共用服务形式提供，新增业务模块时可直接复用，避免重复建设；
- 与外部系统的对接（AD、SIEM、短信平台等）应采用标准化接口封装，单个外部系统变更或替换时，影响范围可控。

### 13.6 国际化（i18n）【新增 · V1.9 · CR-001-3】

平台适用范围含**海外业务子公司**，需支持中 / 英文切换：

- 前端文案**外置语言包**（不硬编码文案），顶栏提供**语言切换**控件；
- **日期 / 数字 / 时区本地化**；
- 制度等内容支持**可选双语**；
- 新增模块/页面须同步纳入语言包，避免出现未翻译的硬编码文案。

### 13.7 多域控身份集成【新增 · V1.9 · CR-003】

集团下各子公司（含海外）可能各自运行独立 AD 域/林，平台须显式支持**多域控（多身份源）**集成：

- **多身份源配置：** 以「身份源（identity source）」为单位配置 N 个域控连接——域/林、LDAP(S) 地址（可多控制器高可用）、独立服务账号（独立密钥管理）、Base DN、字段映射、同步频率、绑定 `org_id`；
- **账号唯一性：** 由全局 `username` 改为 **(domain_id, username)** 复合唯一；登录标识采用 **UPN（user@domain）/ 域限定名（DOMAIN\\user）**；
- **认证路由：** 按域选择 / UPN 后缀 / 域前缀解析目标域，路由到对应域控校验；密码策略由各域 AD 承载；SSO 支持多 IdP（按子公司）；
- **组织映射与隔离：** 各域组织树挂到其子公司 `org_id` 子树；**各子公司管理本域用户与角色，集团管理跨域聚合视图**；
- **不依赖域间信任：** 各域独立对接即可；多域同一自然人不自动合并（按需由 M6 人工关联）；
- **可用性：** 单域不可用仅影响该域登录/同步，其余域照常；同步中断维持权限快照，恢复后增量补同步；
- **边界：** 仍为 AD → 平台**单向同步**，平台不维护账号体系；本项仅将「单域」扩展为「多域」，不改变同步与隔离的基本原则。

### 13.8 高自由度可配置（第一性产品原则）【升格 · V1.9 · CR-001-2】

用户核心诉求为**尽量多的自由度、更精细化地管理与自定义各模块**。将「配置化、反硬编码」由架构策略**升格为显式产品第一性原则**，并扩展可配置面：

- 在既有可配置面（表单模板/打分/报告映射、通知规则、角色、体系、法规源）基础上，新增可配置面：**自定义字段**（在对象上加业务字段）、**自定义列表视图与筛选**、**自定义看板**、**自定义报表模板**、**自定义工作流/审批流**、**自定义 KPI/指标**、**自定义通知场景与文案**；
- **配套治理（防过度自由致混乱）：** 沿用「集中管控、分级开放」治理思路全局化——结构性/底线项归口管理员 + **发布前预览测试**；自定义项纳入 M8 权限与**变更留痕**；
- **数据层预留：** 自定义字段需在数据模型预留扩展机制（如 **JSONB 扩展列 + 字段元数据登记**），**必须在数据模型阶段考虑**，后期补救成本高。

## 十四、数据要求

### 14.1 数据分类与保护

- 平台数据按敏感程度分类管理，至少区分公开/内部/敏感三级，子公司风险评估数据、审计发现、证据材料归为敏感级；
- 敏感数据的访问严格受 M8 权限引擎约束，并记录访问日志；
- 数据脱敏：在非必要场景（如跨子公司的汇总统计）下，应支持对明细敏感数据的脱敏处理。

### 14.2 数据备份与恢复

- 平台数据需定期备份，备份频率与恢复时间目标（RTO/RPO）在设计阶段结合数据重要性确认；
- 证据留痕库等监管举证相关数据需具备可靠的备份机制，确保在故障场景下不丢失。

### 14.3 历史数据迁移

- 公司现有以 Excel/Word 形式管理的制度文件、历史风险评估记录、审计记录需支持迁移导入平台；
- 制度文件已全部为可编辑电子文档，可直接解析导入；历史评估/审计记录的迁移需明确字段映射规则，对于无法结构化的历史记录，至少支持作为附件留存与检索；
- 迁移过程需保证数据完整性，并保留迁移前后的核对记录。

- ❌ **漏洞闭环管理**（漏洞台账/CVSS/扫描器对接/修复 SLA 闭环）属漏洞管理平台职责；GRC 仅通过接口采集「漏洞修复时效」等 **KRI 指标**；
- ❌ **SIEM / 漏洞扫描平台本体**不建设，GRC 仅做接口对接采集 KRI，不实现 SIEM/扫描能力；
- ⚠️ **完整安全事件管理（IR/SOC）不在本平台**，仅「**重大事件的监管报送**」纳入本平台监管事项（M11）。

集成通用要求：

- 所有外部系统对接应采用标准化接口（如 REST API、Webhook）封装，并对接口异常（超时、认证失败、数据格式异常）具备容错与告警机制；
- 对接涉及的认证凭证（如 AD 服务账号、短信平台密钥）需安全存储，不得明文留存于配置文件或代码中；
- 数据采集类对接（SIEM、日志平台、漏洞管理系统）需明确采集频率与数据格式，由相关系统运维团队共同确认接口规范；
- 对外出站对接（代码仓库 GitHub / 内网 Git、外部模型端点）须经脱敏/审批/出站白名单与出站审计治理，凭证统一入密钥库不落明文（CR-004）。
