## 一、引言

### 1.1 目的
定义 GRC 平台的总体技术架构，作为详细设计、开发、部署、测试的统一依据。本架构以五大质量目标为第一性原则：**易迭代、易维护、易安装、高可用、模块隔离（新增功能不牵一发动全身）**。

### 1.2 范围
覆盖 M1–M10 全部模块的总体结构、技术选型、模块解耦机制、部署与安装方案、高可用与可运维设计。数据模型细节见 D1-2，安全细节见 D1-3，表单引擎细节见 D1-6。

### 1.3 读者
架构师、开发负责人、运维/DevOps、测试负责人、项目经理。

### 1.4 术语
沿用需求文档术语表（GRC、RBAC、KRI、RAG、统一控件库、Policy Attestation、数据隔离等）。补充：

| 术语 | 说明 |
|---|---|
| 模块化单体 | 单一可部署应用内部按强边界划分模块，兼顾解耦与部署简单 |
| 横向共享内核 | M6/M8/M10 等被业务模块共用的基础服务 |
| 防腐层（ACL） | 隔离外部系统（AD/SIEM/短信等）的适配层，外部变更不渗透到核心 |
| 事件总线 | 模块间以"发布/订阅事件"解耦的中间件 |

## 二、架构质量目标（可度量）与对应策略

> 这是本架构的"宪法"。每条目标都映射到具体架构策略，并在后文落地。

| # | 质量目标 | 可度量标准 | 架构策略（详见章节） |
|---|---|---|---|
| Q1 | **模块隔离 / 不牵一发动全身** | 新增一个业务模块或事件类型，改动文件不外溢到其它业务模块；模块间无环依赖 | 模块边界 + 横向共享内核 + 事件驱动 + 配置化扩展点 + 防腐层（§4） |
| Q2 | **易迭代** | 业务可变项（模板/规则/角色/体系/法规源）调整**零改码、零发版**；接口/库结构演进可平滑升级 | 配置化优先 + API 版本化 + DB 迁移版本化（§4、§13） |
| Q3 | **易维护** | 技术栈组件数最小化；统一日志/监控；单人可定位故障 | 精简技术栈 + 可运维性设计（§5、§12） |
| Q4 | **易安装** | Linux 一条命令启动；首次使用浏览器向导完成配置；支持离线安装 | 全栈容器化 + 一键脚本 + Web 安装向导 + 离线包（§10） |
| Q5 | **高可用** | 目标可用性 99.5%；应用无单点；故障自愈；RTO/RPO 达标 | 无状态应用 + 水平扩展 + DB 主备 + 健康自愈 + 备份（§11） |
| Q6 | 性能 | 常规操作 / 检索 ≤3s；AI 首字 ≤5s（需求 13.1） | 缓存、异步、索引、向量检索（§6、§11） |
| Q7 | 安全合规 | 组织级数据隔离零越权；留痕防篡改；审计可举证 | 集中式权限 + 防腐 + 留痕（§9，详见 D1-3） |

## 三、架构总览

### 3.1 分层架构

```
┌──────────────────────────────────────────────────────────────┐
│  接入层   Nginx/Caddy 反向代理 · TLS · 负载均衡 · 静态资源        │
├──────────────────────────────────────────────────────────────┤
│  前端     Web SPA（设计令牌驱动·多主题） · Web 安装向导            │
├──────────────────────────────────────────────────────────────┤
│  应用层（模块化单体，无状态，可多实例）                            │
│  ┌────────── 业务模块 ──────────┐  ┌──── 横向共享内核 ────┐       │
│  │ M1制度 M2风险 M3审计 M4法规   │  │ M6组织/台账           │       │
│  │ M5问答 M7供应商               │  │ M8权限与审批流        │       │
│  └──────────────┬───────────────┘  │ M10通知中心           │       │
│                 │   只经接口/事件     │ M9看板/留痕           │       │
│  ┌──────────────▼───────────────────┴──────────────────┐       │
│  │  应用内核：事件总线 · 工作流引擎 · 表单引擎 · 报表引擎   │       │
│  └──────────────────────────────────────────────────────┘       │
├──────────────────────────────────────────────────────────────┤
│  集成层（防腐层 ACL：每个外部系统一个适配器）                     │
│  AD · SIEM · 日志平台 · 漏洞管理 · 短信网关 · 企微 · 飞书 · 邮件 · 法规爬虫 │
├──────────────────────────────────────────────────────────────┤
│  独立工作进程（可独立伸缩/独立失败，不拖垮主应用）                 │
│  AI/RAG 服务 · 法规爬虫 Worker · 报表导出 Worker · 调度器          │
├──────────────────────────────────────────────────────────────┤
│  数据层  PostgreSQL(+pgvector) · Redis · MinIO 对象存储           │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 模块全景与依赖（呼应需求 §二）

横向支撑模块（M6/M8/M10/M9）被业务模块（M1–M5/M7）共用，**业务模块之间不直接相互调用**，一律经"稳定接口"或"事件"交互，从根上避免互相耦合。

```
                 ┌───────── 事件总线（发布/订阅）─────────┐
                 │                                       │
 M1制度 ──┐   M2风险 ──┐  M3审计 ──┐  M4法规 ──┐          ▼
          │           │          │          │      M10通知中心（订阅各模块事件）
          ▼           ▼          ▼          ▼
   ┌──────────────────────────────────────────────┐
   │  M6 组织/资产台账   M8 权限与审批流   M9 看板/留痕  │  ← 横向共享内核
   └──────────────────────────────────────────────┘
            ▲                         ▲
   M5问答（数据源覆盖M1/M2/M3/M4，隔离依托M8）   M7供应商（复用M2表单引擎）
```

依赖规则（强制，CI 校验）：
- 业务模块 → 横向内核：✅ 允许（经接口）
- 业务模块 → 业务模块：❌ 禁止直接依赖；如需协作，走事件或经内核
- 横向内核 → 业务模块：❌ 禁止（内核不反向依赖业务）
- 任何环依赖：❌ 禁止

### 3.3 技术选型（建议方案，含理由）

> 选型原则：**组件数最小化（易安装/易维护）+ 企业生态成熟（易招人/易维护）+ 与既有设计令牌兼容**。

| 层 | 选型（建议） | 理由 |
|---|---|---|
| 前端 | Vue 3 + TypeScript + Vite（组件库基于已有设计令牌封装） | 设计规范是框架无关的 CSS 令牌；Vue3 国内生态成熟、易招人；可直接消费 tokens.css 多主题 |
| 后端 | Spring Boot 3（Java 17），**模块化单体** | 金融/政务主流；与工作流引擎 Flowable 同生态；模块化结构兼顾解耦与单体部署简单 |
| 工作流 | Flowable（嵌入） | M8 多级审批流（加签/转办/超时），不自研（需求 4.4 建议复用成熟模型） |
| 数据库 | PostgreSQL 16 + **pgvector** | 一库多用：关系数据 + JSONB（表单弹性结构）+ 向量检索（RAG），**减少组件 → 易安装** |
| 全文检索 | PostgreSQL 全文 / 可选 Elasticsearch | 初期用 PG 全文足够（文件规模有限，需求 3.3）；规模大时再引 ES |
| 缓存/会话/事件 | Redis（会话外置 + Redis Stream 事件总线） | 应用无状态化的关键；事件总线轻量，单机即可，集群可换 RabbitMQ/Kafka |
| 对象存储 | MinIO（自托管，S3 兼容） | 附件/证据材料；内网自托管，避免依赖公有云（金融/政务内网友好） |
| AI/RAG | 模型接入层（可配置：本地 vLLM/Ollama 或内网 API）+ RAG 编排 | 模型可插拔；隔离为独立服务，便于单独伸缩与替换 |
| 反向代理 | Nginx 或 Caddy（Caddy 自动 TLS） | 接入、TLS、静态资源、负载均衡 |
| 容器/编排 | Docker + Docker Compose（默认）；可选 Kubernetes（大规模） | 一条命令起服务的基础；同一镜像两种部署形态 |

> **关键决策**：采用**模块化单体**而非微服务。理由——微服务会显著增加部署与运维复杂度，直接违背"一条命令安装 / SOP 级运维 / 易维护"目标；而模块化单体在强边界 + 事件解耦下同样能做到"不牵一发动全身"，并保留把高负载/异构部分（AI、爬虫、报表）拆为独立工作进程的灵活性。

## 四、模块化与解耦设计（核心：避免牵一发动全身）

> 这是本项目最被强调的诉求，单列并给出**五重机制**。

### 4.1 强模块边界（Bounded Module）
- 每个模块（M1–M10）= 独立的代码包 + 独立的领域模型 + 对外只暴露 `接口契约（API/Service）`，内部实现私有。
- 模块间**不得**访问彼此的内部类/表；跨模块协作只能通过 ① 内核接口 ② 事件。
- 用构建期约束（如 ArchUnit / Spring Modulith / 模块依赖检查）在 **CI 中强制**依赖规则，越界即构建失败 → 从工程上杜绝"偷偷耦合"。

### 4.2 横向共享内核（M6/M8/M9/M10 作为服务）
- M6 组织/台账、M8 权限审批、M9 看板/留痕、M10 通知，封装为**共用服务接口**，业务模块统一调用。
- 好处：通知逻辑、权限逻辑、对象清单**只实现一次**（需求 §二说明），避免重复建设与不一致；某业务模块变更不影响这些公共能力。

### 4.3 事件驱动解耦
- 模块产生**领域事件**（如 `评估已完成`、`整改逾期`、`法规已更新`、`制度已发布`），发布到事件总线；M10 通知中心、M9 看板**订阅消费**。
- 新增一个事件类型或新增一个订阅者（如未来新通知场景）→ **不改动事件生产方**，只加订阅，天然满足"不牵一发动全身"。
- 事件总线抽象接口，底层默认 Redis Stream，可平滑替换为 RabbitMQ/Kafka。

### 4.4 配置化扩展点（业务可变部分外置为配置/元数据）
呼应需求 13.5："业务可变部分均以配置实现，运营人员自助调整而非依赖研发改码"。以下均为**配置数据**而非代码：

| 可变项 | 承载方式 | 新增/调整时 |
|---|---|---|
| 评估/审计模板、打分模型、报告映射 | 表单引擎元数据（DB） | 配置即生效，零发版 |
| 统一控件库与跨体系映射 | 控件元数据（DB） | 配置 |
| 内置体系脚手架（等保/ISO/PBOC…）+ 自定义新体系 | 体系定义（DB） | 配置新增体系 |
| 通知规则（事件→级别→通道） | 规则引擎配置（DB） | 配置 |
| 权限角色与数据范围 | RBAC 配置（DB） | 配置 |
| 法规跟踪源、采集规则 | 爬虫规则配置（DB） | 配置 |

→ 平台"功能演进"的大部分发生在**配置层**，根本不触碰代码，迭代风险最低。

### 4.5 防腐层（外部系统隔离）
- 每个外部系统（AD/SIEM/日志/漏洞/短信/企微/飞书/邮件/法规网站）= 一个**适配器**，实现统一的内部接口。
- 外部系统改版/替换时，**只改对应适配器**，核心业务零感知（呼应需求 13.5"单个外部系统变更影响范围可控"）。
- 例：短信网关更换厂商 → 仅替换短信适配器；企业微信 Webhook 失效 → 仅影响企微适配器并告警。

### 4.6 演进影响面对照表（直观证明"不牵一发"）

| 变更场景 | 受影响范围 | 是否需发版 |
|---|---|---|
| 新增一种通知场景/通道 | M10 配置 + （可能）1 个通道适配器 | 通道适配器需，规则不需 |
| 新增一个认证体系（如新行业认证） | 体系配置 + 模板（控件库复用） | 否（配置） |
| 调整某评估打分权重/报告章节 | 模板配置 | 否（配置） |
| AD 升级/换 SSO | AD 适配器 | 适配器需 |
| 新增一个业务模块（如"业务连续性管理"） | 新模块 + 复用 M6/M8/M9/M10 | 新模块需，既有模块不动 |
| 升级 AI 模型 | 模型接入层配置 | 否（配置） |

## 五、关键子系统架构

### 5.1 表单引擎（M2，最高难点 → 详见 D1-6）
- 两层解耦：**结构配置层**（域→控制项→问题）与**打分/报告映射层**分离存储与演进。
- 元数据驱动：模板=数据，渲染器读元数据动态生成表单/报告，**新增体系不需新增代码**。
- 版本化：模板有版本号，评估启动即固化版本快照，后续模板修订不影响进行中评估。
- M3 审计检查表、M7 供应商评估**复用同一引擎**，按"用途"字段区分，不另建第二套。

### 5.2 权限与数据隔离（M8 → 详见 D1-3）
- 集中式：所有模块的权限判断、M5 问答的数据过滤、M10 的通知路由，**统一调用 M8**，一套模型全平台复用。
- 四元组：`用户-角色-组织-数据`；组织维度隔离在**数据访问层统一拦截**（而非各模块各写一遍）。

### 5.3 通知中心（M10）—— 三层 + 事件驱动
- 通道管理（适配器）/ 规则引擎（可配置）/ 分级降噪，三层设计（需求 12.4）。
- 经事件总线消费各模块事件；发送记录写入 M9 留痕。新事件类型零侵入接入。

### 5.4 AI/RAG 服务（M5）—— 独立部署
- 拆为独立服务：模型推理资源（可能 GPU）与主应用解耦，可单独伸缩、单独升级、故障不拖垮主业务。
- 检索阶段强制经 M8 做组织级过滤（安全红线，详见 D1-3）。
- 向量存储用 pgvector（复用 PG，少组件）。

### 5.5 法规爬虫（M4）—— 独立 Worker
- 独立容器进程，调度采集；**失败隔离**（爬虫挂不影响平台其它功能）。
- 健康监控：连续多次未抓到预期内容 → 触发 M10 告警运维（需求 6.3）。
- 采集规则配置化，网站改版时改配置而非改码。

### 5.6 报表导出 —— 异步
- Word/PDF 大批量导出走队列 + Worker 异步生成，完成经 M10 通知用户下载（需求 13.1）。

### 5.7 证据留痕库（M9 → 详见 D1-2/D1-3）
- 写入即只读 + 哈希校验 + 操作日志独立存储；保留期限取各监管口径最大值。

## 六、数据架构（概述，详见 D1-2）
- 单库 PostgreSQL 承载关系数据 + JSONB（表单弹性结构）+ pgvector（向量）。
- **组织维度隔离为建模第一原则**（需求 7.3：后期补救成本极高）。
- 分类分级（公开/内部/敏感）、脱敏、版本化（制度/模板/评估快照）、留痕防篡改。
- 附件/证据材料存 MinIO，元数据与哈希存 PG。

---

## 七、集成架构（详见 D1-5）
- 所有外部对接经防腐层适配器，统一接口 + 容错 + 告警。
- 数据流向：AD→平台（用户/组织同步）；SIEM/日志/漏洞→平台（KRI 采集）；平台→短信/企微/飞书/邮件（单向推送）；法规网站→平台（爬虫）。
- 凭证（AD 服务账号、短信密钥）经密钥管理安全存储，不明文落配置/代码。

---

## 八、安全架构（概述，详见 D1-3）
认证（AD/SSO）、授权（RBAC 四元组）、数据隔离与 AI 越权防护、传输/存储加密、不可篡改审计日志、Web 安全、凭证安全。**AI 数据隔离作为重点安全约束单独越权测试**。

## 九、部署架构与安装方案 ★（重点诉求）

### 9.1 全栈容器化
平台所有组件以容器交付，由 Docker Compose 编排：

```
services: nginx(caddy) · grc-app(可多实例) · grc-ai · grc-crawler · grc-worker
          · postgres(+pgvector) · redis · minio
volumes:  pgdata · miniodata · config · logs   （持久化，升级不丢数据）
```

### 9.2 「一条命令启动」
**在线环境：**
```bash
# 方式一：一键脚本（拉取 compose + 镜像并启动）
curl -fsSL https://<内网制品库>/install.sh | bash

# 方式二：已有 compose 文件
docker compose up -d
```
**离线环境（金融/政务内网常无外网，重要）：**
```bash
# 1. 传入离线镜像包  2. 导入  3. 启动
docker load -i mandao-grc-offline.tar
./install.sh            # 启动全栈，自动等待依赖就绪
```
启动后服务自检（健康检查），就绪即可访问。

### 9.3 「Web 向导安装」（首次配置）
浏览器访问 `http://<服务器IP>:8080/setup`，向导分步引导（类似 GitLab/Nextcloud 体验）：

1. **环境自检**：检测 DB/Redis/MinIO 连通、磁盘、端口；
2. **数据库**：用内置 PG（默认）或填写外部 PG 连接；
3. **初始化**：自动建库、执行迁移、灌入内置体系脚手架（等保/ISO/PBOC…）；
4. **管理员账号**：创建平台超级管理员；
5. **AD 对接**（可跳过，后续配）：域控地址、服务账号、同步策略；
6. **通知通道**：企微/飞书 Webhook、邮件 SMTP、短信网关参数（可逐个测试发送）；
7. **License/参数**：留痕保留期限等监管参数确认；
8. **完成**：写入配置、标记已安装、跳转登录。

> 向导把"需要人工确认的接口参数"集中在一处可视化填写并即时测试连通，**运维无需手改配置文件**，契合"小白可操作"。

### 9.4 两种部署形态（同一套镜像）
| 形态 | 适用 | 拓扑 |
|---|---|---|
| **单机一体化** | 试点/中小规模/快速上线 | 单服务器，Compose 全栈 |
| **高可用集群** | 生产规模 | 多 app 实例 + Nginx 负载 + PG 主备 + Redis 哨兵 + MinIO 多盘；可上 K8s |

### 9.5 升级与回滚
```bash
./upgrade.sh            # 拉新镜像 → 备份 → 自动 DB 迁移 → 滚动重启 → 健康校验
./rollback.sh <版本>    # 回滚镜像 + 数据库回滚点
```
- 数据库迁移用 Flyway/Liquibase 版本化，**向前兼容**，升级不破坏进行中的评估。
- 配置与数据在 volume，升级保留。

## 十、高可用与可靠性 ★

| 维度 | 设计 |
|---|---|
| 应用无单点 | 应用层**无状态**（会话/缓存外置 Redis）→ 多实例 + Nginx 负载，单实例故障不影响服务 |
| 自愈 | 容器健康检查 + 重启策略；探活端点 `/health`、`/ready` |
| 数据库 HA | PostgreSQL 主备 + 自动故障转移（生产形态）；单机形态定时备份 |
| 备份恢复 | 全量 + 增量备份；**证据留痕库可靠备份**（监管举证不可丢，需求 14.2）；明确 RTO/RPO（设计阶段定值） |
| 异步削峰 | 报表导出、爬虫、AI 走独立 Worker/队列，突发负载不拖垮主交互 |
| 优雅降级 | AI/爬虫等非核心组件故障时，核心业务（制度/评估/审计/通知）仍可用 |
| 限流熔断 | 对外部接口（短信额度、企微/飞书频率限制）做限流，避免被限流封禁（需求 12.1） |

## 十一、可运维性（为 SOP 级运维文档铺路）★

> 运维文档要"小白可照做"，前提是平台本身把运维动作**脚本化、标准化、自检化**。架构据此设计：

- **统一日志**：结构化日志集中输出（可选接 ELK/Loki），含操作审计日志（不可篡改、独立存储）。
- **健康与诊断**：`/health`、`/ready`、`/metrics`（Prometheus 格式）；内置 `./doctor.sh` 一键体检（检查各组件、磁盘、外部连通、爬虫健康、备份状态）。
- **一键脚本集**（配套 SOP 文档逐条对应）：
  - `install.sh` 安装 · `upgrade.sh` 升级 · `rollback.sh` 回滚
  - `backup.sh` 备份 · `restore.sh` 恢复 · `doctor.sh` 体检 · `logs.sh` 收集日志
- **配置集中**：所有可调参数集中在 `config/`（或 Web 向导/管理后台），不散落代码。
- **监控告警**：关键指标（可用性、队列积压、爬虫失效、通知失败率、留痕写入）阈值告警，复用 M10 通道。

→ 运维 SOP（D4-2）即可写成"执行某脚本 → 看到某输出 → 完成"的傻瓜步骤。

## 十三、架构决策记录（ADR 摘要）

| ADR | 决策 | 理由 |
|---|---|---|
| ADR-01 | 采用**模块化单体**而非微服务 | 满足易安装/SOP 运维/易维护；强边界+事件已能解耦 |
| ADR-02 | **PostgreSQL + pgvector 一库多用** | 减少组件数，关系/JSONB/向量统一，利于安装与维护 |
| ADR-03 | **事件驱动**做模块间解耦 | 新增事件/订阅零侵入既有模块 |
| ADR-04 | **防腐层**隔离全部外部系统 | 外部变更影响范围可控 |
| ADR-05 | 工作流引擎选 **Flowable 嵌入** | 不自研审批流，复用成熟模型 |
| ADR-06 | AI/爬虫/报表 拆**独立 Worker** | 异构/高负载/易失败部分隔离，互不拖累 |
| ADR-07 | **全栈容器化 + Web 安装向导 + 离线包** | 一条命令安装 + 小白可配置 + 内网适配 |
| ADR-08 | 业务可变项一律**配置化** | 迭代发生在配置层，降低发版与回归风险 |

## 十四、风险与权衡

| 风险 | 说明 | 应对 |
|---|---|---|
| 模块化单体边界腐化 | 团队偷偷跨模块直连，久之退化为大泥球 | CI 强制依赖检查（ArchUnit/Modulith），越界即失败 |
| pgvector 在超大检索量下性能 | 数据量极大时向量/全文性能受限 | 预留切换 ES/Milvus 的抽象接口；监控检索耗时 |
| 一体化部署的高可用上限 | 单机形态非真高可用 | 提供集群形态；生产用集群 |
| 表单引擎过度通用 | 退化为万能表单生成器（需求 4.4） | 通用配置 + 内置脚手架双轨；详见 D1-6 |
| 离线升级的迁移风险 | 内网升级无法在线回滚镜像 | 升级前自动备份 + 回滚脚本 + 演练 |
